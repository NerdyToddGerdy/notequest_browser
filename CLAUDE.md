# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser-based digital adaptation of *NoteQuest* (by Tiago Junges), a solo, dice-driven, pen-and-paper dungeon-crawler board game. Target deployment is GitHub Pages (static hosting, no backend).

`docs/game-rules-reference.md` is the authoritative transcription of the rulebook (Core Book + Expanded World) â€” treat it as the source of truth for any table, mechanic, or dice logic. Don't invent rules not present there. The purchased rulebook PDF itself is intentionally not tracked in git (see `.gitignore`) since this repo is public.

`design/mockups/*.html` are standalone, self-contained (fonts embedded as base64) interactive layout studies used to validate screen designs before they're built as real components. They are not part of the app build, and are left as-is once ported â€” don't edit them expecting it to affect the app. Both current mockups (`character-creation.html`, `dungeon-building.html`) have been ported to real code; treat the mockups as historical design reference, not spec-of-record (the real code has since diverged in places, e.g. the dungeon screen's sidebar).

**Scope discipline**: v1 targets the Core Book rules only (character creation, procedural dungeon crawl, combat, torches/darkness, death/graveyard). Expanded World content (hexcrawl, cities, politics, warfare, multiplayer, advanced classes) is an explicit future phase â€” don't build toward it prematurely.

```
npm run preview   # serve the production build locally
npm run lint      # eslint .

```

Run a single test file: `npx vitest run src/engine/__tests__/character.test.ts`.

Node 20.17 is installed locally; some newer devDependencies want >=20.19 and npm prints `EBADENGINE` warnings on install. These are warnings only â€” install/build/test all work â€” but if a dependency install starts hard-failing, that mismatch is the first thing to check.

## Architecture

**Data-driven engine, not per-screen hardcoding.** Game rules live as typed data in `src/data/` (`races.ts`, `classes.ts`, `spells.ts`, `dungeonTypes.ts` â€” Segments/Secret Passage/Dungeon Name tables, shared across all 6 dungeon types â€” and `dungeonTables.ts` â€” the per-type Trap/Room Content/Monsters tables), keyed by dice-roll value, e.g. `RACE_TABLE[7]` for a 2d6 roll of 7. Pure game logic lives in `src/engine/` and takes an injectable `RNG = () => number` (defaulting to `Math.random`) so tests can pass a seeded generator (`src/test/mulberry32.ts`) for deterministic assertions instead of asserting on ranges. UI components consume `src/engine` functions and never re-implement roll logic themselves.

**The dungeon engine** (`src/engine/dungeon.ts`, `dungeonState.ts`, `dungeonReducer.ts`) is the more involved half. A dungeon is a tree of `levels[]`, each holding `segments[]` (rooms/corridors/staircases, positioned in an abstract pixel coordinate space for the map) and `connectors[]`. `dungeon.ts` holds two kinds of pure logic that are easy to conflate but are kept separate in the file: (1) *rules* â€” table rolls, `classifyDoorOpen()` (whether opening a given door is an ordinary roll, a staircase descent, the automatic depth-3 Final Room, the "no stairs ever found" dead-end Final Room, or a second staircase reusing an already-discovered level/Final Room); (2) *spatial layout* â€” `placeChild`/`placeIslandRoot`'s collision-avoiding placement algorithm, which is **not** a NoteQuest rule, just this project's own way of drawing a coherent map (the physical game doesn't need coordinates). `dungeonReducer.ts` applies `classifyDoorOpen()`'s result via Immer (`produce`) â€” deeply-nested state (`levels[i].segments[j].doors[k]`) makes manual immutable spreads unwieldy, hence the dependency. The reducer's extra `rng` parameter (for the silent Room Content/Monsters rolls) means it can't be passed directly to React's `useReducer` â€” `DungeonScreen.tsx` wraps it in a 2-arg `reduceDungeon` first.

**Torches and doors**: `DungeonState.torches`/`alive` track the resource that actually matters in this game â€” every door first rolls the shared `OPEN_DOOR_TABLE` (trap/locked/unlocked) via `RESOLVE_DOOR_LOCK`, *before* `OPEN_DOOR` ever runs; only "unlocked" (or a resolved lock/trap) proceeds to the existing Segments-table roll. Torch-spending (entering the dungeon, picking a lock, searching a secret passage, the ditch trap's `torchCost`) all flow through the reducer's `spendTorches()` helper, which flips `alive` to `false` (the Darkness) instead of going negative. `DungeonMap.tsx`'s door-click handler mirrors the reducer's math on the *client* side (checking `state.torches` before offering "Pick Lock", before animating a segment roll after a trap) purely so it knows whether to bother animating further dice or stop â€” the reducer is still the sole authority on whether the character actually dies. Locked doors show a "Pick Lock (1 torch) / Break Door (free)" popover; breaking a door or setting off a trap alerts the monsters beyond it (see Combat below).

**Combat** (`src/engine/combat.ts`) is pure, RNG-injected logic exactly like the rest of the engine; `dungeonReducer.ts` is the only thing that mutates `DungeonState.combat`/`hp`/`coins`. Whenever `buildSegment()` creates a new room segment (dungeon entrance, an ordinary door, a fresh staircase-descent level, or a second-staircase reuse) and that room's `resolveRoomExtras()` roll produced a `MonsterTemplate`, `startCombatIfMonsters()` spawns a `CombatState` via `spawnMonsters()`. Whether the player or the monsters get the first attack is decided by `wasNoisy` (true only when a door was broken open or a trap fired â€” see the "Combat" section of `docs/game-rules-reference.md`); a noisy start immediately runs one `applyMonsterTurn()` before the player can act. Once `state.combat` is non-null, `OPEN_DOOR`/`RESOLVE_DOOR_LOCK`/`ROLL_SECRET_PASSAGE`/`SWITCH_LEVEL` all no-op (the dungeon map itself also disables door buttons) â€” the only action is `PLAYER_ATTACK`, one full round per dispatch: `resolvePlayerAttack()` resolves the player's hit (Stoneskin/Intangible/Weakness, plus whichever roll-of-1 special ability fires â€” Explosive short-circuits into an immediate self-destruct that damages the player directly, bypassing normal damage math), the reducer applies the resulting events (Firebreath/Sorcery queue `bonusDamage`, Deathtouch/Paralyze queue onto the monster for its *own* next attack, Regeneration heals immediately, Horde/Necromancy always spawn a fixed `HORDE_ORC`/`NECROMANCY_SKELETON` regardless of whether the triggering monster died), then `checkUndeadRevival()` on any newly-defeated Undead monster before removing it. If any monsters remain, `applyMonsterTurn()` closes out the round (summing `damage + bonusDamage`, consuming queued Paralyze/Deathtouch, then clearing those queues) â€” paralysis is tracked as `combat.paralyzedTurns` and simply skips the player's next N `PLAYER_ATTACK` dispatches (monsters still act). Victory (`finishIfVictorious()`) rolls `rollLoot()` once per Loot-tagged monster actually removed from the fight, credits `coins` (Treasure/Key results are logged as flavor only â€” no inventory yet), marks the segment `monstersDefeated`, and clears `combat`. There is no armor system, so all damage â€” including Poison, which per the rulebook only matters for bypassing armor â€” goes straight to `DungeonState.hp`; hitting 0 sets `alive = false` and `deathCause = "combat"` (as opposed to `"darkness"` from running out of torches), which `DungeonScreen.tsx` uses to pick the death message. `CombatPanel.tsx` is the only UI surface for this â€” it rolls the player's weapon die (`character.cls.weaponDamage`, stored on `DungeonState.weaponFormula` at dungeon creation) and dispatches `PLAYER_ATTACK` with the raw roll; the reducer, not the component, applies the weapon's modifier.

**Screens** live in `src/ui/screens/<ScreenName>/`, one file + one CSS Module each. `App.tsx` holds the only navigation state (`character: CreatedCharacter | null`) and switches between `CharacterCreationScreen` and `DungeonScreen` â€” there's no router, and with only two screens that's a deliberate choice, not an oversight; revisit if a third screen needs it. Navigation is bidirectional: death sends the player back to `CharacterCreationScreen` via `DungeonScreen`'s `onNewAdventurer` prop (`App.tsx` just does `setCharacter(null)`), since a dead character is permanently dead â€” there's no way to resurrect or keep playing one past `!state.alive`. **Shared components** live in `src/ui/components/`: `Die`/`DicePool` (see below), `CharacterSheet` (renders a `CreatedCharacter`, used in the dungeon screen's sidebar), `Pack` (see Opening a Treasure), `Graveyard` (see below), and the dungeon-specific `DungeonMap`/`LevelTabs`/`RoomInspector`/`RollLog`. `src/ui/rollTiming.ts` has the shared `revealDelay()` timing helper both screens use to delay revealing a roll's result until its dice animation finishes.

**The Graveyard** (`src/engine/graveyard.ts`) is the one piece of state that survives a reload â€” everything else in this app is still fully in-memory. `loadGraveyard()`/`addGraveyardEntry()` read/write a JSON array under the `notequest:graveyard` localStorage key; both take an injectable `storage: Storage` parameter (mirroring the RNG-injection pattern elsewhere in `engine/`) so their tests run in Vitest's default Node environment, which has no `localStorage`. `DungeonScreen` records an entry in a `useEffect` keyed on `state.alive` flipping to `false` (so it fires exactly once per death, not on every re-render of the death panel), then `CharacterCreationScreen` loads the list once on mount (a fresh mount happens every time the player returns here, including right after a death) and renders it via the `Graveyard` component if it's non-empty.

**Resuming an unbeaten dungeon**: per the rulebook, dying (or otherwise leaving) doesn't erase a dungeon -- a later character can pick up exactly where it was left. `App.tsx` lifts a `pendingDungeons: PendingDungeon[]` array (in-memory only, unlike the Graveyard) above both screens, keyed by a `runId` (`crypto.randomUUID()`) that a `DungeonScreen` instance mints once at mount and re-mints on `handleStartNewDungeon`. Progress is captured generically in an unmount cleanup (`useEffect(() => () => onLeaveDungeon(...), [])`, reading the *latest* state via a ref kept fresh by its own render-triggered effect, since writing a ref during render itself isn't allowed) â€” this fires no matter *why* the screen unmounts (death, a "Retreat to Town" click, or a brand new character navigating away), so App's `handleLeaveDungeon` doesn't need to special-case the reason: it just drops the run if `isDungeonBeaten()` or it never went anywhere, otherwise upserts it. The resume choice is offered in two places, both ultimately dispatching the same `RESUME_DUNGEON` action: `CharacterCreationScreen` lists each pending entry as an alternative to the normal "Descend" button (picking one carries the chosen `PendingDungeon` through `onDescend` down to `DungeonScreen`, whose `useReducer` initializer calls `RESUME_DUNGEON` instead of `createInitialDungeonState()`); and, for a character who already chose to descend fresh, `DungeonScreen`'s own "Roll for Dungeon" gate repeats the same list (via a `pendingDungeons` prop threaded down from `App.tsx`) so that choice isn't locked in at character creation -- picking one there calls `handleResumeFromGate()`, which reassigns this `DungeonScreen` instance's `runId` to the picked entry's before dispatching `RESUME_DUNGEON` mid-session, so future saves update the right pending entry instead of the throwaway id this run started with. That action keeps the map/exploration fields (`levels`, `stats`, `log`, `dungeonName`, etc.) but resets every character-specific field (`torches`/`hp`/`maxHp`/`weaponFormula`/`spellUses`/`alive`) to the *new* character's own values â€” and, since Immer deep-freezes everything `produce()` returns, `structuredClone()`s the incoming persisted `DungeonState` before touching it, or mutating the frozen arrays throws. If the previous character died mid-fight, that segment's monsters (there can be at most one such segment dungeon-wide, since every other room's combat had already resolved by the time further doors could open) are respawned at full HP via the same `startCombat()` helper ordinary room entry uses, matching "a room that still has monsters recovers their health."

**Remains**: "all your equipment will be on the floor of that room to be recovered by your next character" (and the Darkness section says the same). `DungeonState.characterName` (seeded alongside the other character-derived fields at creation/resume) labels whose remains they are. `leaveRemains()` in `dungeonReducer.ts` fires at every death site -- the Darkness branch of `spendTorches()` (now threaded an optional `segId` from each of its six call sites, since the relevant room is whichever segment's action triggered the torch spend, not necessarily the currently-selected one) and both of `applyMonsterTurn()`'s kill branches plus `PLAYER_ATTACK`'s Explosive self-destruct branch (using `combat.segId`, since a combat death's location is unambiguous) -- and drops the dying character's current coins/Treasures/Keys/`heldItems` (the Pack contents -- see Opening a Treasure) onto `SegmentState.remains`, merging (`names` array grows, counts add, `heldItems` concatenates) if an earlier death already left something there. `RoomInspector` shows a "Remains" row plus a free "Recover Remains" button (no roll, matching the rulebook not listing this as a torch-costing action) that dispatches `COLLECT_REMAINS`, crediting the current character and clearing the segment's `remains`.

**`Die` component** (`src/ui/components/Die/Die.tsx`) is a real 3D CSS cube (six absolutely-positioned faces inside a `perspective` container, `rotateX`/`rotateY` per face), not an image or icon-swap. It's driven by a `value` (1-6) + `rollToken` (bump this to trigger a new roll animation even if the value repeats) + optional `delayMs` (for staggering dice in a pool) + `size` prop. The rotation math (`SHOW_ROTATION` table + `angleTo` helper) picks the shortest forward rotation to the target face and adds a couple of extra full turns for a tumble effect â€” see the component for the derivation. `DicePool` just renders N `Die`s with staggered delays for 2d6/etc. rolls. Screens own the *timing* of when to reveal a roll's result (see `revealDelay()` in `CharacterCreationScreen.tsx`) â€” the Die component itself doesn't expose a completion callback.

**Theme** (`src/ui/theme/`): `tokens.css` (CSS custom properties â€” the ink/parchment/gold palette, font stacks), `fonts.css` (`@font-face` for the three self-hosted fonts in `fonts/`: Metamorphous for display, Spectral for body, JetBrains Mono for stats/numbers â€” all OFL, embedded locally, no CDN), `global.css` (resets + the torchlit-dungeon body background, imported once in `main.tsx`). Component-level styling uses CSS Modules (`*.module.css`); shared tokens/fonts are global.

**Testing**: Vitest, default environment is `"node"` (fast, no DOM). A `jsdom` dependency in this environment currently throws `ERR_REQUIRE_ESM` when jsdom is loaded as the *global* Vitest environment â€” if you add component tests with React Testing Library, opt that specific file into jsdom via a `// @vitest-environment jsdom` docblock at the top of the file rather than changing the global config, or re-diagnose the jsdom/csstools ESM interop issue first.

**Deployment**: `vite.config.ts` sets `base` conditionally on `process.env.GITHUB_ACTIONS` (`/notequest_browser/` in CI, `/` locally) for GitHub Pages project-page hosting. No GitHub Actions workflow exists yet â€” that's still a TODO.

**Spellcasting** (`DungeonState.spellUses`, a `Record<spellRoll, remainingUses>` seeded at dungeon creation by `computeSpellUses()` from the character's `spells`/`fixedGrants`) covers all six Basic Spells via the `CAST_SPELL` action. Heal and Light are usable anytime (in `CombatPanel` or, out of combat, as a "Cast" button on `CharacterSheet`); Cold Ray, Lightning, and Fireball require an active fight and are rendered as extra per-monster or room-wide buttons in `CombatPanel`. Spell damage runs through `resolveSpellDamage()` in `combat.ts`, which shares Stoneskin/Intangible filtering with `resolvePlayerAttack()` but â€” since it's a fixed value, not a weapon "damage roll" â€” never triggers the roll-of-1/roll-of-6 abilities (Explosive, Firebreath, Weakness, etc.). Casting any spell mid-fight consumes the round exactly like `PLAYER_ATTACK` (the monsters counter-attack afterward), except Cold Ray also flags its target `skipNextAttack` so it sits out that round, and Teleport instead ends combat immediately with no counter-attack â€” this implementation's stand-in for "fleeing a fight," since the app has no notion of player position to send a rules-accurate Teleport to an out-of-combat destination room. A spell's uses, once spent, stay spent â€” there's no Rest yet to recover them.

**The Boss fight**: each dungeon type's `DUNGEON_TABLES[key].boss` (a `Record<1-6, MonsterTemplate>`, authored from the rulebook's per-dungeon "Table: Boss") is rolled once via `resolveBoss()` in `dungeon.ts` the moment the Final Room segment is built â€” both in the `descend-final` case (auto-triggered at depth 3) and `dead-end-final` (no stairs ever found). Per the rulebook, there's no Content/Monsters roll alongside it, only the Boss. The rolled `MonsterTemplate` is attached to the Final Room's existing `SegmentState.monsters` field and fed through the same `startCombatIfMonsters()` plumbing as an ordinary room, just with a new `isBoss` flag threaded onto `CombatState` â€” `finishIfVictorious()` checks that flag to grant a flat 2d6 Treasures instead of the normal per-monster Loot table (Treasures are flavor-only, like ordinary Loot, since there's no inventory system). `DungeonScreen.tsx` shows a gold victory panel once the Final Room's boss is marked defeated.

**Chests and Loot**: `RoomContentEntry.hasChest` flags the Room Content rows that describe an actual Chest (vs. passive flavor text like "1d6 coins on the floor," which isn't mechanically applied â€” see stub list below); a chest can also turn up via `secretPassageResult === "You have found a hidden Chest!"`. `RoomInspector` offers a free "Open Chest" action (no torch cost, per the rulebook) once one is available and unopened, rolling two independent dice â€” the higher becomes coins, the lower becomes Treasures, unless both come up 1, in which case the chest was empty and rolls on the dungeon's Trap table instead. `DungeonState.treasures`/`keys` are real counters (credited by both Chests and the Loot monster ability's `rollLoot()`) shown on `CharacterSheet` alongside coins â€” Keys don't do anything yet (no key-unlocks-any-door shortcut), but Treasures are openable (next).

**Opening a Treasure**: each dungeon type's `DUNGEON_TABLES[key].treasure` (a `Record<1-6, RewardOutcome>`) models the *Treasure column only* of the rulebook's per-dungeon "Table: Reward" â€” the Wonders and Magic Item columns (and Prison's Weapon-table redirect) reference armor/weapon items and a town economy this app doesn't have, so those rows resolve as flavor-only text via `RewardEffect`'s `"flavor"` kind, same as this codebase's other "not modeled" outcomes; rows repeated near-verbatim across dungeon types (Health Potion, Magic Scroll, Valuable jewel, the two redirects) are shared constants (`HEALTH_POTION`, `MAGIC_SCROLL`, etc. in `dungeonTables.ts`), mirroring the existing `BLADE_TRAP`/`CLICK_NOTHING` pattern for traps. The `OPEN_TREASURE` action is usable anytime (not tied to a room, unlike Chests) via a die-roll bar rendered in `DungeonScreen` whenever `treasures > 0`; mid-combat it consumes the round exactly like `CAST_SPELL` (paralysis included), since an unattended Health-Potion-on-demand would otherwise trivialize fights. `RewardEffect.kind: "restoreAllSpells"` (Tomb's Mana Potion) needs each spell's *max* uses, which `DungeonState` doesn't otherwise track â€” the action carries a `maxSpellUses` computed client-side via `computeSpellUses()` (the same helper `CharacterSheet` already uses) rather than adding a new persisted field for it. The rulebook's "worth N Coins in the town" outcomes (Ornament, Religious Object, Sinister Idol, Valuable jewel) don't convert to coins on the spot â€” there's no town to sell them in yet â€” so `RewardEffect`'s `"heldValue"`/`"heldValueRoll"` kinds instead push a `HeldItem` (`{ name, worth }`) onto `DungeonState.heldItems`, shown in the `Pack` sidebar card (`src/ui/components/Pack/`, positioned between `CharacterSheet` and the Ledger card) until a Town/sell system exists to cash them in.

**Still stubbed / not yet built** (see the scope note rendered on the dungeon screen itself, which is the live source of truth for this list): the armor system, race/class special combat abilities (Grave Digger's +2 vs Undead, Cook's coin-per-kill, Rinoceroid's horn attack), automatic Room Content rewards described only in flavor text (loose coins, Magic Scrolls, Magic Items â€” no dice are rolled or credited for these), and the Wonders/Magic Item/Weapon reward columns (armor and weapon items aren't modeled); HP damage from traps beyond the torch-costing ditch trap (blade/dart/etc. are logged as flavor only); Town/City actions beyond the bare "Retreat to Town" navigation (Rest, Fix Armor, Buy Torches, Sell Items â€” there's currently no way to refill torches or heal, whether starting fresh or resuming, and the Pack's held items have nowhere to actually be sold yet); and any persistence beyond the Graveyard (a reload loses `pendingDungeons` along with everything else in-progress -- resuming only survives character deaths/retreats within the same browser session, not a reload).
