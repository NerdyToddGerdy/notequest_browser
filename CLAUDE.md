# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser-based digital adaptation of *NoteQuest* (by Tiago Junges), a solo, dice-driven, pen-and-paper dungeon-crawler board game. Target deployment is GitHub Pages (static hosting, no backend).

`docs/game-rules-reference.md` is the authoritative transcription of the rulebook (Core Book + Expanded World) — treat it as the source of truth for any table, mechanic, or dice logic. Don't invent rules not present there. The purchased rulebook PDF itself is intentionally not tracked in git (see `.gitignore`) since this repo is public.

`design/mockups/*.html` are standalone, self-contained (fonts embedded as base64) interactive layout studies used to validate screen designs before they're built as real components. They are not part of the app build, and are left as-is once ported — don't edit them expecting it to affect the app. Both current mockups (`character-creation.html`, `dungeon-building.html`) have been ported to real code; treat the mockups as historical design reference, not spec-of-record (the real code has since diverged in places, e.g. the dungeon screen's sidebar).

**Scope discipline**: v1 (Core Book rules only — character creation, procedural dungeon crawl, combat, torches/darkness, death/graveyard) is complete. The project has since moved into the Expanded World phase (hexcrawl, cities, politics, warfare, multiplayer, advanced classes) — hexcrawl itself (issue #19) already shipped; the rest, including local multiplayer/Group Play (#29) and networked cross-browser sharing (#6), is all active, in-scope work now (issues #20, #22-30, #6) rather than deferred — pick up whichever is next roughly in the dependency order laid out across those issues, or whichever the user asks for by number, without treating any of them as premature.

## Commands

```
npm run dev       # start the Vite dev server
npm run build     # typecheck (tsc -b) then production build to dist/
npm run preview   # serve the production build locally
npm run lint      # eslint .
npm run format    # prettier --write .
npm run test      # vitest run (single pass)
npm run test:watch
npm run test:e2e  # playwright test -- e2e/, real browser regression tests
```

Run a single test file: `npx vitest run src/engine/__tests__/character.test.ts`.

`e2e/` holds Playwright specs (`playwright.config.ts`, `tsconfig.e2e.json`) -- a separate suite from Vitest's `src/**/__tests__/`, added for regressions that only show up end-to-end (localStorage session state → real rendered UI), not reachable by testing pure reducer/engine functions in isolation. `vite.config.ts`'s `test.exclude` keeps Vitest from trying to run these files itself (they use `@playwright/test`'s own `test`/`expect`, not Vitest's). `webServer` in `playwright.config.ts` starts `npm run dev` automatically and reuses an already-running one locally (`reuseExistingServer: !process.env.CI`).

Node 20.17 is installed locally; some newer devDependencies want >=20.19 and npm prints `EBADENGINE` warnings on install. These are warnings only — install/build/test all work — but if a dependency install starts hard-failing, that mismatch is the first thing to check.

## Architecture

**Data-driven engine, not per-screen hardcoding.** Game rules live as typed data in `src/data/` (`races.ts`, `classes.ts`, `spells.ts`, `dungeonTypes.ts` — Segments/Secret Passage/Dungeon Name tables, shared across all 6 dungeon types — and `dungeonTables.ts` — the per-type Trap/Room Content/Monsters tables), keyed by dice-roll value, e.g. `RACE_TABLE[7]` for a 2d6 roll of 7. Pure game logic lives in `src/engine/` and takes an injectable `RNG = () => number` (defaulting to `Math.random`) so tests can pass a seeded generator (`src/test/mulberry32.ts`) for deterministic assertions instead of asserting on ranges. UI components consume `src/engine` functions and never re-implement roll logic themselves.

**The dungeon engine** (`src/engine/dungeon.ts`, `dungeonState.ts`, `dungeonReducer.ts`) is the more involved half. A dungeon is a tree of `levels[]`, each holding `segments[]` (rooms/corridors/staircases, positioned in an abstract pixel coordinate space for the map) and `connectors[]`. `dungeon.ts` holds two kinds of pure logic that are easy to conflate but are kept separate in the file: (1) *rules* — table rolls, `classifyDoorOpen()` (whether opening a given door is an ordinary roll, a staircase descent, the automatic depth-3 Final Room, the "no stairs ever found" dead-end Final Room, or a second staircase reusing an already-discovered level/Final Room — see below); (2) *spatial layout* — `placeChild`'s collision-avoiding placement algorithm, which is **not** a NoteQuest rule, just this project's own way of drawing a coherent map (the physical game doesn't need coordinates). `dungeonReducer.ts` applies `classifyDoorOpen()`'s result via Immer (`produce`) — deeply-nested state (`levels[i].segments[j].doors[k]`) makes manual immutable spreads unwieldy, hence the dependency. The reducer's extra `rng` parameter (for the silent Room Content/Monsters rolls) means it can't be passed directly to React's `useReducer` — `DungeonScreen.tsx` wraps it in a 2-arg `reduceDungeon` first. A second staircase down to an already-discovered level (`reuse-normal`) links straight to that level's own existing entrance (`segments[0]`) rather than building anything new (issue #54) — the exact same "one shared destination" shape `reuse-final` already gave a second staircase down to an already-found Final Room, and neither needs a roll or shows up in `bumpStatsForNewSegment`'s tallies, since nothing new is built (see `DungeonMap`'s `AUTOMATIC_KINDS`). An earlier version instead placed a second, physically disconnected "island" entry point for `reuse-normal` (`placeIslandRoot`, since removed) — reported as two separate paths appearing on the destination level's map, which turned out to be this deliberate-at-the-time design rather than a placement bug, but wasn't what players expected from "the same staircase network," so it was replaced with the merged-entrance behavior instead.

**Torches and doors**: `DungeonState.torches`/`alive` track the resource that actually matters in this game — every door first rolls the shared `OPEN_DOOR_TABLE` (trap/locked/unlocked) via `RESOLVE_DOOR_LOCK`, *before* `OPEN_DOOR` ever runs; only "unlocked" (or a resolved lock/trap) proceeds to the existing Segments-table roll. Torch-spending (entering the dungeon, picking a lock, searching a secret passage, the ditch trap's `torchCost`) all flow through the reducer's `spendTorches()` helper, which flips `alive` to `false` (the Darkness) instead of going negative. `DungeonMap.tsx`'s door-click handler mirrors the reducer's math on the *client* side (checking `state.torches` before offering "Pick Lock", before animating a segment roll after a trap) purely so it knows whether to bother animating further dice or stop — the reducer is still the sole authority on whether the character actually dies. Locked doors show a "Pick Lock (1 torch) / Break Door (free)" popover; breaking a door or setting off a trap alerts the monsters beyond it (see Combat below). Beyond `torchCost`, `TrapEntry` also carries `damage` (flat HP loss, e.g. Acid Spout), `bladeTrap` (the identical row-1 Blade Trap across every dungeon type: a silent roll of 1 kills outright, a 2 "loses an arm" as flavor only -- no hand-economy system exists to enforce that), and `monsters` (an ambush -- Crypt's Bats, Tomb's Skeletons -- spawned into combat exactly like a room's own encounter, always noisy). All three trap-firing sites (`RESOLVE_DOOR_LOCK`, `ROLL_SECRET_PASSAGE`, `ROLL_CHEST`'s empty-chest trap) funnel through `resolveTrapOutcome()`, which checks for a `trapImmunity` item first (consuming it and blocking the whole trap, torch cost included) before falling through to `applyTrapEffect()` for whichever of the three mechanical shapes applies.

**Combat** (`src/engine/combat.ts`) is pure, RNG-injected logic exactly like the rest of the engine; `dungeonReducer.ts` is the only thing that mutates `DungeonState.combat`/`hp`/`coins`. Whenever `buildSegment()` creates a new room segment (an ordinary door, a fresh staircase-descent level, or a second-staircase reuse) and that room's `resolveRoomExtras()` roll produced a `MonsterTemplate`, `startCombatIfMonsters()` spawns a `CombatState` via `spawnMonsters()` -- except the dungeon entrance itself, which is exempt from Monsters entirely (a deliberate design/balance call, not from the rulebook, which says nothing about the entrance specifically): `resolveRoomExtras()` takes an `isEntrance` flag that still rolls the Monsters dice, for RNG-consumption parity with every other room, but discards the result, so the very first room a player steps into can never ambush them before they've opened a single door. Whether the player or the monsters get the first attack is decided by `wasNoisy` (true only when a door was broken open or a trap fired — see the "Combat" section of `docs/game-rules-reference.md`); a noisy start immediately runs one `applyMonsterTurn()` before the player can act. Once `state.combat` is non-null, `OPEN_DOOR`/`RESOLVE_DOOR_LOCK`/`ROLL_SECRET_PASSAGE`/`SWITCH_LEVEL` all no-op (the dungeon map itself also disables door buttons) — the only action is `PLAYER_ATTACK`, one full round per dispatch: `resolvePlayerAttack()` resolves the player's hit (Stoneskin/Intangible/Weakness, plus whichever roll-of-1 special ability fires — Explosive short-circuits into an immediate self-destruct that damages the player directly, bypassing normal damage math), the reducer applies the resulting events (Firebreath/Sorcery queue `bonusDamage`, Deathtouch/Paralyze queue onto the monster for its *own* next attack, Regeneration heals immediately, Horde/Necromancy always spawn a fixed `HORDE_ORC`/`NECROMANCY_SKELETON` regardless of whether the triggering monster died), then `checkUndeadRevival()` on any newly-defeated Undead monster before removing it (skipped entirely if the player holds an `ignoresMonsterAbility: "undead"` item — see Armor & Weapons). If any monsters remain, `applyMonsterTurn()` closes out the round: it splits the tally into `poisonDamage` (Poison "cannot be absorbed by armor or other means," so it always hits `hp` immediately, before anything else) and `absorbableDamage` (everyone else's `damage + bonusDamage`, deferred to `combat.pendingDamage` — see Armor & Weapons — if the player has usable armor, else applied to `hp` the same way poison was), then consumes queued Paralyze/Deathtouch and clears those queues — paralysis is tracked as `combat.paralyzedTurns` and simply skips the player's next N `PLAYER_ATTACK` dispatches (monsters still act). Victory (`finishIfVictorious()`) rolls `rollLoot()` once per Loot-tagged monster actually removed from the fight, credits `coins` (Treasure/Key results are logged as flavor only — no inventory yet), marks the segment `monstersDefeated`, and clears `combat`. Hitting 0 HP sets `alive = false` and `deathCause = "combat"` (as opposed to `"darkness"` from running out of torches) and clears `draft.combat = null` too, at every one of its four death sites (`applyMonsterTurn()`'s Deathtouch and normal counter-attack branches, `PLAYER_ATTACK`'s Explosive self-destruct branch, and `RESOLVE_DAMAGE`'s deferred-armor-choice branch) — otherwise `CombatPanel` kept rendering (and its Attack button kept rolling dice and dispatching no-op `PLAYER_ATTACK`s) right alongside the death overlay, since `DungeonScreen.tsx` renders it on `{state.combat && ...}` alone; `CombatPanel`'s own `canAct` also checks `hp > 0` directly as a second line of defense, in case a future death path forgets to clear `combat` too. `DungeonScreen.tsx` uses `deathCause` to pick the death message. `handleMonsterDefeat()`'s non-revived-kill branch also tallies `killsByName` (keyed by the monster's own lowercased `name`) and `killsByAbility` (one increment per ability the monster has, so a multi-ability monster counts toward more than one) alongside the existing `monsterKills`/`bossKills` split — the same "no formal taxonomy, just match the string" precedent Armor & Weapons' tag-matching already established, added specifically to unblock Advanced Classes' (#23) kill-count requirements like Ruthless's "10 Imps" (name-based) or Ghostbuster's "10 intangible beings" (ability-based) that `monsterKills`/`bossKills` alone can't answer. Both fields follow the exact same lifecycle as `monsterKills`/`bossKills` everywhere they're already threaded (`AdventurerResources`, `RESUME_DUNGEON` resets them to `{}` for a new character, `RETURN_TO_DUNGEON` carries them over exactly). `CharacterSheet`'s stat row shows `monsterKills` as a plain "Kills" number, but it's also a `<button>` (`.statClickable`) opening `KillBreakdownModal` (`src/ui/components/KillBreakdownModal/`) -- a dismiss-only informational modal (the first of its kind; `ConfirmDialog`'s backdrop/overlay styling was the visual starting point, but this has no confirm/cancel choice) listing `killsByName`, sorted highest-count first, capitalized for display (the underlying keys stay lowercased, matching `matchesTags()`'s own case-insensitive precedent). `CombatPanel.tsx` is the primary UI surface for this — it rolls the player's weapon die (`state.weapon?.formula ?? character.cls.weaponDamage` — see Armor & Weapons for how a found weapon overrides the class one) and dispatches `PLAYER_ATTACK` with the raw roll; the reducer, not the component, applies the weapon's modifier and any bonus effects. When `combat.pendingDamage` is set, `CombatPanel` instead renders an absorption chooser (HP or a specific armor piece, each showing current/max) that dispatches `RESOLVE_DAMAGE`; `PLAYER_ATTACK`/`CAST_SPELL`/`OPEN_TREASURE` all no-op until it's resolved, same as the existing paralyzed-turn gate.

**Screens** live in `src/ui/screens/<ScreenName>/`, one file + one CSS Module each. `App.tsx` holds the only navigation state and switches between `CharacterCreationScreen`, `WorldScreen`, and `DungeonScreen` — there's no router. `TownScreen` isn't in that list: it's a hex-generic component `WorldScreen` renders in place of the map whenever the player is standing on a City/Fortress hex (see Town/Town Square unification below), not a separate `App.tsx`-level screen of its own. Flow is `CharacterCreationScreen` → `WorldScreen` (landing on home, a city hex, so `TownScreen` is what's actually shown first) → `DungeonScreen`, with `DungeonScreen` able to send the player back to World (a voluntary retreat, alive) or all the way back to `CharacterCreationScreen` (death, via `onNewAdventurer` — a dead character is permanently dead, there's no way to resurrect or keep playing one past `!state.alive`). See **Town** below for how resources move between screens. **Shared components** live in `src/ui/components/`: `Die`/`DicePool` (see below), `CharacterSheet` (renders a `CreatedCharacter` plus live stats, used in both `TownScreen`/`WorldScreen`'s map view and the dungeon sidebar), `Equipment` and `Pack` (see Armor & Weapons / Opening a Treasure; each takes an optional `onFixArmor`/`onSell` to add a per-row action button, used only in `TownScreen`), `Graveyard` (see below), and the dungeon-specific `DungeonMap`/`LevelTabs`/`RoomInspector`/`RollLog`. `src/ui/rollTiming.ts` has the shared `revealDelay()` timing helper screens use to delay revealing a roll's result until its dice animation finishes.

**The Graveyard** (`src/engine/graveyard.ts`) was the first piece of state to survive a reload; `session.ts` (below) later generalized the same pattern to everything else. `loadGraveyard()`/`addGraveyardEntry()` read/write a JSON array under the `notequest:graveyard` localStorage key; both take an injectable `storage: Storage` parameter (mirroring the RNG-injection pattern elsewhere in `engine/`) so their tests run in Vitest's default Node environment, which has no `localStorage`. `DungeonScreen` records an entry in a `useEffect` keyed on `state.alive` flipping to `false` (so it fires exactly once per death, not on every re-render of the death panel), then `CharacterCreationScreen` loads the list once on mount (a fresh mount happens every time the player returns here, including right after a death). It's also shown on `TownScreen` (loaded the same way, once on mount) so a still-living character can see who's already died without waiting for their own death.

**`RecordsPanel`** (`src/ui/components/RecordsPanel/`) is what `CharacterCreationScreen`/`TownScreen` actually render for the Graveyard -- a tab switcher between `Graveyard` and the newer `DungeonsList` (`src/ui/components/DungeonsList/`, mirroring the Graveyard's own "running record" panel styling, fed `dungeonHistory` unfiltered -- every dungeon any character has touched, same "every adventurer" scope as the Graveyard's own list), shown only once *both* have at least one entry; with just one populated, that one renders directly with no tab bar, matching the Graveyard's own pre-#45 behavior exactly. `DungeonsList`'s per-entry status (Cleared/Unfinished) reuses `isDungeonBeaten()`; its unrecovered-remains count uses a new `countUnlootedRemains()` (`dungeonState.ts`, a sibling to the existing boolean `hasUnlootedRemains()`) that sums `remains.names.length` across every segment in every level, since a single room's remains can already hold more than one fallen adventurer's name. Clicking an entry never warps/travels to it (unlike the pre-#44 `activeDungeonElsewhere` shortcut this project deliberately removed) -- "must travel back to its hex" holds without exception; its own optional "Locate" button (issue #79, see below) only ever *shows* where it is on the map, the same distinction.
- **A right column, not a card of its own**: `TownScreen`'s Adventure section text (`.gateCopy`) is capped at `60ch`, well short of the card's full width -- rather than leave that width empty and stack `RecordsPanel` as its own full-width card below (still exactly what `CharacterCreationScreen` does, and still the fallback shape for `RecordsPanel`/`Graveyard`/`DungeonsList` themselves), `TownScreen` instead renders it inside a `.recordsCol` sitting beside Adventure in a two-column `.adventureRow` grid (`minmax(0, 60ch) minmax(240px, 1fr)`, collapsing to one column under `640px`) -- Records adds no extra card height at all on any screen wide enough for it, only rendered when `hasRecords` (`graveyard.length > 0 || dungeonHistory.length > 0`) is true, else Adventure stays plain full width rather than leaving a half-empty row. `Graveyard`/`DungeonsList`/`RecordsPanel` all gained an optional `compact` prop for this: it drops the standalone panel's own border/shadow/padding/width (the `.recordsCol` div already supplies a frame) and forces the row layout each already had as a `@media (max-width: 560px)` fallback -- a *viewport* query that would never fire just because the panel is sitting in a narrow *column* on an otherwise wide screen, so `compact` applies the identical single-column row styles unconditionally via a `.rowCompact` modifier class instead. `CharacterCreationScreen`'s own `RecordsPanel` usage is untouched (`compact` defaults to `false`), still the original full-width card.
- **Sorting and locating (issues #79/#80)**: `dungeonState.ts`'s new `sortDungeonsForDisplay()` stably sorts unfinished dungeons before cleared ones, preserving whatever secondary order it's handed within each group -- deliberately just that one grouping step, not a full comparator, so two different callers can each supply their own meaningful secondary order: `WorldScreen.tsx` pre-sorts `dungeonHistory` closest-to-farthest from `world.player` (a new `hexState.ts` helper, `findHexForRunId()`, turns a `PendingDungeon.id` back into a coordinate via the same `HexTile.dungeonRunId` stamp `withDungeonRunId()` already sets, then `hexDistance()` measures it -- both reused from Politics/Warfare) before ever reaching `DungeonsList`, while `CharacterCreationScreen` (no `WorldState`/player position to measure distance from) instead reverses its own raw, unsorted history for a recency fallback. `DungeonsList`'s new optional "Locate" button (`onLocate`, only ever wired up by `WorldScreen`/`TownScreen`, never `CharacterCreationScreen`, which has no map to show it on) calls `findHexForRunId()` the same way, then reuses the exact same `showMap`/`selectedHex` state a direct map click already drives -- selecting, not traveling to, the hex (see `RecordsPanel`'s own note above for why that distinction matters).
- **No-Affinity map indicator (issue #81)**: every one of the SVG hex loop's four corners was already claimed (dungeon/remains/building/political badges), so a hex the character's race has no `hasAffinity()` for instead gets a dashed, `var(--danger)`-colored outline on the hex polygon itself (`WorldScreen.tsx`) rather than a fifth corner glyph -- visible at a glance, and naturally overridden by the existing gold `isPlayer`/`isSelected` outlines when either applies (a forbidden hex can still be the player's own current tile, per issue #78's still-open lockout bug, or one merely being inspected).

**Session persistence** (`src/engine/session.ts`) covers everything else App-level: `character`, `resources`, `dungeonHistory`, and `activeRunId`, as one `SessionState` blob under the `notequest:session` key (same injectable-`storage` pattern as the Graveyard). `App.tsx` seeds its four separate `useState` calls from a single `loadSession()` call on mount, then a single `useEffect` watching all four re-persists the whole blob on any change, rather than threading a save call through every individual setter. `screen`/`selectedRunId` deliberately aren't persisted -- transient navigation state, not worth remembering; a reload just resumes wherever `world.player` physically was (itself persisted), or Character Creation if there's no character. `handleNewAdventurer` nulling out `character`/`resources`/`activeRunId` (while leaving `dungeonHistory` untouched, since the dead character's own abandoned dungeon should stay resumable) is enough to correctly clear the persisted session too, via that same effect -- no separate "clear" call needed.

**Settings hard reset** (issue #50): `session.ts`/`graveyard.ts` each export a `clearSession()`/`clearGraveyard()` (same injectable-`storage` shape as their `load`/`save` siblings, `storage.removeItem()` wrapped in the same try/catch) for when a player actually wants to nuke everything, not just retire one character. `App.tsx`'s `handleHardReset()` calls both, then resets every piece of state it owns (`character`/`resources`/`activeRunId`/`selectedRunId`/`dungeonHistory`/`world`/`forcedTypeRoll`/`worldFreshRunId`) back to its own initial value and lands on `screen: "world"` (harmless -- with `character` now null, the very next render falls through to Character Creation regardless of `screen`). Note `clearSession()` only clears what's on disk at that instant: `App.tsx`'s own persistence `useEffect` (above) re-fires on the next render with the freshly-nulled state and immediately re-writes `notequest:session` with `EMPTY_SESSION`'s shape -- not a bug, since an empty blob round-trips through `loadSession()` to the exact same "no character" result a missing key would. The UI surface is a new shared `Footer` component (`src/ui/components/Footer/`), replacing 4 near-identical inline `<footer>` blocks that had drifted slightly (`CharacterCreationScreen` used its own margin/font-size values) -- takes `screenLabel` (e.g. `"THE DUNGEON"`, rendered as "NOTEQUEST · THE DUNGEON") and `onHardReset`, and owns its own `ConfirmDialog`-gated "Settings" button so no screen has to wire up the confirmation flow itself. Rendered by `CharacterCreationScreen`, `TownScreen`, `WorldScreen` (its own map view, alongside `TownScreen`'s -- both need `onHardReset` threaded from `App.tsx`, since `WorldScreen` renders `TownScreen` in place rather than switching to a different top-level screen, see Town above), and `DungeonScreen`.

**Random names** (`src/data/names.ts`, issue #40): not a rulebook mechanic -- a v1-scoped extra. `FIRST_NAME_TABLE`/`LAST_NAME_TABLE` are `Record<string, Record<1-6, string>>` keyed by exact `RaceDef.name` (all 11 races authored, each with its own naming theme -- e.g. Dwarf clan-surnames, Slimemen onomatopoeic first names with descriptor "last names" like "the Damp"), plus a shared `"default"` fallback bucket for any race without a table of its own yet (see issue #22, "New Races"). `rollName()` (`character.ts`) rolls one die each into the two tables and joins the results; `CharacterCreationScreen`'s "Random Name" button (next to the name input) calls it with the currently-rolled race's name (or `"default"` before one's picked) and fills the field directly -- no roll animation, and the result stays freely editable afterward, same as typing a name in by hand. Every new race issue #22 adds falls back to this same `"default"` bucket -- none of them get their own naming theme yet.

**New Races** (Expanded World, `docs/game-rules-reference.md` lines 1302-1388, issue #22): three additional 1d6 race tables (`UNCOMMON_RACE_TABLE`/`EXOTIC_RACE_TABLE`/`MONSTROUS_RACE_TABLE`, `src/data/races.ts`) the player can pick instead of the Core Book's own 2d6 `RACE_TABLE` -- "Instead of rolling a race on the base table, you can choose one of these tables." These are genuinely separate, self-contained tables, not more dice folded into the existing 2d6 list: `CharacterCreationScreen.tsx`'s new `raceTable` state (`RaceTableKey`, a small row of toggle buttons above the existing "Roll for Race" button) picks which one is active, and switching tables clears whatever was already rolled -- a fresh choice, not a modifier stacked on top. `rollRaceFromTable()` (`character.ts`) is the single entry point: `"core"` delegates straight to the existing `rollRace()`, everything else is a flat 1d6 lookup. A handful of rows across these tables re-list a race that's already in `RACE_TABLE` verbatim (Lightbugster/Pixie/Slimemen/Cat-Person in Uncommon, Rinoceroid in Exotic, Dragonkin in Monstrous) -- those reuse the existing entry directly rather than re-authoring identical data. Prohibited Races (a fourth table, explicitly non-canonical -- "ideas of users of the Facebook group... use at your own risk," 36 entries via a 1d6-group/1d6-row roll fundamentally different from a flat 1d6 or a 2d6 sum) is deliberately not offered at all in this pass, a scoping decision, not an oversight.

Half-Human ("Roll a new race and use the advantage of that") resolves entirely at roll time inside `rollRaceFromTable()`: rolling it immediately also rolls Core `rollRace()` and merges that race's `ability` text and spell grants (`randomSpells`/`fixedSpell`) onto Half-Human's own name/HP, so the rest of the app only ever sees one coherent `RaceDef` -- nothing downstream (Character Creation's spell-requirement math, `DungeonState.raceName` ability checks, Affinity) needs to know "a Half-Human pretending to be a Dwarf" is two races layered together. The extra roll's dice are appended to the returned array (`[roll, ...inherited.dice]`) so `DicePool` still animates every die that contributed -- a Half-Human result shows 3 dice, not 1, since it's really a 1d6 (which table row) plus a 2d6 (which Core race).

Of the eighteen new races' abilities, only a few got a real mechanical hook, chosen for being genuinely self-contained rather than needing infrastructure this codebase doesn't have yet: Patovsky/Sharkin's water-walking (`hasWaterWalk()`, `hexTables.ts` -- checked alongside `WorldState.hasBoat` in both `WorldScreen.tsx`'s `canTravelTo()` and `hexReducer.ts`'s `MOVE` case, the same "UI mirrors, reducer is the actual authority" split every other World mechanic already uses); Pandakhan/Centaur's travel-cost multiplier (`travelCostMultiplier()`, 2x/0.5x respectively, layered on top of `travelCost()`'s base terrain cost in `WorldScreen.tsx`'s `handleTravel()` the same way Elven Boots' forest override already is -- Centaur's half rounds up via `Math.ceil` so a move is never free); and Ogre's unconditional +2 damage (`dungeonReducer.ts`'s `attackBonus()`, right alongside Grave Digger's own Undead-only +2, the same function/precedent). Pumpkinkin's "3 uses of the Vimes spell" and Corvino's "5 random Advanced Spells" grants were already real data (`randomSpellsTable`/`fixedSpell` in `src/data/races.ts`) the moment #24 shipped the Nature/Advanced tables -- now that Vimes itself is a real, castable spell (issue #61, Tier 2), Pumpkinkin's grant is fully functional end to end, not just correctly-tracked data with nothing to spend it on. **Samambro**'s "when you die, roll a die; 3+ and you come back with 1 HP" (issue #60) is also now real: `trySamambroSurvival()` is checked at all seven of `dungeonReducer.ts`'s `draft.alive = false` sites (Darkness, both trap-death branches, Deathtouch, the normal counter-attack, Explosive self-destruct, and the deferred-armor-choice branch) -- each one calls it immediately before flipping `alive`, and an early `return` on success skips that site's own death bookkeeping (`deathCause`/`leaveRemains`/`combat = null`) entirely, since the character never actually died. Only rolls for an actual Samambro (no "wasted roll" for every other race). This needed threading a new `rng: RNG` parameter through `spendTorches()` and `applyMonsterTurn()` (neither previously took one), since both are called from many sites throughout the reducer and Samambro's check needs the same shared per-dispatch `rng` everything else already uses. **Ogre**'s "Cannot use potions, scrolls or wear armor" (issue #60, the other half of an ability that already shipped its +2 damage half) is also now real, guarded at every site that would otherwise grant one of those three things: `resolveWonder()` (`dungeonReducer.ts`) redirects an Ogre to the sellable-item path (see the dedicated `#83` note below) whenever the outcome would otherwise grant *something* (a potion effect like Potion of Fury/Luminescence, a `randomSpell` scroll grant, or a worn `wonderItem`/`grantsHp` trinket like a Jester Hat or Potion of Luck) -- a pure-flavor row with nothing to grant even for a non-Ogre (e.g. "Lamp") is untouched, since there's nothing to salvage there either. `resolveMagicItem()` only special-cases its `grants === "armor"` branch -- an Ogre still fully benefits from `"[Weapon] of X"` items, since the restriction never mentions weapons. `OPEN_TREASURE`'s direct `healAll` (Health Potion)/`restoreAllSpells` (Mana Potion)/`randomSpell` (Magic Scroll) cases and `applyRoomContentReward()`'s `magicScrolls` case each get their own early check (the Treasure/room reward is still "found"/spent, and -- as of issue #83 -- sold rather than discarded outright). `COLLECT_REMAINS` recovers a fallen adventurer's coins/Treasures/Keys/held items/weapons as normal but leaves their armor behind in the remains -- unlike every other site here, this armor is real and already sitting in the room, so it's left recoverable by a later non-Ogre character instead of being force-sold; issue #83's report confirmed this specific case reads as intentional, not as gear vanishing, so it was left as-is. Outside a dungeon, `town.ts`'s `canDrinkVerdosaPotion()`/`canBrewHealthPotion()` both gained an `isOgre` parameter (disabling the button entirely rather than letting the action fire and do nothing) -- necessary because both are gated on something *other* than the drinker's own race (a Goblin-*culture* city's Culture Action; the Alchemist Advanced Class, acquirable independent of race), so an Ogre could otherwise still reach them. Everything else here still resolves as flavor-only ability text, the same "documented, deliberate simplification" precedent as `bladeTrap`'s roll-of-2 or `WeaponEntry.twoHanded`, rather than inventing rules the rulebook doesn't fully specify: Goblin's roll-of-1 self-destruct (ambiguous whether it also damages the Goblin themselves) and Fungoid's "at any time" 1-provision heal (needs provisions to exist inside a dungeon run too, which `DungeonState` doesn't track at all -- see the Town section's own note on why).

**Unusable gear is sellable instead of vanishing** (issue #83): every one of Ogre's "cannot use potions, scrolls or wear armor" sites above used to just log a flavor message and discard the roll outright. `resolveMagicItem()`'s armor branch now always rolls the base `ARMOR_TABLE` (even for an Ogre, so RNG consumption is identical to anyone else rolling the same branch) and, for an Ogre, calls the shared `addHeldItem()` helper (issue #82's own Pack-cap-aware chokepoint, so an Ogre's sellable find respects `MAX_HELD_ITEMS`/`pendingPackItem` exactly like any other found item) with `{ name: entry.name, worth: Math.max(1, base.maxHp) }` instead of returning early -- worth derived from the piece's own `maxHp` since armor has no rulebook coin value of its own to draw on (a documented simplification, same tier as `bladeTrap`'s roll-of-2). `resolveWonder()`'s per-branch Ogre check (see above) does the same: a trinket with its own `grantsHp` sells for that HP amount, while a potion/scroll-shaped effect (`combatDamageBonus`/`grantsTorches`/`randomSpell`, or any other non-flavor effect) sells for a flat `OGRE_UNUSABLE_TREASURE_WORTH` (3 coins, matching the scale of this dungeon type's other small heldValue Treasures -- Religious Object/Sinister Idol). `applyRoomContentReward()`'s `magicScrolls` case bundles the whole roll into one `HeldItem` ("N Magic Scrolls", worth `count * OGRE_UNUSABLE_TREASURE_WORTH`) rather than one item per scroll, so a "2d6 Magic Scrolls" reward can't flood the Pack with a dozen near-worthless entries. `OPEN_TREASURE`'s `healAll`/`restoreAllSpells`/`randomSpell` cases each get the same treatment with a hardcoded item name ("Health Potion"/"Mana Potion"/"Magic Scroll" -- the only outcome using each of those `RewardEffect` kinds anywhere in the tables, so there's no ambiguity to resolve from `RewardOutcome.text` alone, which has no separate `name` field the way `WonderEntry`/`MagicItemEntry` do). Confirmed with the user (this issue's own "scope question for whoever picks this up") that scrolls/potions should get the same treatment as armor, and that Wonders' worn-trinket rows specifically should too, rather than scoping this to just the Magic Item armor branch the issue called out as its clearest case.

Also deliberately out of scope for this pass: the rulebook's own per-race starting location ("Monstrous Races... start the game in an Orc or Goblin town," "Exotic Races... start out in a water territory surrounded by water territories, on a boat") -- every new race still starts at the normal `world.home` ("Haven") like every Core race does, since a real per-race starting area would need new world-generation infrastructure (a second "home" concept) this app doesn't have. `RACE_AFFINITY`'s old combined `"Orc/Ogre"` row was split into separate exact-name `Orc`/`Ogre` rows (identical values) now that issue #22 made both independently playable -- every other new race has no explicit rulebook Affinity row at all and correctly falls through to the existing `DEFAULT_AFFINITY` catch-all, needing no changes there.

**Town** isn't a distinct `App.tsx` screen -- `TownScreen` is a hex-generic "Town Square" component that `WorldScreen` renders in full (map hidden) whenever the player is standing on any City/Fortress hex, home included (home is just `world.home`, a human city on a plain, no different from any other city hex once found). It's `WorldScreen` that computes and hands down `hasDungeon`/`dungeonGateCopy`/`onEnterDungeon` for whichever hex the player is currently on (see per-hex dungeon persistence below) and `onExploreWorld` (toggles `WorldScreen`'s own map view back on, see below) -- `TownScreen` itself has no notion of "home" vs. "a city I found," which is what makes City Actions and dungeon access behave identically everywhere. `App.tsx` lifts an `AdventurerResources` object (`src/engine/town.ts` — `torches`/`hp`/`maxHp`/`coins`/`treasures`/`keys`/`heldItems`/`armor`/`weapon`/`spellUses`/`provisions`, i.e. everything a living character carries between dungeon and town) alongside `character`; it's seeded from the character's starting stats the moment `CharacterCreationScreen` calls `onCharacterCreated` (which also resets `world.player` back to `world.home` -- a new adventurer starts fresh at the city, not wherever the last one's body was left -- and always lands on `screen: "world"`, never straight into a dungeon), and overwritten wholesale from the dungeon's own fields whenever `DungeonScreen` calls `onReturnToTown(runId, dungeon)`. `town.ts`'s `rest()`/`buyTorch()`/`buyProvision()`/`sellItem()`/`fixArmor()` are pure functions (no RNG needed -- all fixed-cost per the rulebook) that `TownScreen` calls and hands the result to `onUpdateResources`; `canRest()`/`canBuyTorch()`/`canBuyProvision()`/`canFixArmor()` gate the buttons (1 coin; torches/provisions additionally capped at the rulebook's 10/20, a piece additionally must actually be damaged). Selling reuses `Pack`'s `onSell` prop, crediting the coins a `HeldItem` already carries as its `worth` (see Opening a Treasure) rather than re-rolling a price; fixing reuses `Equipment`'s `onFixArmor` prop the same way (see Armor & Weapons). `rest()` heals to `resources.maxHp` specifically (not a hardcoded/derived value) — this only works correctly because `maxHp` is threaded as its own field everywhere a dungeon gets (re)constructed (see the `maxHp` note below); before that fix, returning to a dungeon below full HP silently clamped the character's *true* max HP down to whatever they currently had, making Rest heal to a shrinking ceiling instead of the real max.

**Resuming an unbeaten dungeon**: per the rulebook, dying (or otherwise leaving) doesn't erase a dungeon -- a later character can pick up exactly where it was left, and returning to town to rest doesn't either. `App.tsx` lifts a `dungeonHistory: PendingDungeon[]` array (persisted via `session.ts` like everything else App-level, same as the Graveyard's own localStorage persistence in spirit) above the Town/Dungeon screens, keyed by a `runId` (`crypto.randomUUID()`) that a `DungeonScreen` instance mints once at mount and keeps for its whole lifetime. Progress is captured generically in an unmount cleanup (`useEffect(() => () => onLeaveDungeon(...), [])`, reading the *latest* state via a ref kept fresh by its own render-triggered effect, since writing a ref during render itself isn't allowed) — this fires no matter *why* the screen unmounts (death, a "Retreat to Town" click, or beating the Final Room), so App's `handleLeaveDungeon` doesn't need to special-case the reason: it just drops the run if it never went anywhere (`levels.length === 0`), otherwise upserts it -- unlike `pendingDungeons` before it, **beaten runs are kept too**, not dropped, purely as a historical record (World's per-hex badges below are the only place this shows up now; `isDungeonBeaten()` is computed on demand wherever "is this one still resumable?" matters, rather than stored as a field). `App.tsx` separately tracks `activeRunId` -- the *current, still-living* character's own paused run, if any -- set whenever `onReturnToTown` reports a non-empty, unbeaten dungeon. Choosing what to enter next just sets App's `selectedRunId` and switches to the dungeon screen -- every case (this character's own active run, a fresh roll, or taking over a different, abandoned run) now comes from `WorldScreen`'s `onEnterDungeon`, per whichever hex the player is standing on (see per-hex dungeon persistence below); there's no separate "continue" action anymore since standing on the hex your own run is tied to and clicking "Enter Dungeon" resolves to the exact same place. App then derives, from whether `selectedRunId === activeRunId`, which of two very different resume paths `DungeonScreen`'s mount-time `useReducer` initializer takes (both land in the same `restoreMapFromPersisted()` reducer helper for the shared map-copying/monster-respawn logic, but differ in what happens to resources and where the player starts):
- **`RESUME_DUNGEON`** -- a *new* character taking over a *dead* one's map, found by traveling back to the hex it's tied to. Resets every character-specific field (`torches`/`hp`/`maxHp`/`weaponFormula`/`spellUses`/`coins`/`treasures`/`keys`/`heldItems`/`armor`/`weapon`) to the new character's own starting `resources`, logs "A new adventurer takes up the fallen's path.", and -- since `restoreMapFromPersisted()`'s `resetToEntrance` flag is `true` for this path -- forces `activeLevel`/`selectedSegId` back to the entrance segment regardless of where the previous character died deep in the dungeon: "he will find his backpack and clothes on the floor" implies discovering the remains by exploring, not teleporting to them. (The full multi-level map is untouched; only the initially-displayed position resets.) The dead character's own belongings stay behind as recoverable `remains` (see below) -- they are not inherited.
- **`RETURN_TO_DUNGEON`** -- the *same still-living* character coming back to the hex their own run is tied to and clicking "Enter Dungeon" there. `DungeonScreen`'s `useReducer` initializer dispatches this automatically on mount whenever `activeDungeon` is set (skipping the gate entirely), carrying over `resources` *exactly* -- including `coins`/`treasures`/`keys`/`heldItems`, which `RESUME_DUNGEON` deliberately does not -- with `resetToEntrance: false`, so it picks up in the exact room/level it left off in. Logs "You return to the dungeon." instead.

Both actions `structuredClone()` the incoming persisted `DungeonState` before touching it, since Immer deep-freezes everything `produce()` returns and mutating a frozen array throws. Either way, if the previous session died or left mid-fight, that segment's monsters (there can be at most one such segment dungeon-wide, since every other room's combat had already resolved by the time further doors could open) are respawned at full HP via the same `startCombat()` helper ordinary room entry uses, matching "a room that still has monsters recovers their health." Both actions also carry an explicit `maxHp` field (as does the plain fresh-roll path, `DungeonScreen`'s `createInitialDungeonState(..., resources.maxHp)` call) -- `createInitialDungeonState()`'s `startingHp` param only ever set *current* HP historically, defaulting `maxHp` to match it, which was harmless when every entry point started at full health but broke the moment Town made "retreat below full HP, then come back" possible.

**Remains**: "all your equipment will be on the floor of that room to be recovered by your next character" (and the Darkness section says the same). `DungeonState.characterName` (seeded alongside the other character-derived fields at creation/resume) labels whose remains they are. `leaveRemains()` in `dungeonReducer.ts` fires at every death site -- the Darkness branch of `spendTorches()` (now threaded an optional `segId` from each of its six call sites, since the relevant room is whichever segment's action triggered the torch spend, not necessarily the currently-selected one) and both of `applyMonsterTurn()`'s kill branches plus `PLAYER_ATTACK`'s Explosive self-destruct branch (using `combat.segId`, since a combat death's location is unambiguous) -- and drops the dying character's current coins/Treasures/Keys/`heldItems`/`armor`/`weapon` (the Pack and Equipment contents -- see Opening a Treasure / Armor & Weapons) onto `SegmentState.remains`, merging (`names` array grows, counts add, `heldItems`/`armor` concatenate, `weapon` fills in only if nothing was left behind already) if an earlier death already left something there. `RoomInspector` shows a "Remains" row plus a free "Recover Remains" button (no roll, matching the rulebook not listing this as a torch-costing action) that dispatches `COLLECT_REMAINS`, crediting the current character and clearing the segment's `remains`.

**Hexploring the World**: the rulebook's Expanded World hex-travel system (`docs/game-rules-reference.md` lines ~828-1050), built as a deliberately thin first pass with clearly-scoped follow-ups, not the full section -- see below for exactly what's still missing. `src/engine/hexState.ts` models the map as a sparse `Record<string, HexTile>` keyed by `hexKey({q, r})` (axial, pointy-top coordinates, mirroring `DungeonState.segments`' own lazy-reveal pattern -- a tile only exists once its terrain+location are rolled), populated 6-at-a-time by `revealNeighborsInPlace()` ("roll one die in the Terrain table... roll one more die to see if there's a location... if 6, roll on the Location table") every time the player enters a new hex. `createInitialWorldState()` fixes the start at `{0,0}`, a human city on a plain (not rolled -- "Choose a hex to start with"), climate hardcoded `"hot"` (the `COLD_TERRAIN_TABLE` in `src/data/hexTables.ts` is fully authored but nothing selects it yet). `src/engine/hexReducer.ts` is its own small state machine (not an extension of `dungeonReducer.ts` -- different persisted lifetime, no combat) with a single `MOVE` action, validated against `hexNeighbors()` and `isImpassable()` (water, or a `rocks` location) before mutating `player`/re-revealing that hex's own neighbors. `src/ui/screens/WorldScreen/WorldScreen.tsx` renders the map as an SVG (axial-to-pixel projection, hex polygons colored by terrain), with only City/Fortress/Ruins/Rocks locations interactive today -- Portal/Oasis/Volcano/Reef/Thin Ice/"nothing" render as inert flavor labels, their actual entry-roll effects not implemented yet. `App.tsx`'s `Screen` is just `"world" | "dungeon"` (plus Character Creation, gated separately) -- Town isn't its own screen, see Town above; `world: WorldState | null` persists across characters like `dungeonHistory`, created (or player-position-reset to `world.home`) in `handleCharacterCreated`.

- **Provisions**: a new `AdventurerResources` field, parallel to torches but a genuinely separate resource (the rulebook keeps them distinct -- travel/food vs. dungeon light) -- `MAX_PROVISIONS` (20), `canBuyProvision`/`buyProvision` (`src/engine/town.ts`, same 1-coin shape as `buyTorch`), and `payTravelCost()` (spends provisions first; any shortfall at all costs a flat 1 HP for the move, not scaled to how many provisions were actually short -- a Mountain move at 0 provisions costs the same 1 HP as a Plains move at 0 provisions, not 3x as much just because Mountains normally cost more provisions -- deliberately floored at 1 HP besides, so running out of provisions alone can never kill the character, unlike the rulebook's real lethal hex hazards, which are part of the still-deferred Events on Travel/location-effects layer below).
- **Terrain-based Dungeon Type**: `DUNGEON_TYPE_BY_TERRAIN` (`src/data/hexTables.ts`) maps each terrain to a 1d6 table of `DUNGEON_TYPES` roll numbers ("Table: Dungeon Type, by terrain") -- the rulebook's real table references 8 dungeon types this app hasn't built yet (Citadel, Mine, Cave, Laboratory, Pyramid, Ziggurat, Necropolis, plus a Fortress's optional Sewers -- tracked as issue #30, "Deadly Dungeons"), each substituted with the closest thematic match among the 6 that exist (documented on the table itself), same precedent as `bladeTrap`'s flavor-only roll-of-2 or `WeaponEntry.twoHanded`. `App.tsx`'s `onEnterDungeon` handler rolls the current hex's terrain-fated type once and stores it as `forcedTypeRoll`, passed to `DungeonScreen` unconditionally now (every dungeon trip originates from World) -- `handleRollDungeon` uses it for just the first of its 3 dice (`forcedTypeRoll ?? rollDie()`), so the "Roll for Dungeon" ritual and its 2 free name-flavor dice stay intact; only the type itself is fated by where the player found it, not free chance.
- **Town Square unification**: `WorldScreen` renders `TownScreen` in full (not a sidebar card) whenever standing on a City/Fortress hex and not voluntarily viewing the map (`showMap` local state, reset to `false` on every `handleTravel` so arriving anywhere shows "the appropriate thing" -- the city screen if it's a City/Fortress, the map otherwise). `TownScreen` itself stays hex-agnostic (its one exception being `cityName`, see below): it takes `hasDungeon` (`WorldScreen`'s own `canEnterDungeon` check -- true for a City/Fortress/Ruins regardless of whether a dungeon's been found there yet, covering both a fresh roll and a known one, but *false* once `currentDungeonStatus === "beaten"` -- there's nothing left to do in an already-cleared dungeon, and `RETURN_TO_DUNGEON`/`RESUME_DUNGEON` would just redisplay the existing victory panel rather than letting the Boss be re-fought), `dungeonGateCopy`, `onEnterDungeon`, and `onExploreWorld` (sets `showMap(true)`) -- City Actions (Rest/Buy Torches/Buy Provisions/Fix/Sell) live only in `TownScreen` now, reusing `town.ts`'s pure functions exactly as before. While voluntarily viewing the map from inside a city (`showMap` true), the map view shows a "Return to the City" control (`showMap(false)`) instead of its own redundant Ruins-style "Enter Dungeon" card -- that card (`canEnterDungeon && !inCityOrFortress`) is reachable only for a genuine Ruins hex, which stays map-view-only, no City Actions, matching "Explore as if it were a dungeon" rather than a settlement.
- **City/Fortress names** (issue #49): every Human City used to be indistinguishable from every other Human City except by position -- `HexTile.name` (`hexState.ts`, optional for back-compat with a pre-#49 persisted session) fixes this the same way dungeons already got proper names (`dungeonTypes.ts`'s `DUNGEON_NAME_SECOND`/`DUNGEON_NAME_THIRD`): `rollCityName()` combines two 1d6 rolls (prefix, suffix) from a new per-culture `src/data/cityNames.ts` table into one compound word (e.g. "Iron" + "hold" -> "Ironhold"), rolled once inside `revealNeighborsInPlace()` itself, in the same pass that decides the hex's `location` -- a City/Fortress location and its name are set together, atomically, never separately. Not from the rulebook, mirroring `names.ts`'s per-race character-name generator (issue #40) -- "a lot of small flavor tables, keyed by race/culture," just for settlements instead of people. Home is the one exception: its name ("Haven") is fixed rather than rolled, matching the "fixed, not rolled" precedent its terrain/location already have (`createInitialWorldState()` doesn't go through the rulebook's own hex-reveal roll for the starting hex either). Surfaced everywhere the generic `LOCATION_LABEL` used to be the only identity available: `WorldScreen`'s map hex label and its `currentPlaceLabel` (used for `dungeonGateCopy`'s prefix, and as the Graveyard's `dungeon`/"place" field for a Gamble/Thug Life/Arena death -- an "Ironhold" death now reads better than a generic "Human City" one), `HexInspector`'s new `cityName` prop (takes over the panel's headline in place of `locationLabel`, which still shows underneath in its own Location row -- both are shown, not one replacing the other), and `TownScreen`'s header, where the city's name now takes the prominent `.sheetLabel` heading spot and "Town Square" demotes to a small eyebrow above it (mirroring `DungeonScreen`'s own `.dungeonEyebrow`/`.dungeonName` pairing for the generated dungeon name) -- this is the one place `TownScreen` isn't fully hex-agnostic anymore, via a plain `cityName: string` prop `WorldScreen` computes and passes down (reusing the same `currentPlaceLabel` it already needed for `onCharacterDied`, not a second parallel computation).
- **Per-hex dungeon persistence**: `HexTile.dungeonRunId` (`src/engine/hexState.ts`) remembers which `PendingDungeon` was found on a given hex, "drawing the dungeon on the map" instead of re-rolling a fresh one every visit -- set once, on first entry, never cleared even once beaten. World's `onEnterDungeon` (`App.tsx`) checks the current hex's tile first: if `dungeonRunId` is already set, it just sets `selectedRunId` to it, letting the existing `isOwnRun`/`resumeDungeon` derivation that already renders `DungeonScreen` pick `RETURN_TO_DUNGEON` (still this character's own paused run) vs. `RESUME_DUNGEON` (someone else's abandoned one) -- no World-specific resume logic needed. On a genuinely first visit, the terrain-fated roll above still happens, but the run's id is minted in `App.tsx` itself (`worldFreshRunId`, passed to `DungeonScreen` as `externalRunId`, used only when `activeDungeon`/`resumeDungeon` are both unset) rather than `DungeonScreen` self-minting one via `crypto.randomUUID()` on mount -- World needs to know the id up front to stamp it onto the hex immediately, which is always safe since there's no way to leave `DungeonScreen` before a dungeon actually exists (its Retreat button only renders once `hasDungeon` is true, a *different*, `DungeonScreen`-local `hasDungeon` than `TownScreen`'s prop of the same name). `dungeonGateCopy` ("a dungeon awaits here" / "your unfinished dungeon is still here" / "the dungeon here has already been cleared") comes from a `dungeonInfoFor()` lookup (`dungeonHistory` + `isDungeonBeaten()`/`hasUnlootedRemains()`) reused per-hex on the map itself -- a small ⚔/✓ SVG badge renders on any hex with a known dungeon (cleared vs. not, visible at a glance without standing on it), plus an independent 💀 badge whenever `hasUnlootedRemains()` is true (a dungeon can be beaten by someone else while an earlier death's loot still sits unrecovered in a different room, so this is checked separately from beaten-status, not folded into it). A hex's dungeon, once drawn on the map, stays that hex's dungeon for good -- there's deliberately no "abandon this one and roll a different one here instead" action (a "Start a New Dungeon" button briefly existed and was removed -- besides having no combat guard, letting a player re-roll a hex's dungeon on a whim contradicted the whole point of a dungeon being a persistent, findable place other characters can resume too).

- **`HexInspector`**: `WorldScreen`'s equivalent of the dungeon map's `RoomInspector` -- a `.hexInspectorOverlay` card, positioned/styled byte-for-byte like `.roomInspectorOverlay` (`position: absolute; right; bottom` on the now-`position: relative` `.mapCard`), describing whichever hex is selected (terrain, location, `dungeonInfoFor()`'s status/remains). Read-only for travel: clicking a hex on the map (`WorldScreen.tsx`'s `handleHexClick`) still travels immediately when it's a passable, in-range neighbor -- unchanged from before `HexInspector` existed -- and only *falls back* to selecting it for inspection (local `selectedHex: HexCoord | null`, mirroring `RoomInspector`/`state.selectedSegId`'s selected-vs-current split) when it isn't a valid travel target, since there'd be nothing else useful for that click to do. `selectedHex` falls back to the player's own tile when `null`; `handleTravel` resets it to `null` so arriving anywhere describes the new tile by default. It does have one action button, though (issue #59): "Enter Dungeon" for a non-City/Fortress hex (Ruins, or a plain hex "Ask" found -- `HexTile.dungeonMarked`) moved from a separate `.actionCard` that used to sit below the whole map into this panel itself, right next to the Dungeon status row it already showed -- a report that the map's dungeon-badged hex looked like it had "no way to enter it" turned out to mean the entry point existed but lived somewhere disconnected from the status describing it, not that it was actually missing. Gated on `isCurrentTile && canEnterDungeon` (both props `WorldScreen` already computed for the old card, just threaded one level deeper) -- you can only ever enter a dungeon on the hex you're standing on, so the button only shows while `HexInspector` is describing that hex specifically, not some other one merely being inspected; `canEnterDungeon` itself still excludes City/Fortress hexes (`&& !inCityOrFortress`, preserved from the old card), since those keep their own single "Enter Dungeon" entry point in `TownScreen`'s Adventure section, even while voluntarily viewing the map from inside one.
- **Map zoom + pan** (`src/ui/hooks/useZoomGesture.ts`): a shared gesture-*detection* hook (wheel + two-finger pinch, normalized to a single `onZoom({ factor, clientX, clientY })` callback) used by both `DungeonMap` and `WorldScreen`'s hex map -- deliberately not shared on the *application* side, since the two maps zoom completely differently. `DungeonMap` applies `factor` as a CSS `transform: scale()` on `.canvas` (clamped `[0.5, 2.5]`), correcting `scrollLeft`/`scrollTop` in a `useLayoutEffect` keyed on `scale` so the cursor's content point stays put; a "Reset Zoom" button (top-right, so it doesn't collide with `.roomInspectorOverlay` in the bottom-right) appears whenever `scale !== 1`. `WorldScreen` instead resizes/repositions the SVG `viewBox` itself (`viewBoxOverride: {x,y,w,h} | null`, `null` = today's auto-fit-everything), converting the client-space focal point to SVG user-space via `svgRef.current.getScreenCTM()!.inverse()` (correctly accounts for `preserveAspectRatio` letterboxing) and clamping width between ~4 hexes wide and 1.5x the natural full-fit size -- a matching "Reset View" button clears the override. Since #33 made every known hex clickable (not just neighbors), zooming needed click-drag panning too (there's no native scroll on an inline SVG the way `DungeonMap`'s `.scroll` div gets) -- it reuses `DungeonMap`'s exact `didDrag`/pointer-capture pattern (a moved-past-threshold pointer-down is a pan, not a click; a capturing click handler swallows the resulting click) so panning and hex-selection coexist without either one accidentally triggering the other. One gotcha specific to `WorldScreen`: its very first render shows `TownScreen` (the `<svg>` doesn't exist yet, so `svgRef.current` is `null` when `useZoomGesture`'s mount effect first runs), and only mounts the map later in the *same* component instance once `showMap` flips true -- a `RefObject`'s identity never changes, so a `[ref]`-only effect dependency would attach its listeners exactly once, while `ref.current` is still `null`, and never notice the DOM node arriving later. `useZoomGesture` fixes this generally (both consumers benefit) by re-running its attach effect on every render instead of gating on `[ref]`.

A hex-tied dungeon is resumable **only** by physically traveling back to its hex -- `TownScreen` has no dungeon-finding beyond whatever `WorldScreen` hands it for the *current* hex (see above), so a fresh character can't jump straight into a predecessor's hex-found dungeon from some other city; they have to actually go find the hex again, the same as they'd have to the first time. This applies without exception, including to the current character's own still-active paused run: an earlier increment added a character-scoped "Continue the Dungeon" shortcut (`activeDungeonElsewhere`/`onContinueActive`) that warped `world.player` to the run's hex from anywhere, directly contradicting this rule -- removed (#44) rather than kept, since the rule itself is the intended design, not the shortcut.

**Still stubbed / not yet built** (Hexploring the World): Events on Travel (the 2d6 wilderness monster/weather table for non-location hexes); Portals (a 3d6 destination table) and the other inert `LocationKind`s' entry-roll effects (Reef's aground/Underwater Cave check, Oasis's mirage/heal check, Volcano's Volcanic Cave, Thin Ice's death check); Ruins' own 2d6-by-terrain dungeon table and its unique Entrails/Mega Dungeon outcomes (closer in scope to issue #30, "Deadly Dungeons," than to #20); the Fortress-Sewers sub-roll (references the not-yet-built Sewers dungeon type, also #30).

**Affinity, Culture Actions, and Hire Boat** (issue #20 Stage 1 -- `docs/game-rules-reference.md` lines 928-1089): the first three City/Fortress systems from that stub list, now built. `src/data/affinity.ts`'s `hasAffinity(raceName, location)` implements "Table: Affinity" verbatim (14 rulebook rows, all 11 playable races matching one exactly) via `RACE_AFFINITY`/`CULTURE_BY_LOCATION`/a `DEFAULT_AFFINITY` fallback for the "Other race..." row -- a race with no Affinity for a hex's culture can't even travel there: `hexReducer.ts`'s `MOVE` action (which now also takes a `raceName`) rejects it the same tier as `isImpassable()` already does for water/Rocks, and `WorldScreen.tsx` folds both checks into the one `canTravelTo()` gate used by both click-to-travel and the neighbor-highlighting logic; `HexInspector` explains why with a "You are not welcome here" note when inspecting a hex you can't reach. Six new `town.ts` functions implement "Different Cultures"' one bonus City Action per culture (Human/Dwarf/Elf/Gnome/Goblin/Orc) -- several reference systems this codebase doesn't have (Curses, hand-economy, day-passage), which resolve as flavor-only coin sinks per the project's established precedent (`bladeTrap`'s roll-of-2, `WeaponEntry.twoHanded`) rather than triggering new engine work; Elf's Elven Boots is the one exception with a real mechanical effect (forest travel drops to 1 provision), implemented as `hasElvenBoots()` checking for a case-insensitively-matched `itemName` on an equipped armor piece -- the same substring-match precedent Armor & Weapons' tag-matching already established -- rather than a new `ItemEffect` kind, since it's a World-map concern, not a dungeon-combat one. `TownScreen.tsx` renders whichever one bonus action matches the current hex's culture (`cultureActionFor()`, `culture: CityCulture | null` computed by `WorldScreen.tsx` and passed down, keeping `TownScreen` exactly as hex-agnostic about culture as it already was about which specific city). Hire Boat adds a `WorldState.hasBoat` flag (a new `HIRE_BOAT` hex action, validated server-side same as everything else in `hexReducer.ts` -- must actually be standing in a City/Fortress adjacent to a water tile) that `isImpassable()` now takes as an optional third parameter to lift the water restriction specifically; `MOVE` clears it automatically the instant the player lands on non-water terrain again, matching "once you enter non-water terrain you leave the boat" exactly.

**"Ask"** (the last of issue #20's original City Action list): "you can ask where the nearest dungeon to town is ... roll 1d6 ... in this hex there will be a dungeon ... if the hex has Water or another City or Ruins, go clockwise to the next hex until you find land that has no location and isn't water." Unlike every other dungeon-bearing hex so far, the found hex is deliberately *not* the one the player is standing on -- `HexTile.dungeonMarked` (`hexState.ts`) is a new, separate flag from `dungeonRunId` for exactly this: it "draws the dungeon on the map" the instant Ask finds it, well before the player has ever traveled there or a `PendingDungeon` exists for it (that still only happens once "Enter Dungeon" is later used on that hex, via the same first-entry flow every other dungeon hex already goes through -- `dungeonMarked` only ever widens *which* hexes can offer that button, it never itself mints a run). `findAskedDungeonHex()` walks `HEX_DIRECTIONS` clockwise from a rolled starting side across the current hex's own six neighbors (already revealed, since a neighbor of any entered hex always is) for the first that's land with no location, mirroring the rulebook's skip-and-continue rule exactly; `withDungeonMarked()` is `withDungeonRunId()`'s immutable-stamp sibling. `hexReducer.ts`'s new `ASK_FOR_DUNGEON` action is the actual authority (same "reducer decides, UI just offers the button" pattern as `HIRE_BOAT`): it no-ops outside a City/Fortress, and no-ops again if any neighbor already carries a `dungeonRunId` or `dungeonMarked` ("if you don't already have a dungeon in any adjacent hex, roll 1d6") -- `WorldScreen.tsx`'s own `askedDungeonKnown` mirrors that same neighbor check purely so `TownScreen`'s "Ask" button can render disabled with an explanatory line instead of just silently failing, the same always-visible-but-disabled precedent Spellcasting's "Cast" button already established. Introducing `dungeonMarked` meant widening every check that used to assume "dungeon-bearing" meant `locationHasDungeon()` (a City/Fortress/Ruins location): `canEnterDungeon` now also passes for a `dungeonMarked` hex with no location at all, and the map-view action card (previously gated on `currentTile?.location` to build its label) now falls back to plain gate copy with no location prefix when there isn't one. `dungeonInfoFor()` grew a fourth status, `"found"` (dungeonMarked but no `dungeonRunId` yet), distinct from `"unfinished"` (a `dungeonRunId` actually exists, i.e. the player has actually used Enter Dungeon here at least once) -- collapsing the two would have made the gate copy claim "your unfinished dungeon is still here" for a hex nobody has ever set foot on yet; both still render the same ⚔ map badge and both still satisfy `canEnterDungeon`, they only differ in copy (`HexInspector`'s `DUNGEON_STATUS_COPY` gained the same fourth line).

**"Getting Money"** (issue #58, split off from #20 -- `docs/game-rules-reference.md` lines 1025-1071): all four mini-games for a broke character are built. Hard Work/Gamble/Thug Life live in `town.ts` alongside the "Different Cultures" functions, same file/shape precedent; Arena gets its own `arena.ts` (see below). Hard Work is the one action in this whole "Cities and Fortresses" section the rulebook restricts to City only (everything else -- Rest/Buy/Sell/Fix/Ask/Hire Boat/Gamble/Thug Life -- explicitly says "city or fortress"), so it's the first thing here that needed a City-vs-Fortress distinction at all: `isFortressLocation()` (`hexTables.ts`, a plain suffix check against the four Fortress `LocationKind`s -- Goblin/Gnome never roll one) is threaded down as `WorldScreen`'s own `isFortress` prop; `TownScreen` doesn't render the Hard Work button at all when it's true, and *only* renders "Fight in the Arena" when it is. "Permanently lose 1 HP" -- unlike every other HP loss in the game, Rest can't undo this, so `hardWork()` takes it out of `maxHp` directly (current `hp` clamped down to match if it was already at the old max), guarded so `maxHp` can never reach 0 (the rulebook doesn't list this as lethal, unlike Gamble/Thug Life/Arena). Gamble is really two sub-games behind one button, selected by `resources.coins` itself (never a separate action to pick, matching "if you don't have money, you can bet your life" framing exactly): `gamble()` returns a `GambleResult` with an `outcome` of `"won" | "lost" | "survivedLifeBet" | "diedLifeBet"` rather than mutating `resources` unconditionally, since a `diedLifeBet` result leaves `resources` completely untouched -- `town.ts` has no notion of "the character is dead," that's App.tsx's job alone (the only place holding both `character` and the authority to clear the whole session).

**Thug Life** resolves the whole "Table: Stealing from Travelers" roll in one `resolveThugLife()` call (`town.ts`) -- 2d6 in a City, 3d6 in a Fortress -- returning a `ThugLifeResult` with `died`/`banned` flags alongside the usual updated-`resources`, mirroring Gamble's death-signaling split exactly. The 5-7 "jail" row is the one outcome that can go two different ways from a single follow-up roll: "lose 1d6 HP... if you are still alive, you have fled" -- if that HP loss alone would drop the player to 0, `died: true` fires (`resources` untouched, same as `diedLifeBet`); otherwise `banned: true` comes back alongside the HP already deducted. 15-18 ("1 Crypt/Sanctuary/Palace Treasure!," only reachable on 3d6 -- a City's 2d6 tops out at 12) flattens to a generic `treasures` credit, same flavor-simplification precedent as `bladeTrap`'s roll-of-2 or `DUNGEON_TYPE_BY_TERRAIN`'s thematic substitutions, since this codebase has no per-dungeon-type Treasure concept to hand out one of specifically. The ban itself is **world-scoped, not character-scoped** -- a deliberate call (see the issue) matching this project's existing "the world remembers" pattern (dungeonHistory, discovered hexes) rather than inventing a new per-character state bucket that doesn't exist anywhere else: `WorldState.bannedHexes` (`hexState.ts`, optional for back-compat with a pre-#58 persisted session, always read through `isBannedHex()` so the `?? []` default lives in one place) is checked by `hexReducer.ts`'s `MOVE` case the same tier as `isImpassable()`/`hasAffinity()` already are, and by `WorldScreen.tsx`'s own `canTravelTo()` gate for the UI side. Since resolving Thug Life can touch *both* `resources` (coins/Treasures/HP) and `WorldState` (the ban) from the one roll, it's resolved once in `WorldScreen.tsx`'s `handleThugLife()` (not `TownScreen`, which only ever sees an `AdventurerResources`) and applied to whichever of the two actually changed; `TownScreen` calls it via an `onThugLife: () => ThugLifeResult` prop purely to get back what happened, so it can turn the outcome into copy (a `thugLifeMessage` local state swapping the button's own description text, same "always visible, changes what it says" precedent Ask's `askedDungeonKnown` already established) -- this is also why `TownScreen`'s "banned" copy specifically calls out the consequence: without it there'd be no way to learn why a city visible on the map suddenly refuses entry. The player is deliberately **not** forcibly evicted the instant they're banned -- "you have fled" only blocks *future* entry (enforced by `MOVE`), this visit continues normally, since the rulebook gives no relocation target and forcing one would be new, unscoped behavior.

**Fighting in the Arena** (Fortress-only) gets its own `src/engine/arena.ts` rather than reusing `dungeonReducer.ts`'s `CombatState`, since that type is shaped around a dungeon segment (`segId`), Loot rolls, and the armor-absorption "your call" choice Arena has none of -- but it *does* reuse `combat.ts`'s pure `resolvePlayerAttack()`/`resolveMonsterTurn()` directly, since those already operate on a bare `CombatMonsterState` with no `DungeonState` dependency at all. `ArenaState` is just `{ champion: CombatMonsterState; outcome: "ongoing" | "victory" | "defeat" }` -- deliberately simpler than a dungeon fight: no armor-absorption choice (all damage hits `hp` directly, the Arena section never mentions armor the way the Core Book explicitly does), no spellcasting or race/class abilities (Town/World has no combat-spell infrastructure outside a dungeon run to extend), and no equipped weapon `bonusEffect` (just the base formula) -- same "documented, deliberate simplification" tier as `bladeTrap`'s flavor-only roll-of-2. This still covers every ability an Arena Champion actually has (Deathtouch/Stoneskin/Poison/Weakness/Explosive, per `src/data/arena.ts`'s `ARENA_CHAMPION_TABLE`) correctly, since `resolvePlayerAttack` already handles all of them -- Firebreath/Sorcery/Horde/Necromancy/Regeneration/Paralyze/Undead genuinely don't need wiring, since no Champion on the table has any of them (same "won't build what can't currently be reached" precedent as `DUNGEON_TYPE_BY_TERRAIN`'s unreachable water/glacier rows). `resolveArenaRound()` resolves one full round at a time (player attacks, champion counters immediately if it survived -- Arena has no "noisy start," the player always swings first) and is the one place that needed a bugfix mid-implementation: Explosive can defeat the champion *and* kill the player in the same blast, and the death check has to run **before** the victory check even though `atk.monsterDefeated` is already known -- verified against `dungeonReducer.ts`'s own PLAYER_ATTACK case ("the explosion kills you instantly" fires first, even when the monster is also defeated by the same hit) and a dedicated test case, since checking them in the wrong order would have let a player "win" a fight that also killed them. `TownScreen.tsx` owns the fight's UI entirely as local state (`arena`/`arenaLog`, reset only by starting a new fight) rather than a shared component, swapping in for the whole City Actions section while a fight is ongoing (mirroring `DungeonScreen` swapping `CombatPanel` in for `RoomInspector`) -- deliberately un-animated (no dice roll, just an instant per-click round resolution) to match Hard Work/Gamble/Thug Life's own presentation rather than the dungeon `CombatPanel`'s heavier `Die`-driven one. Victory credits 20 coins directly; defeat calls the same `onCharacterDied("arena", place)` path Gamble/Thug Life use.

**Death outside a dungeon**: Gamble's life-bet, Thug Life, and Arena are the first ways to die anywhere other than `DungeonScreen`, which had no equivalent path at all before issue #58 -- every earlier Graveyard write lived entirely inside `DungeonScreen`'s own `state.alive`-flip effect. `GraveyardEntry.causeOfDeath` widened from the closed `"darkness" | "combat"` union to also include `"gamble" | "thug-life" | "arena"` (`graveyard.ts` exports the non-dungeon subset as `TownDeathCause`, so callers outside `DungeonScreen` don't need to know the full union); `dungeon` -- previously always a dungeon name -- doubles as "where" for these, holding the city/fortress's location label instead (a `GraveyardEntry` shape choice, not a rename, since the field still reads naturally as "place" either way). `App.tsx`'s new `handleTownDeath(cause, place)` is Town/World's own equivalent of that `DungeonScreen` effect -- called directly the instant a losing roll comes back (there's no `DungeonState`/`useEffect` to key off here), it writes the Graveyard entry then reuses `handleNewAdventurer()` verbatim to clear the session. `place` is supplied by `WorldScreen` (already computing a location label for `dungeonGateCopy`) rather than App.tsx keeping its own copy of `LOCATION_LABEL`; `TownScreen` itself only ever calls a pre-bound `onCharacterDied(cause)`, unaware of place at all.

**`Die` component** (`src/ui/components/Die/Die.tsx`) is a real 3D CSS cube (six absolutely-positioned faces inside a `perspective` container, `rotateX`/`rotateY` per face), not an image or icon-swap. It's driven by a `value` (1-6) + `rollToken` (bump this to trigger a new roll animation even if the value repeats) + optional `delayMs` (for staggering dice in a pool) + `size` prop. The rotation math (`SHOW_ROTATION` table + `angleTo` helper) picks the shortest forward rotation to the target face and adds a couple of extra full turns for a tumble effect — see the component for the derivation. `DicePool` just renders N `Die`s with staggered delays for 2d6/etc. rolls. Screens own the *timing* of when to reveal a roll's result (see `revealDelay()` in `CharacterCreationScreen.tsx`) — the Die component itself doesn't expose a completion callback.

**CSS Module class names** (issue #57): `vite.config.ts`'s `css.modules.generateScopedName` is set to `"[name]__[local]"` instead of Vite's default opaque-hash format (`_die_1ewh6_1`) -- `[name]` resolves to the CSS file's own basename including the `.module` suffix (e.g. `Die-module__die`, since every `*.module.css` file here is named after the component it styles and no two share a basename, this stays collision-free without needing a hash at all). Applied uniformly to dev *and* the production build -- unlike a bundle-size-sensitive production app, there's little reason to keep the terser hashed format only in production for a small hobby project, and a readable class name in prod devtools is exactly the point.

**Versioning**: every push to `main` requires a version bump — `package.json`/`package-lock.json`'s `version`, a new dated section in `CHANGELOG.md`, and a matching `vX.Y.Z` git tag on the commit that lands it. Versions are `Major.Minor.Hotfix`: Hotfix for a bug fix with no user-visible feature change (e.g. 1.0.0 → 1.0.1), Minor for a new feature, Major for a significant milestone (e.g. the Core Book's full rule set landing as 1.0.0) or a breaking change. **Spellcasting** (`DungeonState.spellUses`, a `Record<string, remainingUses>` seeded at dungeon creation by `computeSpellUses()` from the character's `spells`/`fixedGrants` -- keyed by `character.ts`'s composite `spellKey(table, roll)` string rather than a bare roll number, see New Spells below for why) covers all six Basic Spells via the `CAST_SPELL` action, which now also carries a `table: SpellTableKey` field alongside `spellRoll`. Heal and Light are usable anytime (in `CombatPanel`, as a "Cast" button on `CharacterSheet` while inside a dungeon out of combat, or -- via `town.ts`'s own `canCastSpell()`/`castSpell()`, since Town/World have no `DungeonState`/reducer to dispatch `CAST_SPELL` against -- as that same "Cast" button in `TownScreen`/`WorldScreen` too, mirroring `CAST_SPELL`'s Heal/Light math exactly: `min(HEAL_AMOUNT, maxHp - hp)` and `min(1, MAX_TORCHES - torches)`); Cold Ray, Lightning, and Fireball require an active fight and are rendered as extra per-monster or room-wide buttons in `CombatPanel`. `combat.ts`'s `OUT_OF_COMBAT_SPELL_NAMES` set (`{"Heal", "Light"}`) is the single source of truth for which spells this applies to, matched by spell **name** rather than roll number -- `CharacterSheet`/`town.ts` both import it rather than keeping their own copies (see New Spells below for why name-matching, not table+roll, is how `CAST_SPELL` decides what's real). The "Cast" button itself always renders for an eligible spell, `disabled` (not omitted) at 0 remaining uses -- it used to disappear entirely once spent, which read as "not supported here at all" rather than "just out of uses right now," the same class of ambiguity #35's Rest-button fix already addressed elsewhere. Spell damage runs through `resolveSpellDamage()` in `combat.ts`, which shares Stoneskin/Intangible filtering with `resolvePlayerAttack()` but — since it's a fixed value, not a weapon "damage roll" — never triggers the roll-of-1/roll-of-6 abilities (Explosive, Firebreath, Weakness, etc.). Casting any spell mid-fight consumes the round exactly like `PLAYER_ATTACK` (the monsters counter-attack afterward), except Cold Ray also flags its target `skipNextAttack` so it sits out that round, and Teleport ends combat immediately with no counter-attack. A spell's uses, once spent, stay spent within a dungeon run — recovering them requires the Town's Rest action (see Town below).

**New Spells** (Expanded World, issue #24): three more 1d6 tables (`NATURE_SPELL_TABLE`, `DEATH_SPELL_TABLE`, `ELEMENTAL_SPELL_TABLE`) plus a 2d6 `ADVANCED_SPELL_TABLE` (`src/data/spells.ts`), none of them a player choice the way Race tables are -- access is always *granted* by something else (a race ability, an Advanced Class, or a Magic Item), never rolled freely at Character Creation. Since these tables reuse the same roll numbers (1-6, or 2-12 for Advanced) as Basic Spells and each other, every spell-identity site in the codebase widened to a composite key: `SpellDef`/`FixedSpellGrant` both carry a `table: SpellTableKey` field alongside `roll`/`spellRoll`, and anywhere spell uses are tracked (`DungeonState.spellUses`/`maxSpellUses`, `AdventurerResources.spellUses`/`maxSpellUses` -- see issue #75's note below for why the max needs its own persisted field too) is keyed by `character.ts`'s `spellKey(table, roll)` string (e.g. `"nature:2"`) rather than a bare number -- `parseSpellKey()` is the inverse, used by the UI (`CombatPanel`, `CharacterSheet`) to resolve a key back into a real `SpellDef` via `SPELL_TABLE_BY_KEY`. `rollSpellFromTable(table, rng)` is the one roll function for all five tables (Advanced alone sums 2d6, same shape as `rollRace()`/`rollClass()`; every other table is a flat 1d6) -- `rollSpell()` is now just `rollSpellFromTable("basic", rng)`. `computeSpellRequirements()`'s `randomSlotsByTable: Partial<Record<SpellTableKey, number>>` (was a single flat count) exists because a race and class can rarely both grant random spells from *different* tables at once -- Corvino's "5 random Advanced Spells" (`randomSpellsTable: "advanced"`) alongside a Scholar/Noble class's own random-Basic grant, which always lands under `"basic"` regardless of a race's table (`ClassDef` has no table field of its own).
- **Why `CAST_SPELL` dispatches by spell name, not `(table, roll)`**: several New Spells rows are the *identical* Core spell re-listed under a different table (Elemental's Cold Ray/Lightning/Fireball, roll 3/4/5, are word-for-word Basic's roll 4/5/6) -- `combat.ts`'s `KNOWN_CASTABLE_SPELL_NAMES`/`TARGETED_SPELL_NAMES`/`COMBAT_ONLY_SPELL_NAMES`/`OUT_OF_COMBAT_SPELL_NAMES` sets are matched against `SpellDef.name` so both copies share the one real implementation in `dungeonReducer.ts`'s `CAST_SPELL` case, rather than needing a duplicate `case` per table. A spell whose name isn't in `KNOWN_CASTABLE_SPELL_NAMES` is simply never offered a "Cast" button anywhere -- not clickable-and-silently-doing-nothing, just absent, the same "documented, deliberate simplification" precedent as `bladeTrap`'s roll-of-2 or `WeaponEntry.twoHanded`.
- **What's mechanically real vs. deferred**: Heal, Light, Teleport, Cold Ray, Lightning, Fireball (Basic's original six, plus Elemental's three duplicates of Cold Ray/Lightning/Fireball), and -- as of issue #61 -- Natural Cure, Insect Rain, Magic Blast, Banish the Dead, Vimes, Paralyze, Ethereal Body, Magic Shield, Absorb Soul, and Fire of the Dead all do something real when cast. Every other New Spells effect (Camouflage, Create Food, Summon Wolf, Summon Skeleton, Awakening, Summon Elemental, Stone Armor, Collapse, Open Portal, Fly, Reload Mana) is rolled, tracked, and displayed correctly (name/effect text/uses remaining all resolve through the real tables) but has no "Cast" button and no mechanical effect yet. Several reference systems this codebase doesn't have at all -- an ally-in-combat concept for every Summon spell, a temporary/expiring armor concept for Stone Armor, a design decision for Collapse, the not-yet-built Events on Travel/Portals systems (`docs/game-rules-reference.md`'s own still-stubbed list, see Hexploring the World above) -- so building them out is follow-up scope, not a gap in this pass.
- **Natural Cure/Insect Rain/Magic Blast (issue #61, Tier 1)**: each reuses an existing `CAST_SPELL` case's exact shape with only the fixed amount changed -- Natural Cure (Nature 1, 12 HP) mirrors Heal; Insect Rain (Nature 6/Advanced 2, 7 room-wide damage) mirrors Fireball; Magic Blast (Advanced 10, 12 single-target damage, no freeze) mirrors Lightning. Natural Cure is also usable out of combat (added to `OUT_OF_COMBAT_SPELL_NAMES` alongside Heal/Light) -- `town.ts`'s `castSpell()` needed an explicit `"Natural Cure"` branch of its own, since its previous `if (name === "Heal") {...} else {/* assume Light */}` shape would have otherwise silently granted a torch instead of healing the moment Natural Cure became castable there too. **Banish the Dead** (Death 3, "Destroy any Undead in the same area") is the one genuinely new case: filters `combat.monsters` by the `undead` ability and destroys each one outright via `handleMonsterDefeat()`'s new `bypassRevival` parameter -- a decisive banishment shouldn't let the Undead ability's own roll-of-1 revival save a monster the spell says it destroys.
- **Vimes/Paralyze/Ethereal Body/Magic Shield/Absorb Soul/Fire of the Dead (issue #61, Tier 2)**: six spells needing genuinely new `CombatState`/`CombatMonsterState` fields, all cast mid-fight and resolved the instant they're cast, exactly like every other spell (the round still ends with the monsters' own counter-attack immediately afterward -- casting one of these doesn't skip that). **Vimes** (Nature 2, targeted, "leaves a monster without attacking for 1d6 turns") and **Paralyze** (Advanced 5, room-wide, fixed 2 turns) both set a new `CombatMonsterState.silencedTurns` field -- a multi-round version of Cold Ray's single-round `skipNextAttack` -- decremented once per round in `applyMonsterTurn()`, which skips that monster's damage contribution entirely (not just this round, every round until it reaches 0) rather than clearing after one skip. **Ethereal Body** (Death 1/Advanced 12, "all damage you take is reduced by 1 point... until the end of the fight") sets a new `CombatState.damageReduction` field (doesn't stack from a second cast), applied per monster hit in `applyMonsterTurn()` before the poison/absorbable split -- deliberately reduces poison damage too, reading "all damage" literally, unlike armor/Magic Shield which poison already bypasses by name ("cannot be absorbed by armor or other means"). **Magic Shield** (Advanced 8, "can absorb 4 damage points. Can cast more than one") pushes onto a new `CombatState.shields: number[]` array -- each cast is its own independently-depleting pool, drained oldest-first (and dropped once empty) against `absorbableDamage` in `applyMonsterTurn()`, *before* the existing armor-or-HP choice is even offered, so a fully-absorbed hit never triggers `RESOLVE_DAMAGE` at all; poison bypasses shields the same way it bypasses armor. **Absorb Soul** (Death 2, "after a fight, recover 5 HP for each monster killed") and **Fire of the Dead** (Death 4, "...you get 2 torches for every monster killed") are the two deferred-to-victory triggers: each sets its own boolean flag (`combat.absorbSoulActive`/`fireOfTheDeadActive`) when cast, paid out by `finishIfVictorious()` off the *same* `combat.engulfableBodies` count Slimemen's Engulf Body ability already tracks (every monster actually killed this fight, not revived) -- reused rather than tracked twice, and applies to a Boss victory too since neither spell's text carves out an exception for one.
- **Spell uses granted after Character Creation, and the Rest bug (issue #75)**: found while verifying #61's new spells in a browser, unrelated to that issue -- `CharacterSheet`'s Spells list, `rest()`, and `OPEN_TREASURE`'s Mana Potion all used to compute a spell's *max* uses via `computeSpellUses(character.spells, character.fixedGrants)`, data fixed at Character Creation and never updated afterward. Any spell granted *later* -- every Advanced Class spell grant above, Hirelings' Rent Wizard/Elf Soldier/Gnome Helper, Gnome's own `learnRandomSpell()` Culture Action, or a Magic Scroll -- had no entry in that computed map at all: it never appeared in the Spells list, and worse, `rest()`'s `spellUses: { ...maxSpellUses }` (a full replace, not a merge) silently wiped it back down to the character's original starting allotment on every Rest -- which also meant Anti-Paladin's "loses all Healing spells" got silently undone the next time the player rested, since their original Heal grant (if any) reappeared in the recomputed max. Fixed by adding a genuinely persisted `maxSpellUses: Record<string, number>` to both `AdventurerResources` and `DungeonState` (mirrored exactly like `spellUses` itself, including the same `createInitialDungeonState()`/`RESUME_DUNGEON`/`RETURN_TO_DUNGEON`/`loadSession()` threading), seeded at Character Creation and bumped by the identical amount at every site that already bumps `spellUses` -- `grantSpellUses()` (shared by every Advanced Class random-spell grant and every spell-granting Hireling), the new `grantFixedSpellUse()` helper (Cleric/Paladin/Bard's fixed grants), Anti-Paladin's Heal-zeroing (now zeroes the ceiling too, so it actually sticks), `learnRandomSpell()`, and `OPEN_TREASURE`'s `magicScrolls`/`randomSpell` reward sites. `rest()`/`canRest()` dropped their separate `maxSpellUses` parameter entirely in favor of reading `resources.maxSpellUses` directly (a second, independently-passed value could only ever drift from the persisted one); `CharacterSheet` now takes `maxSpellUses` as a prop instead of computing it internally, so `spellKeys` (which spells even get listed) comes from the always-up-to-date persisted field rather than a permanently-stale creation-time snapshot. `session.ts`'s `loadSession()` back-fills a save from before this field existed by taking, per spell key, the *higher* of the creation-time computation or whatever `spellUses` already holds -- an old save can't recover the exact ceiling it used to have, but this stops it from being regressed any further than the bug had already left it. A closely related bug caught in the same verification pass: `CharacterSheet` also always displayed `character.totalHp` (also fixed at creation) as the max-HP denominator regardless of any live `maxHp`, so any HP-bonus Advanced Class's effect never showed up in the "X / Y HP" stat or the low-HP warning threshold -- fixed by adding a `maxHp` prop (falling back to `character.totalHp` if not passed) alongside the `maxSpellUses` one, threaded from `resources.maxHp`/`state.maxHp` at all three screens that render it.

**Advanced Classes** (Expanded World, issue #23): 45 purchasable classes (`docs/game-rules-reference.md`'s "Table: Advanced Classes") a character stacks on top of Race/Class, each with a coin cost, a text requirement, an HP bonus, and often a passive ability -- fully authored in `src/data/advancedClasses.ts`'s `ADVANCED_CLASS_TABLE`, keyed by name (a stable id, not a dice roll, since these are bought, not rolled) rather than the usual roll-number `Record`. Most requirements reference systems this codebase doesn't have yet (Buildings/Politics for Noble/Lord/King/Emperor, per-terrain travel counters) -- rather than block on building all of those first, this pass follows the same "build the real mechanic, wire up what's honestly checkable, defer the rest with a documented reason" shape as New Races/New Spells: `src/engine/advancedClasses.ts`'s `REQUIREMENT_CHECKS` covers 27 classes answerable from state the game *already* tracks (`killsByName`/`killsByAbility`/`bossKills`/`monsterKills`, `character.spells`/`fixedGrants`, the Graveyard, another already-acquired class, or `resources.armor`) plus a small set of one-time achievement flags (see Milestones below) -- every other entry in `ADVANCED_CLASS_TABLE` still shows its real cost/requirement/HP/ability text in the UI (for flavor and completeness) but is permanently gated behind `isAdvancedClassTrackable()` returning false, rather than a live-but-always-false check that would misleadingly read as "you just haven't done it yet." `AdventurerResources.advancedClasses: string[]` (mirrored on `DungeonState`, threaded through every construction site exactly like `raceName`/`className` -- `RESUME_DUNGEON` resets it to `[]`, `RETURN_TO_DUNGEON` carries it over exactly) is the list of names acquired so far; `canAcquireAdvancedClass()`/`acquireAdvancedClass()` gate on not-already-owned + affordable + requirement-met, then spend the coins, push the name, bump `hp`/`maxHp` by the flat `hpBonus` (a permanent buff, same direct-mutation precedent as `hardWork()`'s permanent HP loss), and apply whichever ability is mechanically real via a small per-class switch. `loadSession()` back-fills `advancedClasses ?? []` for a session persisted before this field existed -- the same "optional for back-compat" precedent as `WorldState.bannedHexes`, and a real bug caught during this pass: without it, a returning player's existing save crashes the app outright the instant `TownScreen` tries to read the (missing) field.

  - **7 abilities #70 shipped requirement-checkable but left un-granted, fixed alongside #72**: auditing every one of the (then-)27 trackable classes' `abilityText` against `applyAdvancedClassAbility()`'s switch (issue #72's own investigation) found Scholar/Necromancer/Necromaster's spell grants and Merchant/Blacksmith/Thief's flag-reuse abilities were never wired -- acquiring any of the 6 did nothing beyond the flat HP bonus. - **Travel counters (issue #72)**: Lumberjack, Druid, Survivor, Pirate, Bard, and Cook are the last of #68's "cheap" bucket, all needing the same shape of addition -- a lifetime World-map travel counter. `AdventurerResources.travelStats: TravelStats` (`town.ts`) bundles `forestsCrossed`/`desertsCrossed`/`territoriesSailed`/`citiesVisited`/`provisionsSpentTotal` into one object for the same threading-surface reason `milestones` is bundled -- but unlike `milestones`, none of it is mirrored on `DungeonState` at all, since every signal only ever changes via World-map travel (`WorldScreen.tsx`'s `handleTravel()`) or Town, never inside a dungeon run; `App.tsx`'s `handleReturnToTown` carries it forward from `prev` rather than reading a `dungeon.X` field, the same way `provisions` already has to (`DungeonState` has no equivalent to read). `forestsCrossed`/`desertsCrossed` are Lumberjack/Druid (same signal, thresholds 2 and 6) and Survivor's raw, non-deduplicated per-move tallies; `citiesVisited` is Bard's own counter but stores de-duplicated hex keys instead (a `.length >= 3` check), since "passed through 3 cities" means 3 *distinct* ones, not 3 arrivals at the same one. `provisionsSpentTotal` is tallied inside `payTravelCost()` itself, off the same shortfall-aware `spend` value it already computes (not the raw terrain cost, so a move that ran short of provisions only counts what was actually spent) -- kept there rather than in the new `recordTravelStats()` (which handles the other four, since they need the destination tile's terrain/location that `payTravelCost()`'s signature has no reason to take). `territoriesSailed` (Pirate) only increments for a water hex arrived at while `WorldState.hasBoat` was true *before* the move (read prior to dispatching `MOVE`, since `hexReducer.ts` drops the boat the instant the destination isn't water) -- a water hex crossed via Patovsky/Sharkin's own water-walking, with no boat hired, doesn't count as "sailing." Avenger is approximated as "a previous character exists in the (world-scoped) Graveyard" -- the identical signal Gravedigger's own "lost another character" already checks, just under different flavor text, the same documented-simplification tier as Cleric's "faced an undead." **Hirelings** (Expanded World, issue #25): 16 paid companions (`docs/game-rules-reference.md`'s six "Table: Hireling" tables, one roster per culture) hired in a City/Fortress "to face just one dungeon" -- fully authored in `src/data/hirelings.ts`'s `HIRELING_ROSTERS`/`HUMAN_FORTRESS_HIRELINGS` (the one culture with a distinct Fortress-only tier; every other culture's single table already covers both), looked up via `hirelingsFor(culture, isFortress)` -- the exact same `culture`/`isFortress` signal `TownScreen.tsx`'s `cultureActionFor()`/Hard Work gating already reuse. Unlike Advanced Classes, research for this pass confirmed a Hireling *fighting alongside the player as a real combatant* (its own HP, its own turn, killable by monsters) has zero precedent anywhere in this engine -- `CombatState` only ever models the player (via top-level `DungeonState.hp`) plus an array of enemy `monsters`, and this is the identical "ally-in-combat concept" `CLAUDE.md`'s New Spells note already calls out as missing for the Summon spells (issue #61). Building that properly is out of scope here, same call already made for Summon Wolf/Skeleton/Elemental -- a hired Hireling is purely cosmetic once employed (a small read-only sidebar card in `DungeonScreen`, name/HP/equipment/ability text, no live HP tracking or death), and every mechanical benefit below is a passive check or a one-time grant, never the Hireling actually acting.
- **Expires per dungeon trip, unlike Advanced Classes' permanent stacking**: `AdventurerResources.hireling: string | null` and `DungeonState.hireling: string | null` (`src/engine/hirelings.ts`'s `canHireHireling()`/`hireHireling()`) track the currently-employed name, but the threading deliberately differs from `advancedClasses`' "carry forever" pattern, since "you pay your cost to face just one dungeon (you can go back if necessary)" means it must expire at the right moment: `RESUME_DUNGEON` never carries it (a new character doesn't inherit a dead one's hire, same as every other character-specific field); `RETURN_TO_DUNGEON` carries it over exactly (the same still-living character resuming this same paused trip keeps whoever they hired for it); a genuinely fresh dungeon entry (`DungeonScreen.tsx`'s `else` branch, no `activeDungeon`/`resumeDungeon`) consumes `resources.hireling` into the new run's own `DungeonState.hireling` via the positional `createInitialDungeonState()` call, then a mount-only `useEffect` (same one-time-per-mount shape as the existing Graveyard-recording effect, gated on `!activeDungeon && !resumeDungeon`) clears `resources.hireling` back to `null` through a new `onUpdateResources` prop `DungeonScreen` didn't need before this issue -- the hire is spent the instant it's actually walked into a new dungeon, so it can't be reused for a later, different one without paying again. `App.tsx`'s `handleReturnToTown(runId, dungeon)` sets `hireling: isDungeonBeaten(dungeon) ? null : dungeon.hireling` -- reusing the exact same `isDungeonBeaten()` check `activeRunId`'s own derivation already makes: a still-unbeaten retreat keeps the Hireling attached to this paused trip (so a later `RETURN_TO_DUNGEON` picks it back up correctly), while a beaten dungeon spends it for good, same as starting a genuinely new trip would need a fresh hire. Death already wipes `resources` entirely via `handleNewAdventurer()`, so no separate handling is needed there. `loadSession()` back-fills `hireling ?? null` for a session persisted before this field existed, same "optional for back-compat" precedent `advancedClasses` itself just established (and needed for the identical reason -- an old save would otherwise crash the app the instant `TownScreen`/`DungeonScreen` tried to read the missing field).

**Animals** (Expanded World, issue #26): 19 companions across "Table: Domesticated Animals" and "Table: Mounts" (`src/data/animals.ts`'s `DOMESTICATED_ANIMAL_TABLE`/`MOUNT_TABLE`, one shared `ANIMAL_BY_NAME` lookup mirroring Hirelings' `HIRELING_BY_NAME`) -- trained in the wild, or (Mounts only) bought outright in a qualifying city. - **Two acquisition paths, one shared qualification check**: "This land must be empty (no location) and have at least 2 lands of the same type adjacent to it" (training) vs. "a city that is on the appropriate terrain (and also has at least two of the same land adjacent to it)" (buying a Mount) -- both need "how many of this hex's six neighbors already share its own terrain," which had no existing precedent, so `hexState.ts` gained `countMatchingNeighbors()` plus `qualifiesForTraining()`/`qualifiesForBuyingMount()` built on top of it. Buying a Mount is **culture-agnostic** -- only the city's own terrain has to match, not which culture built it, confirmed by re-reading the rulebook text closely (unlike Hirelings, which are culture-gated). `WorldScreen.tsx` computes both qualifying lists for the current hex only (`trainableAnimals`/`buyableMounts`) and passes them to `HexInspector` (training -- the same "action lives where the qualifying hex is, not a separate card" precedent issue #59 established for Enter Dungeon) and `TownScreen` (buying a Mount, a new section alongside Hire a Hireling) respectively.
- **Training spends provisions regardless of outcome, only a successful roll adds the animal**: `src/engine/animals.ts`'s `trainAnimal()` -- "spend 4 provisions [8 for a Mount] and roll a die; if the number is equal to or greater than the animal's Dif, you managed to train it" -- mirrors Gamble/Thug Life's own "pay the cost, then roll" shape. Buying a Mount (`buyMount()`) has no roll at all, always succeeding if affordable.
- **10 of 19 have a real, working ability**, all reusing an existing mechanism: Owl/Giant Wolf (Forest), Camel (Desert), and Raptor (Swamp) cap their terrain at 1 provision, Goat/Llama cap Mountain at 2 -- the identical `hasElvenBoots()`-style override shape, generalized into `animalTravelCostOverride()` (cheapest applicable cap wins across every owned animal); Griffin's "1 provision for any land" is unconditional and checked first, short-circuiting past every other override including Elven Boots; Mammoth's "+1 extra provision per hex" is a penalty (`animalTravelCostPenalty()`), added on top of whatever override applies rather than competing with it; Dog blocks "Move Silently" entirely in the dungeon, the one non-travel-cost hook this pass needed -- `RESOLVE_ROOM_ENTRY`'s `"moveSilently"` branch no-ops if `draft.animals.includes("Dog")` (reducer-authoritative, mirrored by `RoomEntryPrompt.tsx` disabling the button, same "reducer decides, UI mirrors" split as door-opening). Horse's real ability ("for 1 provision you can move across 2 Plains") is approximated as a flat 1-provision Plains cost, same shape as every other mount, rather than tracking an every-other-hex-free counter this codebase has no precedent for -- a deliberate scoping call, not an oversight.
- **Deferred to a follow-up issue** (#67): the other 9 entries -- Cat/Wolf/Polar Bear are "None." in the rulebook itself; Tiger's "Cannot enter dungeons" is moot (nothing currently models an animal's dungeon presence to restrict, except Dog's rule, which doesn't apply to Tiger); Snake's "Attack deals Poison" needs the same ally-in-combat machinery this whole pass isn't building; Raven's death-survival roll needs the identical guard already deferred for Samambro (#60); Monkey's "carry an extra item" blocks on no item cap existing anywhere (same blocker as Hirelings' Cargo Ogre, #63); Eagle's "open an extra hex with each move" was deferred by explicit scoping call (no existing reveal-more-than-one-ring mechanism, and the rulebook doesn't say which extra hex); and Dolphin's "moves only in water" is nearly moot on its own terms -- research confirmed no `LocationKind` in this game's tables ever generates on Water terrain, so a Dolphin could never actually be bought in a city, only (in theory) trained after already renting a boat to reach a water hex at all.

**Buildings and Politics** (Expanded World, issue #27, `docs/game-rules-reference.md` lines 1668-1721): spending accumulated coins to build House/Tower/Palace/Castle/City/Fortress on an empty hex (`src/data/buildings.ts`'s `BUILDING_TABLE`), and a Political Affinity roll at any City/Fortress hex that makes it an ally, a Vassal, or a permanent enemy (`src/engine/politics.ts`). This unlocks the 5 Advanced Classes that sat permanently unbuildable since issue #23 shipped -- Noble, Knight, Lord, King, Emperor -- since `src/engine/advancedClasses.ts`'s own doc comment explicitly named "Buildings/Politics" as their blocker. The Buildings table's own **item-storage-with-theft-roll** mechanic ("you can store any number of items found in dungeons... roll a die, if it drops higher than Defense, a random item is stolen") is the one piece still deferred -- there's no real "storable item" concept beyond `heldItems`/`worth`, and no existing hook for "whenever you leave a dungeon" combined with "which of several owned buildings." **Warfare** itself (troops, attacking buildings, storming castles), which this note originally deferred alongside it, is now real too -- see below.
- **Buildings**: `buildingCost(kind, terrain, raceName)` doubles the base cost off-Plains, exempting Dwarf/Mountain and Elf/Forest (a plain string-match one-liner, same shape as `hexTables.ts`'s own `hasWaterWalk()`); `buildingRequirementMet()` gates Palace/Castle on owning Noble, City on Lord, Fortress on King. Built only on an empty hex (`qualifiesForBuilding()`, `tile.location === null` -- a hex's `HexTile.building?: BuildingKind` field is a player-owned overlay independent of `location`, so a built-on hex still reads as "empty" for this check, letting it be upgraded again later). Upgrading an already-built hex costs only the difference ("spending only the difference in cost") -- simplified to `max(0, newCost - oldCost)` with no protection against "downgrading" to a cheaper kind for a 0-cost swap, a documented simplification (same tier as `bladeTrap`'s roll-of-2) rather than new bookkeeping for an edge case with no real stakes. `AdventurerResources.buildings: OwnedBuilding[]` (`{ hexKey, kind }`, `src/engine/dungeonState.ts`) is mirrored on `DungeonState` (unlike `TravelStats`) since `dungeonReducer.ts`'s `finishIfVictorious()` needs to read it mid-dungeon for the tax payout below; threaded exactly like `advancedClasses`/`animals` -- `RESUME_DUNGEON` resets it to `[]` (a new character doesn't inherit a dead one's real estate), `RETURN_TO_DUNGEON` carries it over exactly. `src/engine/buildings.ts`'s `canBuildBuilding()`/`buildBuilding()` are the acquisition pair (mirroring `animals.ts`'s shape) -- `buildBuilding()` spends the (possibly upgrade-difference) cost, replaces or appends this hex's own entry in `resources.buildings`, and stamps `world.tiles[hexKey].building` via `hexState.ts`'s new `withBuilding()`. Surfaced in `HexInspector` as a "Build a Building" list (gated `isCurrentTile && tile.location === null`, same per-row list/button shape as "Train an Animal") since it's tied to standing on a qualifying *empty* hex, the same "action lives where the qualifying hex is" precedent issue #59 established for Enter Dungeon -- never in `TownScreen`, which only ever renders on an already-occupied City/Fortress hex. Owning one or more buildings also gets a small read-only "My Buildings" card (`src/ui/components/Buildings/`, same shape/precedent as `Animals`'s "My Animals" card) in `TownScreen`, plus a 🏛 map badge (`WorldScreen.tsx`, a new bottom-right corner alongside the existing ✓/⚔/💀 badges) on any hex with one.
- **Boss-kill tax**: owning Palace/Castle/City/Fortress credits its "Collect Taxes" coin value the instant a Dungeon Boss falls (House/Tower have no tax) -- `data/buildings.ts`'s `buildingTaxTotal(kinds: BuildingKind[])` (a bare `BuildingKind[]`, not full `OwnedBuilding[]` records, so it stays a pure data-file function with no dependency on `engine/dungeonState.ts`'s runtime type) sums every owned building's tax, credited inside `finishIfVictorious()`'s existing `if (combat.isBoss)` branch, right alongside the flat 2d6 Treasures roll it already grants.
- **Politics**: a new `POLITICAL_AFFINITY_TABLE`/`DEFAULT_POLITICAL_AFFINITY`/`politicalAffinityTarget()` in `src/data/affinity.ts` (`docs/game-rules-reference.md` lines 1706-1720) -- a genuinely different table from the existing boolean `RACE_AFFINITY`/`hasAffinity()` (issue #20): this one is a numeric Race x Culture target number for a 2d6 roll-and-compare, same 11-race-plus-"Others" row shape (Orc/Ogre split into two identical rows, same precedent `RACE_AFFINITY`'s own doc comment already explains) but only the rulebook's own 11 literal rows -- unlike `RACE_AFFINITY`'s 14, Cat-Person/Rinoceroid/Lightbugster aren't explicitly listed in the rulebook's own Political Affinity table, so they deliberately fall back to `DEFAULT_POLITICAL_AFFINITY` rather than an invented number, an honest reflection of the rulebook's own omission. `src/engine/politics.ts`'s `resolvePoliticalAffinity()` rolls 2d6 against the target at any City/Fortress hex with no status yet (`canAttemptPoliticalAffinity()`, gated the same as Hire Boat/Ask -- "reducer decides, UI mirrors"); success becomes an ally (or a Vassal, see below), failure becomes a permanent enemy -- `WorldState.politicalStatus?: Record<string, "ally"|"vassal"|"enemy">` (optional, `?? {}`-at-the-read-site back-compat precedent identical to `bannedHexes`) records the outcome via a new `withPoliticalStatus()`/`politicalStatusFor()` pair, and it's permanent: one resolution per hex, ever, no re-rolling a hex that already has a status (matching "a dungeon's beaten status never resets"). Becoming an enemy here is what feeds Warfare's own "Declared Enemies" retaliation (issue #28, see below) -- resolved in `WorldScreen.tsx`'s `handlePoliticalAffinity()` (not `TownScreen`, which only ever sees an `AdventurerResources`) since a single roll touches both `resources` (the Vassal-eligibility check, `talkedToKing`/`vassalCount` milestones) and `WorldState` (the resolved status) -- the same "resolved in WorldScreen, applied to whichever of the two actually changed" split Thug Life's own `handleThugLife()` already established -- surfaced as a new "Political Affinity" City Action button in `TownScreen` (same `.actionBtn` shape as Ask/Hire Boat), plus a read-only "Politics" row in `HexInspector` for *any* inspected City/Fortress hex (not gated on `isCurrentTile`, since the action itself is TownScreen-only but the *status* is worth seeing while merely looking at a hex on the map) and a 🤝/👑/🗡 map badge per outcome.
- **Vassals**: a City hex (never a Fortress -- the rulebook's own vassal language is specifically about Cities under a King, so a Fortress roll can only ever become an ally at best, a documented simplification) becomes a Vassal instead of a plain ally if the roll succeeds AND the player holds Lord or King AND owns a Castle/City/Fortress within 3 hexes of the target (`hexState.ts`'s new `hexDistance()`, the standard axial distance formula, plus a new `parseHexKey()` inverse of `hexKey()` to turn an `OwnedBuilding.hexKey` string back into a coordinate). The rulebook's own text only spells out the Lord-plus-City case explicitly; King's equivalent range for Emperor's "3 vassals" requirement is filled in by direct analogy rather than a separately-specified rule -- documented as a scoping call, same tier as Cleric's "faced an undead" approximation. A successful Vassal roll increments a new `AdvancedClassMilestones.vassalCount` counter.
- **Noble/Lord/King/Emperor/Knight now trackable**: `REQUIREMENT_CHECKS` (`src/engine/advancedClasses.ts`) gains all 5 -- Noble reads a new `AdvancedClassMilestones.talkedToKing` flag (Noble's own requirement, "Talk to the King of a Fortress," approximated as *attempting* a Political Affinity roll at a Fortress at least once, regardless of outcome -- "talking" is the act of visiting and rolling, not a required success -- set by `resolvePoliticalAffinity()` whenever `isFortressHex`); Lord/King read `resources.buildings.some(b => b.kind === "Castle"/"City")`; Emperor reads both an owned Fortress and `vassalCount >= 3`; Knight (`"Be Noble"`) chains on Noble the same way Paladin already chains on Cleric -- which meant widening Paladin's own OR (`"Be a Knight or a Cleric"`) to actually check both now that Knight is real too, not just Cleric alone as it had to be before this issue. Knight's "Gain a Horse" ability reuses the Animals system (issue #26) directly in `applyAdvancedClassAbility()` -- pushes `"Horse"` onto `resources.animals` if there's a free slot and no Mount already owned, silently no-opping otherwise (this file has no logging mechanism at all, unlike `dungeonReducer.ts`, so this matches its existing style rather than a regression).

**Warfare** (Expanded World, issue #28, `docs/game-rules-reference.md` lines 1722-1763): recruiting troops at owned Castle/City/Fortress buildings or Vassals, marching them (plus optionally the character) to attack a City/Fortress, and on success choosing to Annex it (a Vassal) or Loot it (razed to Ruins, flat coins). Also covers "Declared Enemies": every attack also rolls a die for each of the player's other enemies, which can destroy the player's own nearest building. Both halves reuse infrastructure #27 already built -- the target's Defense (`BUILDING_TABLE.City.defense`/`.Fortress.defense`, already 6/12, an exact match to the rulebook's own Warfare numbers) and the Political Affinity roll/table (`politics.ts`, `data/affinity.ts`) for both Annex and the initial "who's an enemy" signal -- so this pass builds the whole rulebook section in one go, including Declared Enemies, since its retaliation check fires synchronously off the same "you attacked a building" trigger the rulebook itself describes ("Whenever you attack a building, roll a die for each enemy..."), not a separate periodic/world-tick system needing new scheduling infrastructure.
- **Mustering Troops**: `src/engine/warfare.ts`'s `canRecruitTroop()`/`recruitTroop()` -- 200 coins (`TROOP_COST`, `src/data/warfare.ts`) at a hex that's either the player's own Castle/City/Fortress-tier `OwnedBuilding` (House/Tower can't muster troops) or one of their Vassals (`politicalStatusFor(...) === "vassal"`), capped at one *currently unspent* troop per source hex. `AdventurerResources.troops: number`/`troopSources: string[]` are **not** mirrored on `DungeonState` -- same "World/Town-only, nothing inside a dungeon run needs it" shape as `TravelStats`, not `buildings`'s own mid-dungeon-readable one.
- **Attacking**: "taking with you all the troops you recruit" -- `resolveAttack()` always spends *everything* in `resources.troops` at once (no partial-commitment UI), plus an optional `joinBattle` toggle for one bonus die (a natural 1 kills the character outright, but *only if the battle is lost* -- rolling a 1 while the troops' own dice already clinch the win is harmless). Rolls `troops` d6 (+1 more if joining) and compares the sum to the target's Defense (City 6, Fortress 12, straight from `BUILDING_TABLE`); troops are always fully spent by the attempt, win or lose, freeing every source to recruit again afterward.
- **Declared Enemies**: resolved inside the same `resolveAttack()` call, regardless of its own win/lose outcome -- for every hex whose Political Affinity is `"enemy"` (excluding the one just attacked), rolls 1d6; 1-3 sends that many troops against the player's own nearest owned building (`hexDistance()` from the enemy hex, tie-broken by lowest Defense), rolled against that building's own Defense; a win destroys it, via a new `withoutBuilding()` hex helper that clears just the `building` field -- the hex reverts to a plain, re-buildable empty hex, not Ruins, since Ruins is specifically what a *stormed City/Fortress location* becomes and a player-built structure was never a `location` to begin with. Skipped entirely if the player owns no buildings.
- **Storming the Castle**: on a successful attack, `resolveStorming()` offers Annex (re-rolls Political Affinity at +2 via the existing `politicalAffinityTarget()`; success grants Vassal status unconditionally -- "gain all the advantages of having a vassal" -- deliberately bypassing the peaceful path's Lord/King-within-3-hexes eligibility check, since this was won militarily; failure auto-falls-through to Loot, per the rulebook's own "you'll only have the option of looting the place" forced-fallback wording, no dead end) or Loot directly (a new `withRazedToRuins()` hex helper sets `location: "ruins"` and clears the generated name; flat coins, 600 for a City or 1000 for a Fortress via `stormingLootPayout()` -- the rulebook's "200 for a castle" branch is unreachable here, since no NPC-generated hex is ever a `Castle` `location`).
- **Attacking your own Vassal is blocked** (`canAttack()`) -- nothing to gain, the rulebook's target is "another Lord/King's" building -- but any other City/Fortress hex is fair game regardless of current ally/enemy/unresolved status, matching "declare war on others" broadly.
- **UI**: "Recruit Troop" appears as its own row in `HexInspector` (an owned Castle/City/Fortress, gated `isCurrentTile`) and as a City Action button in `TownScreen` (a Vassal hex only -- the TownScreen-reachable half of the same gate). "Attack"/"Attack (Join Battle)" are two City Action buttons (`TownScreen`, shown whenever `canAttack`), and a won attack swaps the whole City Actions section for the Annex/Loot choice, mirroring the Arena panel's own "swap the whole section" precedent. A `"lost-death"` result calls `onCharacterDied("warfare")`, the same death path Gamble/Thug Life/Arena already use (`GraveyardEntry.causeOfDeath` gained `"warfare"`). One real bug caught while browser-testing this: `attackMessage`/`pendingStorm` can't be `TownScreen`-local state the way `thugLifeMessage`/`politicalAffinityMessage` are, since a winning Loot razes the target to Ruins, flipping `inCityOrFortress` false and unmounting `TownScreen` on the very next render -- any message held there would vanish before ever being shown. Both are lifted up to and owned by `WorldScreen` instead (reset on travel, same as `trainResultMessage`), passed into `TownScreen` as props, and also surfaced through a new `HexInspector` `warfareMessage` prop for the post-Loot map view the player lands on right after.

**Teleport** ("You teleport to any empty room") is the one spell whose `CAST_SPELL` action carries extra fields (`destLevel`/`destSegId`) rather than just `spellRoll` — clicking the Combat panel's "Flee — Teleport" button (`CombatPanel`'s `onFlee` prop, kept distinct from `onCastSpell` since this spell alone needs a destination first) doesn't dispatch immediately; it swaps the panel for `TeleportPicker` in the same overlay slot, listing every already-discovered "empty" room across all levels (`isTeleportDestination()` in `dungeon.ts` -- a room-type segment, excluding the one currently in combat, with no undefeated/`sneakedPast` monsters and not flagged `needsMonsterReroll` either; corridors and staircases don't count as a "room"). That last exclusion matters specifically for Teleport: a `needsMonsterReroll` room looks empty to `isTeleportDestination()`'s other checks, but `CAST_SPELL`'s Teleport case calls `rerollMonstersIfNeeded()` the instant the player arrives (same as walking there normally would) -- without it, `TeleportPicker` could list a room that immediately springs a fresh fight on arrival, defeating the entire point of fleeing. Only picking a listed room dispatches `CAST_SPELL` with that destination (Cancel dispatches nothing, spending no use); the reducer re-validates the destination itself before applying it, same as every other action. `draft.activeLevel`/`currentSegId`/`selectedSegId` all move to the chosen room -- unlike a death or a Town retreat mid-fight, the fled room's own `CombatState` doesn't persist anywhere, so `rerollMonstersIfNeeded()`'s "the fight you fled from is still waiting" branch (see Monster table re-roll below) is what resumes it, full HP, whenever the player later selects that segment again themselves. `TeleportPicker` also shows a non-interactive "Fleeing from Level X — [Type] (Segment N)" line above the destination list (issue #56) -- the starting room is excluded from its own list, so without it there was no "you are here" reference point to judge the destinations against. The starting room's type/level are looked up from `levels` via `excludeSegId` (the same lookup the destination list itself already builds), rather than threading them down as extra props the caller would have to keep in sync.

**The Boss fight**: each dungeon type's `DUNGEON_TABLES[key].boss` (a `Record<1-6, MonsterTemplate>`, authored from the rulebook's per-dungeon "Table: Boss") is rolled once via `resolveBoss()` in `dungeon.ts` the moment the Final Room segment is built — both in the `descend-final` case (auto-triggered at depth 3) and `dead-end-final` (no stairs ever found). Per the rulebook, there's no Content/Monsters roll alongside it, only the Boss. The rolled `MonsterTemplate` is attached to the Final Room's existing `SegmentState.monsters` field and fed through the same `startCombatIfMonsters()` plumbing as an ordinary room, just with a new `isBoss` flag threaded onto `CombatState` — `finishIfVictorious()` checks that flag to grant a flat 2d6 Treasures instead of the normal per-monster Loot table (Treasures are flavor-only, like ordinary Loot, since there's no inventory system). `DungeonScreen.tsx` shows a gold victory panel once the Final Room's boss is marked defeated. `dungeonHistory` is persisted the instant `bossDefeated` (`isDungeonBeaten(state)`) flips true, via its own `useEffect` right alongside the Graveyard-recording one -- not only on unmount (`onLeaveDungeon`'s other call site) or on the victory panel's "Return to Town" click. Reloading (or just closing the tab) from the victory screen itself, before ever clicking through, skips both of those: a hard reload kills the JS outright, so the unmount cleanup never runs, and World's own beaten-check (`WorldScreen.tsx`'s `dungeonInfoFor()`) would otherwise still be reading the *pre-victory* `dungeonHistory` snapshot on the next load, wrongly showing "Enter Dungeon" again and resuming the stale, still-ongoing fight.

**Chests and Loot**: `RoomContentEntry.hasChest` flags the Room Content rows that describe an actual Chest (vs. passive flavor text like "1d6 coins on the floor," which isn't mechanically applied — see stub list below); a chest can also turn up via `secretPassageResult === "You have found a hidden Chest!"`. `RoomInspector` offers a free "Open Chest" action (no torch cost, per the rulebook) once one is available and unopened, rolling two independent dice — the higher becomes coins, the lower becomes Treasures, unless both come up 1, in which case the chest was empty and rolls on the dungeon's Trap table instead. `DungeonState.treasures`/`keys` are real counters (credited by both Chests and the Loot monster ability's `rollLoot()`) shown on `CharacterSheet` alongside coins — Keys don't do anything yet (no key-unlocks-any-door shortcut), but Treasures are openable (next).

**Secret Passage → Staircase**: a roll of 6 on the Secret Passage table ("A secret door to a Staircase") isn't just flavor text — `buildSecretPassageStaircase()` in `dungeonReducer.ts` builds a real, descendable `type: "staircase"` segment off the room (1 door, "the door in the end," exactly like an ordinary door resolving to the Segments table's own "staircase" outcome), except the door itself is brand new (appended to `seg.doors`, not one of the room's already-rolled doors) and already `opened: true`, since finding it via the search *is* the reveal. Placed in whichever cardinal direction the room doesn't already have a door facing — a non-entrance room always has at least one free direction (`assignDirections` caps a non-entrance room's own door count at 3 of 4), so this only silently no-ops (the flavor text alone still stands) for a fully 4-doored entrance, the one segment type that can actually use all four.

**The Combat panel's dice roll** renders directly under the player's HP row, before the monster list — not at the very bottom of the panel, which used to get pushed out of view once a fight had several monsters.

**Singular monster names** (issue #65): `MonsterTemplate.name` is written however the rulebook prints a row (plural for anything with a dice-based `count`, e.g. "1d6 Goblins"), but a dice roll can resolve to exactly 1 -- `MonsterTemplate.singularName` (optional, only set on templates whose `count` could actually produce 1: Goblins, Bats, Skeleton Soldiers, Giant Rats, Scorpions, Cultists, Serpents, Imps) gives `spawnMonsters()` (`combat.ts`) the correct name to use instead when that happens, so a lone Goblin isn't displayed/logged everywhere as "Goblins." Goblins/Bats/Skeleton Soldiers were also consolidated into shared `GOBLINS`/`BATS`/`SKELETON_SOLDIERS_SWARM` constants in `dungeonTables.ts` (they repeated identically across multiple dungeon types' Monster tables and one trap-ambush each), same "repeated verbatim rows become a shared constant" precedent as `BLADE_TRAP`/`HEALTH_POTION`. The one consumer that had to change alongside this: `killsByName` (keyed by the monster's own lowercased name at kill time) now genuinely splits "goblin" from "goblins" depending on how many died together in a given room, so Advanced Classes' Goblinator (`>= 20` Goblins killed) sums both forms rather than only the plural -- Ruthless's Imps requirement didn't need the same treatment, since Imps are always 2d6 (minimum roll 2, so a solo "Imp" can never actually happen).

**Opening a Treasure**: each dungeon type's `DUNGEON_TABLES[key].treasure` (a `Record<1-6, RewardOutcome>`) models the rulebook's per-dungeon "Table: Reward" -- rows 1-4 give a Treasure-column item directly; rows that instead say "[Roll in the Wonders/Magic Item/Weapon column]" (Palace's rows 5-6; Prison additionally redirects row 4 to the Weapon table) are `RewardEffect.kind: "rerollColumn"`, resolved by an immediate second `rollDie(rng)` into that column's own table -- see Armor & Weapons for what Wonders/Magic Item rolls actually produce. Rows repeated near-verbatim across dungeon types (Health Potion, Magic Scroll, Valuable jewel) are shared constants (`HEALTH_POTION`, `MAGIC_SCROLL`, etc. in `dungeonTables.ts`), mirroring the existing `BLADE_TRAP`/`CLICK_NOTHING` pattern for traps. The `OPEN_TREASURE` action is usable anytime (not tied to a room, unlike Chests) via a die-roll bar rendered in `DungeonScreen` whenever `treasures > 0`; mid-combat it consumes the round exactly like `CAST_SPELL` (paralysis included, and now also gated on `combat.pendingDamage` being resolved first), since an unattended Health-Potion-on-demand would otherwise trivialize fights. `RewardEffect.kind: "restoreAllSpells"` (Tomb's Mana Potion) restores every spell to `DungeonState.maxSpellUses` directly (issue #75 -- previously a client-computed value passed through the action, before that field was persisted). The rulebook's "worth N Coins in the town" outcomes (Ornament, Religious Object, Sinister Idol, Valuable jewel) don't convert to coins on the spot — there's no town to sell them in yet — so `RewardEffect`'s `"heldValue"`/`"heldValueRoll"` kinds instead push a `HeldItem` (`{ name, worth }`) onto `DungeonState.heldItems`, shown in the `Pack` sidebar card (`src/ui/components/Pack/`, positioned between `CharacterSheet` and the Ledger card in the dungeon, and alongside `CharacterSheet` in Town where it can actually be sold — see Town below).

**Armor & Weapons**: `docs/game-rules-reference.md`'s "Table: Armor" (1d6: Ring 0 HP / Bracelets 2 / Boots 3 / Shoulderpads 3 / Helm 4 / Breastplate 10) is identical across all six dungeon types, so it's one shared `ARMOR_TABLE` constant in `dungeonTables.ts`; "Table: Weapon" differs per type, so each `DUNGEON_TABLES[key]` carries its own `weapon: Record<1-6, WeaponEntry>`. A dungeon type's `wonders`/`magicItem` tables (`Record<1-6, WonderEntry | MagicItemEntry>`) model the Reward table's other two columns; all six dungeon types are fully authored. Every Wonder/Magic Item's unique ability is encoded as an `ItemEffect` (`extraHp`/`weaponDamageBonus`/`damageBonusVsTag`/`damageMultiplierVsTag`/`ignoresMonsterAbility`/`trapImmunity`/`doubleChestCoins`/`combatDamageBonus`/`grantsTorches`/`randomSpell`/`lifesteal`/`instantKillOnRoll`/`flavor`) -- a small, reusable vocabulary rather than one bespoke code path per item; `tags` matching (e.g. "+2 damage to Angels", or Crypt's Garlic necklace naming both Vampire *and* Ghoul) is a case-insensitive substring check against `MonsterTemplate.name`, since the rulebook itself has no formal monster-category system. `MagicItemEntry.fixedFormula` overrides the usual "roll the base Weapon table, then layer a bonus on top" shape for uniquely-named weapons that specify their own damage directly (Crypt's Boatman's Oar, 1d6+1). `ignoresMonsterAbility` covers two different shapes the rulebook conflates under one phrase: abilities a *monster* uses against the player (Undead revival, Paralyze, Deathtouch, Firebreath -- checked via `dungeonReducer.ts`'s `ignoresAbility()`) and the player's own attack being defensively blocked (Stoneskin/Intangible's damage-parity filter in `combat.ts`'s `applyDefensiveAbilities()`, e.g. Boatman's Oar bypassing Intangible) -- the latter is threaded into `resolvePlayerAttack()` as an `ignoreAbilities` array (`dungeonReducer.ts`'s `ignoredAbilities()` collects it from `equippedEffects()`), since `combat.ts` has no notion of the player's inventory on its own. `trapImmunity` is enforced by `dungeonReducer.ts`'s `resolveTrapOutcome()`, the single entry point all three trap-firing sites (`RESOLVE_DOOR_LOCK`, `ROLL_SECRET_PASSAGE`, `ROLL_CHEST`) funnel through: it blocks the *whole* trap -- torch cost included -- and consumes the item (one-shot, matching "ignores the next activated trap," not a standing immunity). `WeaponEntry.twoHanded` remains tracked but unenforced -- audited and confirmed there's currently no off-hand/shield slot (`ArmorPieceKind` is all body slots) it could conflict with, so there's nothing to block without first building the hand-economy/limb-tracking system the rulebook's "Your Hands" rule would need.
- **Acquiring items** (`resolveWonder()`/`resolveMagicItem()` in `dungeonReducer.ts`, called from `OPEN_TREASURE`'s `rerollColumn` case): a Wonder either grants its own HP-bearing item (Jester Hat, 2 HP) or, if it has no HP of its own (Amulet of the Dead), still becomes a `DungeonState.armor` entry at `hp: 0, maxHp: 0` -- never offered as a damage-absorption choice (see below), but still equipped/visible and checked by whichever system its `effect` concerns -- except `combatDamageBonus` (Potion of Fury), which instead adds directly to `combat.playerDamageBonus` if a fight is active -- `OPEN_TREASURE` is documented as usable anytime, not tied to a room or combat, so opening one outside a fight is normal, expected usage, not a misuse; the bonus is simply wasted in that case (this app has nowhere to bank it for whatever fight comes next), but `resolveWonder()` still logs that explicitly ("...No fight is happening right now, so it has no effect.") rather than silently discarding it under the same generic "Treasure: ..." text every other Wonder logs on success, which would otherwise read as if it had worked. A Magic Item is always "[Armor] of X" or "[Weapon] of X" (`MagicItemEntry.grants`): resolving one rolls the *base* Armor or Weapon table for the concrete piece/formula, then layers the effect on top -- `extraHp` is baked directly into that new piece's `hp`/`maxHp` (Centurion's [Armor] +1 HP rolls, say, Boots at 3 HP and pushes a 4/4 HP piece), anything else becomes the piece's `effect` (or the weapon's `bonusEffect`); acquiring a new weapon never overwrites `DungeonState.weapon` directly (see "A spare weapons inventory" below) -- every weapon-granting site instead pushes onto `spareWeapons`, matching the precedent `COLLECT_REMAINS` sets for picking up a dead adventurer's gear (also a push, not an overwrite).
- **Wearing them**: `DungeonState.armor: ArmorPiece[]` / `weapon: EquippedWeapon | null` (mirrored on `AdventurerResources` in `town.ts` and `FallenAdventurer` for remains) flow through every place a dungeon gets (re)constructed exactly like `heldItems` already did -- `createInitialDungeonState()`, `RETURN_TO_DUNGEON` (carries them over exactly), and `App.tsx`'s retreat/return round-trip; `RESUME_DUNGEON` deliberately does **not** carry them (a new character doesn't inherit a dead one's gear -- it becomes `remains`, same as coins/heldItems). `Equipment` (`src/ui/components/Equipment/`) renders both, read-only in `DungeonScreen`'s sidebar and with a per-piece "Fix (1 coin)" button in `TownScreen` (wired to `town.ts`'s `canFixArmor()`/`fixArmor()`, the same pure-function shape as `rest()`/`buyTorch()`).
- **A spare weapons inventory** (issue #48): unlike Armor's 5 independent, additive slots, a weapon used to be a single override -- finding a new one silently discarded whatever was equipped, with no salvage and no choice. `DungeonState.spareWeapons: EquippedWeapon[]` (mirrored on `AdventurerResources` and, as plural `weapons`, on `FallenAdventurer`, threaded through every construction site exactly like `armor` already is -- `RESUME_DUNGEON` resets it to `[]`, `RETURN_TO_DUNGEON` carries it over exactly) fixes this: every weapon grant -- both `resolveMagicItem()` branches (called from `applyRoomContentReward()`, an *automatic* grant the instant a room is built with no player action at all, as well as from `OPEN_TREASURE`'s `rerollColumn` -> `magicItem` branch), `OPEN_TREASURE`'s direct `rerollColumn` -> `weapon` branch, and `COLLECT_REMAINS` -- pushes onto `spareWeapons` instead of overwriting `weapon`. The automatic grant site is exactly why the fix isn't an interrupting choice prompt (mirroring `RoomEntryPrompt`'s pattern was considered and rejected): there's no player action to interrupt when a weapon can appear just from opening a door. A weapon found this way is never auto-equipped, full stop, even the very first one -- a deliberate behavior change matching the issue's "give the player an actual choice" literally. `WIELD_WEAPON { index }` (`dungeonReducer.ts`, gated exactly like `COLLECT_REMAINS`: alive, not in combat, no pending room entry -- free, out-of-combat only, no mid-fight swap semantics invented) swaps a chosen spare into `weapon`, pushing whatever was equipped (if anything) back into `spareWeapons`; `town.ts`'s `wieldWeapon()` is the same swap as a pure resources-transform for Town, which has no `DungeonState`/reducer to dispatch against (same split `fixArmor()`/`sellItem()` already establish). `Equipment` renders a "Spare Weapons" list (name/formula/Two-handed, each with a "Wield" button via a new optional `onWield` prop mirroring `onFixArmor`) between the equipped weapon and the armor list, in **both** `DungeonScreen`'s sidebar and `TownScreen` (unlike `onFixArmor`, Town-only) since wielding is meant to be usable mid-dungeon between fights, not just back in town; the row is a two-line layout (name+button on top, formula below) rather than `armor`'s single-line grid, since a long weapon formula plus "· Two-handed" reliably overflowed a single grid row past the sidebar's fixed width. Selling a spare weapon is out of scope -- no `worth`/price data exists for weapons anywhere in this codebase, unlike `HeldItem`.
- **Armor slot uniqueness, a spare armor bench, and a Pack cap** (issue #82): rulebook fidelity gap found alongside #83 -- "Armor is separated into 5 pieces: Shoulderpads, Bracelets, Boots, Helmet, and Breastplate... can't use more than one identical piece" (`docs/game-rules-reference.md` line 201) was never enforced; a player could end up wearing two Boots. `dungeonReducer.ts`'s `addArmorPiece(draft, piece)`/`addArmorPieces(draft, pieces)` are the single chokepoint replacing all 4 raw `draft.armor.push(...)` call sites (`resolveWonder()` x2, `resolveMagicItem()`, `COLLECT_REMAINS`): a piece destined for one of `REAL_ARMOR_SLOTS` (the rulebook's actual 5 -- Ring is a documented 0-HP dud roll, not one of them, and `wonderItem` is an unlimited trinket collection, so neither is slot-unique) benches into `DungeonState.spareArmor: ArmorPiece[]` instead of stacking, if that slot is already occupied in `armor`. `spareArmor` mirrors `spareWeapons` (#48) exactly -- `AdventurerResources.spareArmor`/`FallenAdventurer.spareArmor` threaded the same way, `RESUME_DUNGEON` resets to `[]`, `RETURN_TO_DUNGEON` carries it over exactly -- and gets its own `WIELD_ARMOR { index }` action (`dungeonReducer.ts`)/`wieldArmor()` (`town.ts`), the one place armor's swap logic genuinely differs from `WIELD_WEAPON`'s: since armor has 5 independent slots rather than one, equipping a spare has to find-and-displace *that spare's own slot* specifically (`draft.armor.findIndex((p) => p.piece === chosen.piece)`), not just whatever's currently equipped. `Equipment` renders a "Spare Armor" list identical in shape to "Spare Weapons" (name/HP, a "Wield" button via a new optional `onWieldArmor` prop), in both `DungeonScreen`'s sidebar and `TownScreen`.
  - **Pack cap**: `town.ts`'s `MAX_HELD_ITEMS = 10` (`docs/game-rules-reference.md` line 200, "Load Limit: ...up to 10 items in your backpack") was simply never added, alongside `MAX_TORCHES`. Exactly two sites ever push to `heldItems` -- `OPEN_TREASURE`'s `heldValue`/`heldValueRoll` cases and `COLLECT_REMAINS` -- both now check room against the cap first.
  - **The interactive swap** (confirmed with the user over a simpler "just block the pickup" alternative): `DungeonState.pendingPackItem: HeldItem | null` holds an item that didn't fit. A new `isActionBlocked(state)` helper (`hasPendingRoomEntry(state) || state.pendingPackItem != null`) replaced every one of `hasPendingRoomEntry()`'s call sites, so a pending swap blocks doors, spellcasting, combat, wielding, etc. exactly like a pending room-entry choice already does -- `CombatPanel` gained its own `hasPendingPackItem` prop folded into `canAct` for the same reason `pendingDamage` already is (`OPEN_TREASURE` is dispatchable mid-fight and can itself trigger this). `OPEN_TREASURE` at a full Pack sets `pendingPackItem` instead of pushing (the Treasure is still spent -- it was "found," it just doesn't fit yet). `COLLECT_REMAINS` picks up coins/treasures/keys/armor/spareArmor/weapon/spareWeapons in full regardless (none of those are capped) and takes as many `heldItems` as currently fit; if any are left over, the *first* becomes `pendingPackItem` while the *rest* stay behind in a shrunken `seg.remains` (only `heldItems` populated, everything else already cleared/zeroed) -- nothing is ever silently lost, and clicking Collect Remains again after resolving the pending choice picks up the next overflow, chaining naturally through the same single-item field rather than needing a queue. A new `RESOLVE_PACK_SWAP { discardIndex: number | "decline" }` action resolves it (a number discards that existing row and keeps the incoming item; `"decline"` drops the incoming item for good) and is deliberately exempt from `isActionBlocked` itself, since it's the one action that has to run *while* blocked -- it's what clears the block. A new free `DISCARD_ITEM { index }` action (gated by `isActionBlocked` like everything else, so it's unavailable mid-swap -- `RESOLVE_PACK_SWAP` is the correct action then instead) and `town.ts`'s `discardItem()` let a player drop an item anytime, dungeon or Town, with no Sell/Town required. `Pack` renders a "N/10" capacity line, a free "Discard" button per row, and -- whenever `pendingPackItem` is set -- an inline swap prompt (a "Leave It Behind" button plus each row's normal Sell/Discard buttons replaced by "Discard & Keep") in the same sidebar card, rather than a full main-panel interrupt like `RoomEntryPrompt`'s, since Pack is already a persistent sidebar element in both `DungeonScreen` and `TownScreen`. `pendingPackItem` is deliberately **not** threaded through `RESUME_DUNGEON`/`RETURN_TO_DUNGEON` at all (always `null` in `createInitialDungeonState()`) -- safe because "Retreat to Town" is gated on `!state.pendingPackItem`, so a dungeon state can never be persisted mid-swap in the first place.
- **Taking damage with armor equipped**: "reduce this value from your HP (or armor's HP, if you're using one — your call)" is a genuine player choice, not automatic, so `applyMonsterTurn()` can't just subtract HP synchronously the way it used to. It splits a round's damage into `poisonDamage` (Poison "cannot be absorbed by armor or other means" — always hits `hp` immediately, and first, so a fatal poison tick can't leave a moot choice pending for the rest) and `absorbableDamage` (everyone else's); if the player has at least one armor piece with `hp > 0` (`hasUsableArmor()`), `absorbableDamage` is stored on `CombatState.pendingDamage` instead of applied, and the round stops there. `CombatPanel` renders an absorption chooser whenever `pendingDamage !== null` (same visual slot the paralyzed-turn message uses), listing HP and every piece with `hp > 0`; picking one dispatches `RESOLVE_DAMAGE`, which subtracts from the chosen pool (overflow past a piece's remaining HP spills onto the player's `hp`, destroying the piece at 0) and re-runs the death check. `PLAYER_ATTACK`/`CAST_SPELL`/`OPEN_TREASURE` all no-op while a choice is pending, the same way they already do during paralysis.
- **Fighting with a found weapon**: `attackBonus()`/`attackMultiplier()` in `dungeonReducer.ts` read `DungeonState.weapon?.bonusEffect` (plus `combat.playerDamageBonus`) when `PLAYER_ATTACK` computes `weaponTotal` -- the multiplier applies to just the weapon's own roll, flat bonuses are added after, since only one bonus effect is ever active on a given equipped weapon at a time (this app doesn't model combining two).

**Still stubbed / not yet built**: `WeaponEntry.twoHanded`, tracked but unenforced (see Armor & Weapons) -- there's no off-hand/shield slot in this armor system yet for a two-handed weapon to actually conflict with.

**Move Silently**: a quiet arrival (an unlocked door, a picked lock, a fresh staircase descent) into a room with monsters no longer auto-starts combat -- `finishRoomSegment()` only does that for a noisy one now. Instead it waits for a `RESOLVE_ROOM_ENTRY` action: **Attack First** (free, same outcome this app always gave before this existed) or **Move Silently** (spend 1 torch; roll one die per monster -- resolved inside the reducer via its own `rng`, since a room's count can itself be a dice roll like "1d6 Goblins" and isn't knowable client-side in advance -- any 1 means detected and the monsters attack first, otherwise the room is marked `sneakedPast` and combat never starts at all). `hasPendingRoomEntry()` blocks every other dungeon action the same way an active `CombatState` does, until that choice is made. A `sneakedPast` room's monsters wake up (combat starts, monsters first) if a later noisy action -- breaking a door, a fired trap -- happens in that same segment (`wakeSneakedPastMonsters()`); `rerollMonstersIfNeeded()` and `restoreMapFromPersisted()` both treat a sneaked-past room as still-occupied (no fresh reroll, no auto-resume from just reselecting it). Halfling's "roll two dice and discard the lowest (except in the Boss)" advantage is applied server-side in the same `RESOLVE_ROOM_ENTRY` case; Boss rooms never reach this at all, since `descend-final`/`dead-end-final` call `startCombatIfMonsters()` directly. `RoomEntryPrompt` (`src/ui/components/RoomEntryPrompt/`) is the UI for the choice, rendered in place of `RoomInspector` while pending. Both `DungeonScreen.tsx`'s and `DungeonMap.tsx`'s own client-side `pendingRoomEntry` (mirroring the reducer's `hasPendingRoomEntry()` for render/interaction gating) also check `state.alive` directly -- dying to that same room's monsters never marks `monstersDefeated`, so without this the prompt would otherwise reappear right alongside the death panel (the reducer's own `RESOLVE_ROOM_ENTRY` handler already no-ops once dead, so this was purely a decorative render bug, the same class of gap #38 fixed for the Combat panel).

**Race and class abilities**: `DungeonState.raceName`/`className` (plain strings, matched by exact `RaceDef.name`/`ClassDef.name` -- the same "no formal taxonomy" precedent Armor & Weapons' `matchesTags` already established for monster names) thread through every place a dungeon gets (re)constructed, exactly like `characterName`, so `dungeonReducer.ts` can gate ability checks without needing the full `CreatedCharacter` object. Abilities resolvable entirely client-side instead just read `character.race`/`character.cls` directly, since the owning component (`TownScreen`, `RoomInspector`) already has one or the other in scope.
- **Combat** (`dungeonReducer.ts`): Grave Digger's +2 damage checks `monster.abilities.includes("undead")` in `attackBonus()`, not a name/tag match (Undead-ness is a monster *ability*, not something every undead monster's name announces). Cook's +1 coin (except Undead) and Slimemen's `CombatState.engulfableBodies` tracking both live in `handleMonsterDefeat()`'s non-revived-kill branch -- engulfable bodies accrue for every race (harmless if never consumed), but only Slimemen ever sees the `ENGULF_BODY` action/button (`CombatPanel`'s "Engulf Body" -- consumes one body, heals to `maxHp`, then the round continues exactly like `CAST_SPELL`, monster counter-attack included). Rinoceroid's horn attack is a second per-monster button in `CombatPanel` alongside the normal Attack, dispatching `PLAYER_ATTACK` with `useHorn: true`; the reducer computes a flat `1d6` (no modifier) and skips every equipped-weapon effect (`attackBonus`/`attackMultiplier`'s new `isHorn` parameter, `instantKillOnRoll`, lifesteal) since the horn isn't the equipped weapon -- general combat buffs (Potion of Fury, Grave Digger) still apply.
- **Doors/torches** (`dungeonReducer.ts`): Locksmith's free lock-picking and Lumberjack's chance to gain a torch from breaking a door both live in `RESOLVE_DOOR_LOCK`'s existing `pickLock`/`breakDoor` branches. Miner's "if you run out of torches, you can leave the dungeon" is a `spendTorches()` special case -- instead of the Darkness killing the character, the failed action just... fails, leaving them alive with 0 torches to use the already-existing Retreat to Town button, rather than building a new "forced exit" flow for what's really just "don't die here."
- **Town** (`town.ts`): Blacksmith's `fixArmor()`/`canFixArmor()` take an `isBlacksmith` flag that swaps the cost from 1 coin to 1 torch; Cat-Person's `sellItem()` takes an `isCatPerson` flag that doubles the credited coins. Both are optional params (default `false`) so every other caller/test is unaffected.
- **Dwarf** (`RoomInspector.tsx`, entirely client-side): Find Secret Passage rolls two dice instead of one when `state.raceName === "Dwarf"` and keeps the higher (row 1 is the table's only bad outcome, so discarding the lower die is strictly an advantage) -- reuses `DicePool` (already used for Chests) instead of the single-die `Die` component.

**Monster table re-roll on return**: per the rulebook, resting in Town (or a new character taking over via `RESUME_DUNGEON`) means fresh monsters may have moved into any room that's currently empty -- "you must roll on the Monster table for each empty room you enter," and explicitly "also true if you die and venture out with another character in the same dungeon." `restoreMapFromPersisted()` (shared by both actions, see above) flags every already-built room-type segment that's currently empty or already-cleared (`SegmentState.needsMonsterReroll`) except whichever segment just had its interrupted fight respawned at full HP. The reroll itself is resolved lazily, by `SELECT_SEGMENT` (via the new `rerollMonstersIfNeeded()` helper), the next time the player actually looks at that room -- not eagerly for the whole map at once, since this app only has one `DungeonState.combat` slot at a time and eagerly rolling every empty room could produce several newly-occupied rooms with no way to fight more than one of them. `restoreMapFromPersisted()` also resolves it immediately for whichever segment the player ends up standing in right after the restore -- their own level's entry segment for both `RESUME_DUNGEON` and `RETURN_TO_DUNGEON` (see **Positional movement** below), so returning doesn't leave the player's own starting room silently unrolled. Only `monsters`/`monstersDefeated` are touched -- `roomContent`, chests, and secret-passage state are left exactly as found, since the rulebook penalty is specifically about monsters repopulating, not the room's contents resetting. The flagging loop also skips `seg.isEntrance` (issue #55) -- the entrance is guaranteed empty (Monsters is never rolled there at creation, see #43 above) and, being wherever a `RESUME_DUNGEON` character lands, would otherwise get flagged and immediately rerolled into a fight the instant the dungeon loaded, defeating #43's whole point.

**Positional movement**: `DungeonState.currentSegId` is the segment the player physically occupies -- distinct from `selectedSegId` (what `RoomInspector` displays), though `SELECT_SEGMENT` always keeps the two in lockstep: selecting anything other than the segment you're already in only succeeds if it's *reachable*, and succeeding moves you there (also resolving that segment's pending Monster re-roll, see above). `reachableSegIds()` (`dungeon.ts`) computes the fog-of-war boundary for a level: `currentSegId` itself plus every segment directly connected to it by an already-opened door, walking each segment's doors once to build the (otherwise parent → child only, via `childId`) link in both directions. Segments outside this set render greyed out on `DungeonMap` and can't be selected, and their unopened doors can't be clicked either -- `OPEN_DOOR`/`RESOLVE_DOOR_LOCK` both guard on `action.segId === state.currentSegId` server-side (the UI already disables the buttons, but the reducer is the actual authority), so a new door can only ever be pushed open from wherever the player stands. `currentSegId` auto-advances the moment a door opens into a newly-built segment (all 6 `OPEN_DOOR` classification branches set it alongside `door.childId`) -- opening a door *is* walking through it. Stepping back through an *already-opened* staircase (`DungeonMap`'s descend button, as opposed to `LevelTabs`, which only changes which level's map is *displayed*) also moves the player: `SWITCH_LEVEL` takes an optional `segId` used only by that button, validated the same way as any other move (`reachableSegIds` happily returns a childId belonging to a different level here -- harmless, since cross-level doors are the *only* way that ever happens, and the reducer's own `OPEN_DOOR` branches are what populate them in the first place). A level other than wherever `currentSegId` actually is ends up with an empty reachable set purely as an emergent property of this -- none of its segment ids will ever equal `currentSegId` or be one of its neighbors -- so `LevelTabs` doubles as free, read-only history browsing for any level the player has already moved past, with no separate "is this level still enterable" flag to maintain. `restoreMapFromPersisted()` resets `currentSegId` (and `selectedSegId`) to `activeLevel`'s own entry segment (`segments[0]`) for *both* `RESUME_DUNGEON` and `RETURN_TO_DUNGEON` -- not all the way back to Level 1 for a multi-level run (the level transition itself is already a one-way, already-completed step) but still requiring a walk back through whichever level the player is currently on, so #18's Monster re-roll can't be silently bypassed by landing directly on a stale, freshly-repopulated room. An interrupted fight (see above) is the one exception -- it still resumes exactly where it was, full HP restored, rather than requiring a walk back into it.

**Room Content rewards**: `RoomContentEntry.reward` (a `RoomContentReward` discriminated union: `coins`/`treasures`/`magicScrolls`/`magicItems`, each carrying a `MonsterCount` -- reusing the same type and `resolveMonsterCount()` helper the Monster tables already use for variable group sizes) flags the Room Content rows across all six dungeon types that describe an automatic, mechanically-applied reward rather than passive flavor text. `applyRoomContentReward()` in `dungeonReducer.ts` resolves it the moment the room is built: `coins` credits `DungeonState.coins` (times an optional `multiplier`, for rows like Sanctuary's "2d6 paintings, 2 coins each"), `treasures` credits `DungeonState.treasures`, `magicScrolls` grants that many random Basic Spell uses (one `SPELL_TABLE` roll per scroll, each adding a use via the same bump `computeSpellUses()`'s callers already expect), and `magicItems` rolls that many Magic Items via the existing `resolveMagicItem()` (see Armor & Weapons) -- reusing it here is why a `magicItems` reward consumes two `rollDie()` calls per item, not one (the Magic Item table roll, then that item's own base Armor/Weapon table roll). `finishRoomSegment()` (also new, in `dungeonReducer.ts`) composes this with the existing `startCombatIfMonsters()` call at all 4 real room-building call sites (entrance, `reuse-normal`, `descend-normal`, ordinary `normal` doors) -- reward first, then combat -- so a room can hand out its automatic loot even if it also turns out to hold monsters. The 2 Final-Room/Boss call sites are untouched, since Final Rooms never populate `roomContent`.
