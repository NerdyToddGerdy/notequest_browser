# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser-based digital adaptation of *NoteQuest* (by Tiago Junges), a solo, dice-driven, pen-and-paper dungeon-crawler board game. Target deployment is GitHub Pages (static hosting, no backend).

`docs/game-rules-reference.md` is the authoritative transcription of the rulebook (Core Book + Expanded World) — treat it as the source of truth for any table, mechanic, or dice logic. Don't invent rules not present there. The purchased rulebook PDF itself is intentionally not tracked in git (see `.gitignore`) since this repo is public.

`design/mockups/*.html` are standalone, self-contained (fonts embedded as base64) interactive layout studies used to validate screen designs before they're built as real components. They are not part of the app build, and are left as-is once ported — don't edit them expecting it to affect the app. Both current mockups (`character-creation.html`, `dungeon-building.html`) have been ported to real code; treat the mockups as historical design reference, not spec-of-record (the real code has since diverged in places, e.g. the dungeon screen's sidebar).

**Scope discipline**: v1 targets the Core Book rules only (character creation, procedural dungeon crawl, combat, torches/darkness, death/graveyard). Expanded World content (hexcrawl, cities, politics, warfare, multiplayer, advanced classes) is an explicit future phase — don't build toward it prematurely.

## Commands

```
npm run dev       # start the Vite dev server
npm run build     # typecheck (tsc -b) then production build to dist/
npm run preview   # serve the production build locally
npm run lint      # eslint .
npm run format    # prettier --write .
npm run test      # vitest run (single pass)
npm run test:watch
```

Run a single test file: `npx vitest run src/engine/__tests__/character.test.ts`.

Node 20.17 is installed locally; some newer devDependencies want >=20.19 and npm prints `EBADENGINE` warnings on install. These are warnings only — install/build/test all work — but if a dependency install starts hard-failing, that mismatch is the first thing to check.

## Architecture

**Data-driven engine, not per-screen hardcoding.** Game rules live as typed data in `src/data/` (`races.ts`, `classes.ts`, `spells.ts`, `dungeonTypes.ts` — Segments/Secret Passage/Dungeon Name tables, shared across all 6 dungeon types — and `dungeonTables.ts` — the per-type Trap/Room Content/Monsters tables), keyed by dice-roll value, e.g. `RACE_TABLE[7]` for a 2d6 roll of 7. Pure game logic lives in `src/engine/` and takes an injectable `RNG = () => number` (defaulting to `Math.random`) so tests can pass a seeded generator (`src/test/mulberry32.ts`) for deterministic assertions instead of asserting on ranges. UI components consume `src/engine` functions and never re-implement roll logic themselves.

**The dungeon engine** (`src/engine/dungeon.ts`, `dungeonState.ts`, `dungeonReducer.ts`) is the more involved half. A dungeon is a tree of `levels[]`, each holding `segments[]` (rooms/corridors/staircases, positioned in an abstract pixel coordinate space for the map) and `connectors[]`. `dungeon.ts` holds two kinds of pure logic that are easy to conflate but are kept separate in the file: (1) *rules* — table rolls, `classifyDoorOpen()` (whether opening a given door is an ordinary roll, a staircase descent, the automatic depth-3 Final Room, the "no stairs ever found" dead-end Final Room, or a second staircase reusing an already-discovered level/Final Room); (2) *spatial layout* — `placeChild`/`placeIslandRoot`'s collision-avoiding placement algorithm, which is **not** a NoteQuest rule, just this project's own way of drawing a coherent map (the physical game doesn't need coordinates). `dungeonReducer.ts` applies `classifyDoorOpen()`'s result via Immer (`produce`) — deeply-nested state (`levels[i].segments[j].doors[k]`) makes manual immutable spreads unwieldy, hence the dependency. The reducer's extra `rng` parameter (for the silent Room Content/Monsters rolls) means it can't be passed directly to React's `useReducer` — `DungeonScreen.tsx` wraps it in a 2-arg `reduceDungeon` first.

**Torches and doors**: `DungeonState.torches`/`alive` track the resource that actually matters in this game — every door first rolls the shared `OPEN_DOOR_TABLE` (trap/locked/unlocked) via `RESOLVE_DOOR_LOCK`, *before* `OPEN_DOOR` ever runs; only "unlocked" (or a resolved lock/trap) proceeds to the existing Segments-table roll. Torch-spending (entering the dungeon, picking a lock, searching a secret passage, the ditch trap's `torchCost`) all flow through the reducer's `spendTorches()` helper, which flips `alive` to `false` (the Darkness) instead of going negative. `DungeonMap.tsx`'s door-click handler mirrors the reducer's math on the *client* side (checking `state.torches` before offering "Pick Lock", before animating a segment roll after a trap) purely so it knows whether to bother animating further dice or stop — the reducer is still the sole authority on whether the character actually dies. Locked doors show a "Pick Lock (1 torch) / Break Door (free)" popover; breaking a door or setting off a trap alerts the monsters beyond it (see Combat below).

**Combat** (`src/engine/combat.ts`) is pure, RNG-injected logic exactly like the rest of the engine; `dungeonReducer.ts` is the only thing that mutates `DungeonState.combat`/`hp`/`coins`. Whenever `buildSegment()` creates a new room segment (dungeon entrance, an ordinary door, a fresh staircase-descent level, or a second-staircase reuse) and that room's `resolveRoomExtras()` roll produced a `MonsterTemplate`, `startCombatIfMonsters()` spawns a `CombatState` via `spawnMonsters()`. Whether the player or the monsters get the first attack is decided by `wasNoisy` (true only when a door was broken open or a trap fired — see the "Combat" section of `docs/game-rules-reference.md`); a noisy start immediately runs one `applyMonsterTurn()` before the player can act. Once `state.combat` is non-null, `OPEN_DOOR`/`RESOLVE_DOOR_LOCK`/`ROLL_SECRET_PASSAGE`/`SWITCH_LEVEL` all no-op (the dungeon map itself also disables door buttons) — the only action is `PLAYER_ATTACK`, one full round per dispatch: `resolvePlayerAttack()` resolves the player's hit (Stoneskin/Intangible/Weakness, plus whichever roll-of-1 special ability fires — Explosive short-circuits into an immediate self-destruct that damages the player directly, bypassing normal damage math), the reducer applies the resulting events (Firebreath/Sorcery queue `bonusDamage`, Deathtouch/Paralyze queue onto the monster for its *own* next attack, Regeneration heals immediately, Horde/Necromancy always spawn a fixed `HORDE_ORC`/`NECROMANCY_SKELETON` regardless of whether the triggering monster died), then `checkUndeadRevival()` on any newly-defeated Undead monster before removing it. If any monsters remain, `applyMonsterTurn()` closes out the round (summing `damage + bonusDamage`, consuming queued Paralyze/Deathtouch, then clearing those queues) — paralysis is tracked as `combat.paralyzedTurns` and simply skips the player's next N `PLAYER_ATTACK` dispatches (monsters still act). Victory (`finishIfVictorious()`) rolls `rollLoot()` once per Loot-tagged monster actually removed from the fight, credits `coins` (Treasure/Key results are logged as flavor only — no inventory yet), marks the segment `monstersDefeated`, and clears `combat`. There is no armor system, so all damage — including Poison, which per the rulebook only matters for bypassing armor — goes straight to `DungeonState.hp`; hitting 0 sets `alive = false` and `deathCause = "combat"` (as opposed to `"darkness"` from running out of torches), which `DungeonScreen.tsx` uses to pick the death message. `CombatPanel.tsx` is the only UI surface for this — it rolls the player's weapon die (`character.cls.weaponDamage`, stored on `DungeonState.weaponFormula` at dungeon creation) and dispatches `PLAYER_ATTACK` with the raw roll; the reducer, not the component, applies the weapon's modifier.

**Screens** live in `src/ui/screens/<ScreenName>/`, one file + one CSS Module each. `App.tsx` holds the only navigation state (`character: CreatedCharacter | null`) and switches between `CharacterCreationScreen` and `DungeonScreen` — there's no router, and with only two screens that's a deliberate choice, not an oversight; revisit if a third screen needs it. **Shared components** live in `src/ui/components/`: `Die`/`DicePool` (see below), `CharacterSheet` (renders a `CreatedCharacter`, used in the dungeon screen's sidebar), and the dungeon-specific `DungeonMap`/`LevelTabs`/`RoomInspector`/`RollLog`. `src/ui/rollTiming.ts` has the shared `revealDelay()` timing helper both screens use to delay revealing a roll's result until its dice animation finishes.

**`Die` component** (`src/ui/components/Die/Die.tsx`) is a real 3D CSS cube (six absolutely-positioned faces inside a `perspective` container, `rotateX`/`rotateY` per face), not an image or icon-swap. It's driven by a `value` (1-6) + `rollToken` (bump this to trigger a new roll animation even if the value repeats) + optional `delayMs` (for staggering dice in a pool) + `size` prop. The rotation math (`SHOW_ROTATION` table + `angleTo` helper) picks the shortest forward rotation to the target face and adds a couple of extra full turns for a tumble effect — see the component for the derivation. `DicePool` just renders N `Die`s with staggered delays for 2d6/etc. rolls. Screens own the *timing* of when to reveal a roll's result (see `revealDelay()` in `CharacterCreationScreen.tsx`) — the Die component itself doesn't expose a completion callback.

**Theme** (`src/ui/theme/`): `tokens.css` (CSS custom properties — the ink/parchment/gold palette, font stacks), `fonts.css` (`@font-face` for the three self-hosted fonts in `fonts/`: Metamorphous for display, Spectral for body, JetBrains Mono for stats/numbers — all OFL, embedded locally, no CDN), `global.css` (resets + the torchlit-dungeon body background, imported once in `main.tsx`). Component-level styling uses CSS Modules (`*.module.css`); shared tokens/fonts are global.

**Testing**: Vitest, default environment is `"node"` (fast, no DOM). A `jsdom` dependency in this environment currently throws `ERR_REQUIRE_ESM` when jsdom is loaded as the *global* Vitest environment — if you add component tests with React Testing Library, opt that specific file into jsdom via a `// @vitest-environment jsdom` docblock at the top of the file rather than changing the global config, or re-diagnose the jsdom/csstools ESM interop issue first.

**Deployment**: `vite.config.ts` sets `base` conditionally on `process.env.GITHUB_ACTIONS` (`/notequest_browser/` in CI, `/` locally) for GitHub Pages project-page hosting. No GitHub Actions workflow exists yet — that's still a TODO.

**Spellcasting** (`DungeonState.spellUses`, a `Record<spellRoll, remainingUses>` seeded at dungeon creation by `computeSpellUses()` from the character's `spells`/`fixedGrants`) covers all six Basic Spells via the `CAST_SPELL` action. Heal and Light are usable anytime (in `CombatPanel` or, out of combat, as a "Cast" button on `CharacterSheet`); Cold Ray, Lightning, and Fireball require an active fight and are rendered as extra per-monster or room-wide buttons in `CombatPanel`. Spell damage runs through `resolveSpellDamage()` in `combat.ts`, which shares Stoneskin/Intangible filtering with `resolvePlayerAttack()` but — since it's a fixed value, not a weapon "damage roll" — never triggers the roll-of-1/roll-of-6 abilities (Explosive, Firebreath, Weakness, etc.). Casting any spell mid-fight consumes the round exactly like `PLAYER_ATTACK` (the monsters counter-attack afterward), except Cold Ray also flags its target `skipNextAttack` so it sits out that round, and Teleport instead ends combat immediately with no counter-attack — this implementation's stand-in for "fleeing a fight," since the app has no notion of player position to send a rules-accurate Teleport to an out-of-combat destination room. A spell's uses, once spent, stay spent — there's no Rest yet to recover them.

**The Boss fight**: each dungeon type's `DUNGEON_TABLES[key].boss` (a `Record<1-6, MonsterTemplate>`, authored from the rulebook's per-dungeon "Table: Boss") is rolled once via `resolveBoss()` in `dungeon.ts` the moment the Final Room segment is built — both in the `descend-final` case (auto-triggered at depth 3) and `dead-end-final` (no stairs ever found). Per the rulebook, there's no Content/Monsters roll alongside it, only the Boss. The rolled `MonsterTemplate` is attached to the Final Room's existing `SegmentState.monsters` field and fed through the same `startCombatIfMonsters()` plumbing as an ordinary room, just with a new `isBoss` flag threaded onto `CombatState` — `finishIfVictorious()` checks that flag to grant a flat 2d6 Treasures instead of the normal per-monster Loot table (Treasures are flavor-only, like ordinary Loot, since there's no inventory system). `DungeonScreen.tsx` shows a gold victory panel once the Final Room's boss is marked defeated.

**Still stubbed / not yet built** (see the scope note rendered on the dungeon screen itself, which is the live source of truth for this list): the armor system, race/class special combat abilities (Grave Digger's +2 vs Undead, Cook's coin-per-kill, Rinoceroid's horn attack), Loot's Treasure/Key results (flavor-only, not inventory items); HP damage from traps beyond the torch-costing ditch trap (blade/dart/etc. are logged as flavor only); Town/City actions (Rest, Fix Armor, Buy Torches, Sell Items — there's currently no way to refill torches or heal once in the dungeon); any persistence (a reload loses all progress).
