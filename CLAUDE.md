# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser-based digital adaptation of *NoteQuest* (by Tiago Junges), a solo, dice-driven, pen-and-paper dungeon-crawler board game. Target deployment is GitHub Pages (static hosting, no backend).

`docs/game-rules-reference.md` is the authoritative transcription of the rulebook (Core Book + Expanded World) — treat it as the source of truth for any table, mechanic, or dice logic. Don't invent rules not present there. The purchased rulebook PDF itself is intentionally not tracked in git (see `.gitignore`) since this repo is public.

`design/mockups/*.html` are standalone, self-contained (fonts embedded as base64) interactive layout studies used to validate screen designs before they're built as real components. They are not part of the app build, and are left as-is once ported — don't edit them expecting it to affect the app. Both current mockups (`character-creation.html`, `dungeon-building.html`) have been ported to real code; treat the mockups as historical design reference, not spec-of-record (the real code has since diverged in places, e.g. the dungeon screen's sidebar).

**Scope discipline**: v1 targets the Core Book rules only (character creation, procedural dungeon crawl, combat, torches/darkness, death/graveyard). Expanded World content (hexcrawl, cities, politics, warfare, multiplayer, advanced classes) is an explicit future phase — don't build toward it prematurely.

## Architecture

**Data-driven engine, not per-screen hardcoding.** Game rules live as typed data in `src/data/` (`races.ts`, `classes.ts`, `spells.ts`, `dungeonTypes.ts` — Segments/Secret Passage/Dungeon Name tables, shared across all 6 dungeon types — and `dungeonTables.ts` — the per-type Trap/Room Content/Monsters tables), keyed by dice-roll value, e.g. `RACE_TABLE[7]` for a 2d6 roll of 7. Pure game logic lives in `src/engine/` and takes an injectable `RNG = () => number` (defaulting to `Math.random`) so tests can pass a seeded generator (`src/test/mulberry32.ts`) for deterministic assertions instead of asserting on ranges. UI components consume `src/engine` functions and never re-implement roll logic themselves.

**The dungeon engine** (`src/engine/dungeon.ts`, `dungeonState.ts`, `dungeonReducer.ts`) is the more involved half. A dungeon is a tree of `levels[]`, each holding `segments[]` (rooms/corridors/staircases, positioned in an abstract pixel coordinate space for the map) and `connectors[]`. `dungeon.ts` holds two kinds of pure logic that are easy to conflate but are kept separate in the file: (1) *rules* — table rolls, `classifyDoorOpen()` (whether opening a given door is an ordinary roll, a staircase descent, the automatic depth-3 Final Room, the "no stairs ever found" dead-end Final Room, or a second staircase reusing an already-discovered level/Final Room); (2) *spatial layout* — `placeChild`/`placeIslandRoot`'s collision-avoiding placement algorithm, which is **not** a NoteQuest rule, just this project's own way of drawing a coherent map (the physical game doesn't need coordinates). `dungeonReducer.ts` applies `classifyDoorOpen()`'s result via Immer (`produce`) — deeply-nested state (`levels[i].segments[j].doors[k]`) makes manual immutable spreads unwieldy, hence the dependency. The reducer's extra `rng` parameter (for the silent Room Content/Monsters rolls) means it can't be passed directly to React's `useReducer` — `DungeonScreen.tsx` wraps it in a 2-arg `reduceDungeon` first.

**Torches and doors**: `DungeonState.torches`/`alive` track the resource that actually matters in this game — every door first rolls the shared `OPEN_DOOR_TABLE` (trap/locked/unlocked) via `RESOLVE_DOOR_LOCK`, *before* `OPEN_DOOR` ever runs; only "unlocked" (or a resolved lock/trap) proceeds to the existing Segments-table roll. Torch-spending (entering the dungeon, picking a lock, searching a secret passage, the ditch trap's `torchCost`) all flow through the reducer's `spendTorches()` helper, which flips `alive` to `false` (the Darkness) instead of going negative. `DungeonMap.tsx`'s door-click handler mirrors the reducer's math on the *client* side (checking `state.torches` before offering "Pick Lock", before animating a segment roll after a trap) purely so it knows whether to bother animating further dice or stop — the reducer is still the sole authority on whether the character actually dies. Locked doors show a "Pick Lock (1 torch) / Break Door (free)" popover; breaking a door or setting off a trap alerts the monsters beyond it (see Combat below). Beyond `torchCost`, `TrapEntry` also carries `damage` (flat HP loss, e.g. Acid Spout), `bladeTrap` (the identical row-1 Blade Trap across every dungeon type: a silent roll of 1 kills outright, a 2 "loses an arm" as flavor only -- no hand-economy system exists to enforce that), and `monsters` (an ambush -- Crypt's Bats, Tomb's Skeletons -- spawned into combat exactly like a room's own encounter, always noisy). All three trap-firing sites (`RESOLVE_DOOR_LOCK`, `ROLL_SECRET_PASSAGE`, `ROLL_CHEST`'s empty-chest trap) funnel through `resolveTrapOutcome()`, which checks for a `trapImmunity` item first (consuming it and blocking the whole trap, torch cost included) before falling through to `applyTrapEffect()` for whichever of the three mechanical shapes applies.

**Combat** (`src/engine/combat.ts`) is pure, RNG-injected logic exactly like the rest of the engine; `dungeonReducer.ts` is the only thing that mutates `DungeonState.combat`/`hp`/`coins`. Whenever `buildSegment()` creates a new room segment (an ordinary door, a fresh staircase-descent level, or a second-staircase reuse) and that room's `resolveRoomExtras()` roll produced a `MonsterTemplate`, `startCombatIfMonsters()` spawns a `CombatState` via `spawnMonsters()` -- except the dungeon entrance itself, which is exempt from Monsters entirely (a deliberate design/balance call, not from the rulebook, which says nothing about the entrance specifically): `resolveRoomExtras()` takes an `isEntrance` flag that still rolls the Monsters dice, for RNG-consumption parity with every other room, but discards the result, so the very first room a player steps into can never ambush them before they've opened a single door. Whether the player or the monsters get the first attack is decided by `wasNoisy` (true only when a door was broken open or a trap fired — see the "Combat" section of `docs/game-rules-reference.md`); a noisy start immediately runs one `applyMonsterTurn()` before the player can act. Once `state.combat` is non-null, `OPEN_DOOR`/`RESOLVE_DOOR_LOCK`/`ROLL_SECRET_PASSAGE`/`SWITCH_LEVEL` all no-op (the dungeon map itself also disables door buttons) — the only action is `PLAYER_ATTACK`, one full round per dispatch: `resolvePlayerAttack()` resolves the player's hit (Stoneskin/Intangible/Weakness, plus whichever roll-of-1 special ability fires — Explosive short-circuits into an immediate self-destruct that damages the player directly, bypassing normal damage math), the reducer applies the resulting events (Firebreath/Sorcery queue `bonusDamage`, Deathtouch/Paralyze queue onto the monster for its *own* next attack, Regeneration heals immediately, Horde/Necromancy always spawn a fixed `HORDE_ORC`/`NECROMANCY_SKELETON` regardless of whether the triggering monster died), then `checkUndeadRevival()` on any newly-defeated Undead monster before removing it (skipped entirely if the player holds an `ignoresMonsterAbility: "undead"` item — see Armor & Weapons). If any monsters remain, `applyMonsterTurn()` closes out the round: it splits the tally into `poisonDamage` (Poison "cannot be absorbed by armor or other means," so it always hits `hp` immediately, before anything else) and `absorbableDamage` (everyone else's `damage + bonusDamage`, deferred to `combat.pendingDamage` — see Armor & Weapons — if the player has usable armor, else applied to `hp` the same way poison was), then consumes queued Paralyze/Deathtouch and clears those queues — paralysis is tracked as `combat.paralyzedTurns` and simply skips the player's next N `PLAYER_ATTACK` dispatches (monsters still act). Victory (`finishIfVictorious()`) rolls `rollLoot()` once per Loot-tagged monster actually removed from the fight, credits `coins` (Treasure/Key results are logged as flavor only — no inventory yet), marks the segment `monstersDefeated`, and clears `combat`. Hitting 0 HP sets `alive = false` and `deathCause = "combat"` (as opposed to `"darkness"` from running out of torches) and clears `draft.combat = null` too, at every one of its four death sites (`applyMonsterTurn()`'s Deathtouch and normal counter-attack branches, `PLAYER_ATTACK`'s Explosive self-destruct branch, and `RESOLVE_DAMAGE`'s deferred-armor-choice branch) — otherwise `CombatPanel` kept rendering (and its Attack button kept rolling dice and dispatching no-op `PLAYER_ATTACK`s) right alongside the death overlay, since `DungeonScreen.tsx` renders it on `{state.combat && ...}` alone; `CombatPanel`'s own `canAct` also checks `hp > 0` directly as a second line of defense, in case a future death path forgets to clear `combat` too. `DungeonScreen.tsx` uses `deathCause` to pick the death message. `handleMonsterDefeat()`'s non-revived-kill branch also tallies `killsByName` (keyed by the monster's own lowercased `name`) and `killsByAbility` (one increment per ability the monster has, so a multi-ability monster counts toward more than one) alongside the existing `monsterKills`/`bossKills` split — the same "no formal taxonomy, just match the string" precedent Armor & Weapons' tag-matching already established, added specifically to unblock Advanced Classes' (#23) kill-count requirements like Ruthless's "10 Imps" (name-based) or Ghostbuster's "10 intangible beings" (ability-based) that `monsterKills`/`bossKills` alone can't answer. Both fields follow the exact same lifecycle as `monsterKills`/`bossKills` everywhere they're already threaded (`AdventurerResources`, `RESUME_DUNGEON` resets them to `{}` for a new character, `RETURN_TO_DUNGEON` carries them over exactly). `CharacterSheet`'s stat row shows `monsterKills` as a plain "Kills" number, but it's also a `<button>` (`.statClickable`) opening `KillBreakdownModal` (`src/ui/components/KillBreakdownModal/`) -- a dismiss-only informational modal (the first of its kind; `ConfirmDialog`'s backdrop/overlay styling was the visual starting point, but this has no confirm/cancel choice) listing `killsByName`, sorted highest-count first, capitalized for display (the underlying keys stay lowercased, matching `matchesTags()`'s own case-insensitive precedent). `CombatPanel.tsx` is the primary UI surface for this — it rolls the player's weapon die (`state.weapon?.formula ?? character.cls.weaponDamage` — see Armor & Weapons for how a found weapon overrides the class one) and dispatches `PLAYER_ATTACK` with the raw roll; the reducer, not the component, applies the weapon's modifier and any bonus effects. When `combat.pendingDamage` is set, `CombatPanel` instead renders an absorption chooser (HP or a specific armor piece, each showing current/max) that dispatches `RESOLVE_DAMAGE`; `PLAYER_ATTACK`/`CAST_SPELL`/`OPEN_TREASURE` all no-op until it's resolved, same as the existing paralyzed-turn gate.

**Screens** live in `src/ui/screens/<ScreenName>/`, one file + one CSS Module each. `App.tsx` holds the only navigation state and switches between `CharacterCreationScreen`, `WorldScreen`, and `DungeonScreen` — there's no router. `TownScreen` isn't in that list: it's a hex-generic component `WorldScreen` renders in place of the map whenever the player is standing on a City/Fortress hex (see Town/Town Square unification below), not a separate `App.tsx`-level screen of its own. Flow is `CharacterCreationScreen` → `WorldScreen` (landing on home, a city hex, so `TownScreen` is what's actually shown first) → `DungeonScreen`, with `DungeonScreen` able to send the player back to World (a voluntary retreat, alive) or all the way back to `CharacterCreationScreen` (death, via `onNewAdventurer` — a dead character is permanently dead, there's no way to resurrect or keep playing one past `!state.alive`). See **Town** below for how resources move between screens. **Shared components** live in `src/ui/components/`: `Die`/`DicePool` (see below), `CharacterSheet` (renders a `CreatedCharacter` plus live stats, used in both `TownScreen`/`WorldScreen`'s map view and the dungeon sidebar), `Equipment` and `Pack` (see Armor & Weapons / Opening a Treasure; each takes an optional `onFixArmor`/`onSell` to add a per-row action button, used only in `TownScreen`), `Graveyard` (see below), and the dungeon-specific `DungeonMap`/`LevelTabs`/`RoomInspector`/`RollLog`. `src/ui/rollTiming.ts` has the shared `revealDelay()` timing helper screens use to delay revealing a roll's result until its dice animation finishes.

**The Graveyard** (`src/engine/graveyard.ts`) was the first piece of state to survive a reload; `session.ts` (below) later generalized the same pattern to everything else. `loadGraveyard()`/`addGraveyardEntry()` read/write a JSON array under the `notequest:graveyard` localStorage key; both take an injectable `storage: Storage` parameter (mirroring the RNG-injection pattern elsewhere in `engine/`) so their tests run in Vitest's default Node environment, which has no `localStorage`. `DungeonScreen` records an entry in a `useEffect` keyed on `state.alive` flipping to `false` (so it fires exactly once per death, not on every re-render of the death panel), then `CharacterCreationScreen` loads the list once on mount (a fresh mount happens every time the player returns here, including right after a death). It's also shown on `TownScreen` (loaded the same way, once on mount) so a still-living character can see who's already died without waiting for their own death.

**`RecordsPanel`** (`src/ui/components/RecordsPanel/`) is what `CharacterCreationScreen`/`TownScreen` actually render in the Graveyard's old spot -- a tab switcher between `Graveyard` and the newer `DungeonsList` (`src/ui/components/DungeonsList/`, mirroring the Graveyard's own "running record" panel styling, fed `dungeonHistory` unfiltered -- every dungeon any character has touched, same "every adventurer" scope as the Graveyard's own list), shown only once *both* have at least one entry; with just one populated, that one renders directly with no tab bar, matching the Graveyard's own pre-#45 behavior exactly. `DungeonsList`'s per-entry status (Cleared/Unfinished) reuses `isDungeonBeaten()`; its unrecovered-remains count uses a new `countUnlootedRemains()` (`dungeonState.ts`, a sibling to the existing boolean `hasUnlootedRemains()`) that sums `remains.names.length` across every segment in every level, since a single room's remains can already hold more than one fallen adventurer's name. Clicking an entry doesn't warp/travel to it (unlike the pre-#44 `activeDungeonElsewhere` shortcut this project deliberately removed) -- it's read-only, matching "must travel back to its hex" holding without exception.

**Session persistence** (`src/engine/session.ts`) covers everything else App-level: `character`, `resources`, `dungeonHistory`, and `activeRunId`, as one `SessionState` blob under the `notequest:session` key (same injectable-`storage` pattern as the Graveyard). `App.tsx` seeds its four separate `useState` calls from a single `loadSession()` call on mount, then a single `useEffect` watching all four re-persists the whole blob on any change, rather than threading a save call through every individual setter. `screen`/`selectedRunId` deliberately aren't persisted -- transient navigation state, not worth remembering; a reload just resumes wherever `world.player` physically was (itself persisted), or Character Creation if there's no character. `handleNewAdventurer` nulling out `character`/`resources`/`activeRunId` (while leaving `dungeonHistory` untouched, since the dead character's own abandoned dungeon should stay resumable) is enough to correctly clear the persisted session too, via that same effect -- no separate "clear" call needed.

**Random names** (`src/data/names.ts`, issue #40): not a rulebook mechanic -- a v1-scoped extra. `FIRST_NAME_TABLE`/`LAST_NAME_TABLE` are `Record<string, Record<1-6, string>>` keyed by exact `RaceDef.name` (all 11 races authored, each with its own naming theme -- e.g. Dwarf clan-surnames, Slimemen onomatopoeic first names with descriptor "last names" like "the Damp"), plus a shared `"default"` fallback bucket for any race without a table of its own yet (see issue #22, "New Races"). `rollName()` (`character.ts`) rolls one die each into the two tables and joins the results; `CharacterCreationScreen`'s "Random Name" button (next to the name input) calls it with the currently-rolled race's name (or `"default"` before one's picked) and fills the field directly -- no roll animation, and the result stays freely editable afterward, same as typing a name in by hand.

**Town** isn't a distinct `App.tsx` screen -- `TownScreen` is a hex-generic "Town Square" component that `WorldScreen` renders in full (map hidden) whenever the player is standing on any City/Fortress hex, home included (home is just `world.home`, a human city on a plain, no different from any other city hex once found). It's `WorldScreen` that computes and hands down `hasDungeon`/`dungeonGateCopy`/`onEnterDungeon` for whichever hex the player is currently on (see per-hex dungeon persistence below) and `onExploreWorld` (toggles `WorldScreen`'s own map view back on, see below) -- `TownScreen` itself has no notion of "home" vs. "a city I found," which is what makes City Actions and dungeon access behave identically everywhere. `App.tsx` lifts an `AdventurerResources` object (`src/engine/town.ts` — `torches`/`hp`/`maxHp`/`coins`/`treasures`/`keys`/`heldItems`/`armor`/`weapon`/`spellUses`/`provisions`, i.e. everything a living character carries between dungeon and town) alongside `character`; it's seeded from the character's starting stats the moment `CharacterCreationScreen` calls `onCharacterCreated` (which also resets `world.player` back to `world.home` -- a new adventurer starts fresh at the city, not wherever the last one's body was left -- and always lands on `screen: "world"`, never straight into a dungeon), and overwritten wholesale from the dungeon's own fields whenever `DungeonScreen` calls `onReturnToTown(runId, dungeon)`. `town.ts`'s `rest()`/`buyTorch()`/`buyProvision()`/`sellItem()`/`fixArmor()` are pure functions (no RNG needed -- all fixed-cost per the rulebook) that `TownScreen` calls and hands the result to `onUpdateResources`; `canRest()`/`canBuyTorch()`/`canBuyProvision()`/`canFixArmor()` gate the buttons (1 coin; torches/provisions additionally capped at the rulebook's 10/20, a piece additionally must actually be damaged). Selling reuses `Pack`'s `onSell` prop, crediting the coins a `HeldItem` already carries as its `worth` (see Opening a Treasure) rather than re-rolling a price; fixing reuses `Equipment`'s `onFixArmor` prop the same way (see Armor & Weapons). `rest()` heals to `resources.maxHp` specifically (not a hardcoded/derived value) — this only works correctly because `maxHp` is threaded as its own field everywhere a dungeon gets (re)constructed (see the `maxHp` note below); before that fix, returning to a dungeon below full HP silently clamped the character's *true* max HP down to whatever they currently had, making Rest heal to a shrinking ceiling instead of the real max.

**Resuming an unbeaten dungeon**: per the rulebook, dying (or otherwise leaving) doesn't erase a dungeon -- a later character can pick up exactly where it was left, and returning to town to rest doesn't either. `App.tsx` lifts a `dungeonHistory: PendingDungeon[]` array (persisted via `session.ts` like everything else App-level, same as the Graveyard's own localStorage persistence in spirit) above the Town/Dungeon screens, keyed by a `runId` (`crypto.randomUUID()`) that a `DungeonScreen` instance mints once at mount and re-mints on `handleStartNewDungeon`. Progress is captured generically in an unmount cleanup (`useEffect(() => () => onLeaveDungeon(...), [])`, reading the *latest* state via a ref kept fresh by its own render-triggered effect, since writing a ref during render itself isn't allowed) — this fires no matter *why* the screen unmounts (death, a "Retreat to Town" click, "Start a New Dungeon", or beating the Final Room), so App's `handleLeaveDungeon` doesn't need to special-case the reason: it just drops the run if it never went anywhere (`levels.length === 0`), otherwise upserts it -- unlike `pendingDungeons` before it, **beaten runs are kept too**, not dropped, purely as a historical record (World's per-hex badges below are the only place this shows up now; `isDungeonBeaten()` is computed on demand wherever "is this one still resumable?" matters, rather than stored as a field). `App.tsx` separately tracks `activeRunId` -- the *current, still-living* character's own paused run, if any -- set whenever `onReturnToTown` reports a non-empty, unbeaten dungeon. Choosing what to enter next just sets App's `selectedRunId` and switches to the dungeon screen -- every case (this character's own active run, a fresh roll, or taking over a different, abandoned run) now comes from `WorldScreen`'s `onEnterDungeon`, per whichever hex the player is standing on (see per-hex dungeon persistence below); there's no separate "continue" action anymore since standing on the hex your own run is tied to and clicking "Enter Dungeon" resolves to the exact same place. App then derives, from whether `selectedRunId === activeRunId`, which of two very different resume paths `DungeonScreen`'s mount-time `useReducer` initializer takes (both land in the same `restoreMapFromPersisted()` reducer helper for the shared map-copying/monster-respawn logic, but differ in what happens to resources and where the player starts):
- **`RESUME_DUNGEON`** -- a *new* character taking over a *dead* one's map, found by traveling back to the hex it's tied to. Resets every character-specific field (`torches`/`hp`/`maxHp`/`weaponFormula`/`spellUses`/`coins`/`treasures`/`keys`/`heldItems`/`armor`/`weapon`) to the new character's own starting `resources`, logs "A new adventurer takes up the fallen's path.", and -- since `restoreMapFromPersisted()`'s `resetToEntrance` flag is `true` for this path -- forces `activeLevel`/`selectedSegId` back to the entrance segment regardless of where the previous character died deep in the dungeon: "he will find his backpack and clothes on the floor" implies discovering the remains by exploring, not teleporting to them. (The full multi-level map is untouched; only the initially-displayed position resets.) The dead character's own belongings stay behind as recoverable `remains` (see below) -- they are not inherited.
- **`RETURN_TO_DUNGEON`** -- the *same still-living* character coming back to the hex their own run is tied to and clicking "Enter Dungeon" there. `DungeonScreen`'s `useReducer` initializer dispatches this automatically on mount whenever `activeDungeon` is set (skipping the gate entirely), carrying over `resources` *exactly* -- including `coins`/`treasures`/`keys`/`heldItems`, which `RESUME_DUNGEON` deliberately does not -- with `resetToEntrance: false`, so it picks up in the exact room/level it left off in. Logs "You return to the dungeon." instead.

Both actions `structuredClone()` the incoming persisted `DungeonState` before touching it, since Immer deep-freezes everything `produce()` returns and mutating a frozen array throws. Either way, if the previous session died or left mid-fight, that segment's monsters (there can be at most one such segment dungeon-wide, since every other room's combat had already resolved by the time further doors could open) are respawned at full HP via the same `startCombat()` helper ordinary room entry uses, matching "a room that still has monsters recovers their health." Both actions also carry an explicit `maxHp` field (as does the plain fresh-roll path, `DungeonScreen`'s `createInitialDungeonState(..., resources.maxHp)` call) -- `createInitialDungeonState()`'s `startingHp` param only ever set *current* HP historically, defaulting `maxHp` to match it, which was harmless when every entry point started at full health but broke the moment Town made "retreat below full HP, then come back" possible.

**Remains**: "all your equipment will be on the floor of that room to be recovered by your next character" (and the Darkness section says the same). `DungeonState.characterName` (seeded alongside the other character-derived fields at creation/resume) labels whose remains they are. `leaveRemains()` in `dungeonReducer.ts` fires at every death site -- the Darkness branch of `spendTorches()` (now threaded an optional `segId` from each of its six call sites, since the relevant room is whichever segment's action triggered the torch spend, not necessarily the currently-selected one) and both of `applyMonsterTurn()`'s kill branches plus `PLAYER_ATTACK`'s Explosive self-destruct branch (using `combat.segId`, since a combat death's location is unambiguous) -- and drops the dying character's current coins/Treasures/Keys/`heldItems`/`armor`/`weapon` (the Pack and Equipment contents -- see Opening a Treasure / Armor & Weapons) onto `SegmentState.remains`, merging (`names` array grows, counts add, `heldItems`/`armor` concatenate, `weapon` fills in only if nothing was left behind already) if an earlier death already left something there. `RoomInspector` shows a "Remains" row plus a free "Recover Remains" button (no roll, matching the rulebook not listing this as a torch-costing action) that dispatches `COLLECT_REMAINS`, crediting the current character and clearing the segment's `remains`.

**Hexploring the World**: the rulebook's Expanded World hex-travel system (`docs/game-rules-reference.md` lines ~828-1050), built as a deliberately thin first pass with clearly-scoped follow-ups, not the full section -- see below for exactly what's still missing. `src/engine/hexState.ts` models the map as a sparse `Record<string, HexTile>` keyed by `hexKey({q, r})` (axial, pointy-top coordinates, mirroring `DungeonState.segments`' own lazy-reveal pattern -- a tile only exists once its terrain+location are rolled), populated 6-at-a-time by `revealNeighborsInPlace()` ("roll one die in the Terrain table... roll one more die to see if there's a location... if 6, roll on the Location table") every time the player enters a new hex. `createInitialWorldState()` fixes the start at `{0,0}`, a human city on a plain (not rolled -- "Choose a hex to start with"), climate hardcoded `"hot"` (the `COLD_TERRAIN_TABLE` in `src/data/hexTables.ts` is fully authored but nothing selects it yet). `src/engine/hexReducer.ts` is its own small state machine (not an extension of `dungeonReducer.ts` -- different persisted lifetime, no combat) with a single `MOVE` action, validated against `hexNeighbors()` and `isImpassable()` (water, or a `rocks` location) before mutating `player`/re-revealing that hex's own neighbors. `src/ui/screens/WorldScreen/WorldScreen.tsx` renders the map as an SVG (axial-to-pixel projection, hex polygons colored by terrain), with only City/Fortress/Ruins/Rocks locations interactive today -- Portal/Oasis/Volcano/Reef/Thin Ice/"nothing" render as inert flavor labels, their actual entry-roll effects not implemented yet. `App.tsx`'s `Screen` is just `"world" | "dungeon"` (plus Character Creation, gated separately) -- Town isn't its own screen, see Town above; `world: WorldState | null` persists across characters like `dungeonHistory`, created (or player-position-reset to `world.home`) in `handleCharacterCreated`.

- **Provisions**: a new `AdventurerResources` field, parallel to torches but a genuinely separate resource (the rulebook keeps them distinct -- travel/food vs. dungeon light) -- `MAX_PROVISIONS` (20), `canBuyProvision`/`buyProvision` (`src/engine/town.ts`, same 1-coin shape as `buyTorch`), and `payTravelCost()` (spends provisions first; any shortfall at all costs a flat 1 HP for the move, not scaled to how many provisions were actually short -- a Mountain move at 0 provisions costs the same 1 HP as a Plains move at 0 provisions, not 3x as much just because Mountains normally cost more provisions -- deliberately floored at 1 HP besides, so running out of provisions alone can never kill the character, unlike the rulebook's real lethal hex hazards, which are part of the still-deferred Events on Travel/location-effects layer below).
- **Terrain-based Dungeon Type**: `DUNGEON_TYPE_BY_TERRAIN` (`src/data/hexTables.ts`) maps each terrain to a 1d6 table of `DUNGEON_TYPES` roll numbers ("Table: Dungeon Type, by terrain") -- the rulebook's real table references 8 dungeon types this app hasn't built yet (Citadel, Mine, Cave, Laboratory, Pyramid, Ziggurat, Necropolis, plus a Fortress's optional Sewers -- tracked as issue #30, "Deadly Dungeons"), each substituted with the closest thematic match among the 6 that exist (documented on the table itself), same precedent as `bladeTrap`'s flavor-only roll-of-2 or `WeaponEntry.twoHanded`. `App.tsx`'s `onEnterDungeon` handler rolls the current hex's terrain-fated type once and stores it as `forcedTypeRoll`, passed to `DungeonScreen` unconditionally now (every dungeon trip originates from World) -- `handleRollDungeon` uses it for just the first of its 3 dice (`forcedTypeRoll ?? rollDie()`), so the "Roll for Dungeon" ritual and its 2 free name-flavor dice stay intact; only the type itself is fated by where the player found it, not free chance.
- **Town Square unification**: `WorldScreen` renders `TownScreen` in full (not a sidebar card) whenever standing on a City/Fortress hex and not voluntarily viewing the map (`showMap` local state, reset to `false` on every `handleTravel` so arriving anywhere shows "the appropriate thing" -- the city screen if it's a City/Fortress, the map otherwise). `TownScreen` itself stays hex-agnostic: it takes `hasDungeon` (`WorldScreen`'s own `canEnterDungeon` check -- true for a City/Fortress/Ruins regardless of whether a dungeon's been found there yet, covering both a fresh roll and a known one, but *false* once `currentDungeonStatus === "beaten"` -- there's nothing left to do in an already-cleared dungeon, and `RETURN_TO_DUNGEON`/`RESUME_DUNGEON` would just redisplay the existing victory panel rather than letting the Boss be re-fought), `dungeonGateCopy`, `onEnterDungeon`, and `onExploreWorld` (sets `showMap(true)`) -- City Actions (Rest/Buy Torches/Buy Provisions/Fix/Sell) live only in `TownScreen` now, reusing `town.ts`'s pure functions exactly as before. While voluntarily viewing the map from inside a city (`showMap` true), the map view shows a "Return to the City" control (`showMap(false)`) instead of its own redundant Ruins-style "Enter Dungeon" card -- that card (`canEnterDungeon && !inCityOrFortress`) is reachable only for a genuine Ruins hex, which stays map-view-only, no City Actions, matching "Explore as if it were a dungeon" rather than a settlement.
- **Per-hex dungeon persistence**: `HexTile.dungeonRunId` (`src/engine/hexState.ts`) remembers which `PendingDungeon` was found on a given hex, "drawing the dungeon on the map" instead of re-rolling a fresh one every visit -- set once, on first entry, never cleared even once beaten. World's `onEnterDungeon` (`App.tsx`) checks the current hex's tile first: if `dungeonRunId` is already set, it just sets `selectedRunId` to it, letting the existing `isOwnRun`/`resumeDungeon` derivation that already renders `DungeonScreen` pick `RETURN_TO_DUNGEON` (still this character's own paused run) vs. `RESUME_DUNGEON` (someone else's abandoned one) -- no World-specific resume logic needed. On a genuinely first visit, the terrain-fated roll above still happens, but the run's id is minted in `App.tsx` itself (`worldFreshRunId`, passed to `DungeonScreen` as `externalRunId`, used only when `activeDungeon`/`resumeDungeon` are both unset) rather than `DungeonScreen` self-minting one via `crypto.randomUUID()` on mount -- World needs to know the id up front to stamp it onto the hex immediately, which is always safe since there's no way to leave `DungeonScreen` before a dungeon actually exists (its Retreat/Start-New buttons only render once `hasDungeon` is true, a *different*, `DungeonScreen`-local `hasDungeon` than `TownScreen`'s prop of the same name). `dungeonGateCopy` ("a dungeon awaits here" / "your unfinished dungeon is still here" / "the dungeon here has already been cleared") comes from a `dungeonInfoFor()` lookup (`dungeonHistory` + `isDungeonBeaten()`/`hasUnlootedRemains()`) reused per-hex on the map itself -- a small ⚔/✓ SVG badge renders on any hex with a known dungeon (cleared vs. not, visible at a glance without standing on it), plus an independent 💀 badge whenever `hasUnlootedRemains()` is true (a dungeon can be beaten by someone else while an earlier death's loot still sits unrecovered in a different room, so this is checked separately from beaten-status, not folded into it). Mid-run "Start a New Dungeon" *does* re-tie the hex: `DungeonScreen`'s `onRunIdChanged` prop fires with the freshly self-minted id right alongside its own `setRunId`, and `App.tsx` re-stamps whatever hex the player is currently standing on (`withDungeonRunId()`, the same pure helper `onEnterDungeon`'s first-visit branch uses) -- otherwise the abandoned dungeon would stay "the one on the map" even after a completely different one, rolled in its place, gets explored and beaten.

- **`HexInspector`**: `WorldScreen`'s equivalent of the dungeon map's `RoomInspector` -- a `.hexInspectorOverlay` card, positioned/styled byte-for-byte like `.roomInspectorOverlay` (`position: absolute; right; bottom` on the now-`position: relative` `.mapCard`), describing whichever hex is selected (terrain, location, `dungeonInfoFor()`'s status/remains). Read-only, no button of its own: clicking a hex on the map (`WorldScreen.tsx`'s `handleHexClick`) still travels immediately when it's a passable, in-range neighbor -- unchanged from before `HexInspector` existed -- and only *falls back* to selecting it for inspection (local `selectedHex: HexCoord | null`, mirroring `RoomInspector`/`state.selectedSegId`'s selected-vs-current split) when it isn't a valid travel target, since there'd be nothing else useful for that click to do. `selectedHex` falls back to the player's own tile when `null`; `handleTravel` resets it to `null` so arriving anywhere describes the new tile by default.
- **Map zoom + pan** (`src/ui/hooks/useZoomGesture.ts`): a shared gesture-*detection* hook (wheel + two-finger pinch, normalized to a single `onZoom({ factor, clientX, clientY })` callback) used by both `DungeonMap` and `WorldScreen`'s hex map -- deliberately not shared on the *application* side, since the two maps zoom completely differently. `DungeonMap` applies `factor` as a CSS `transform: scale()` on `.canvas` (clamped `[0.5, 2.5]`), correcting `scrollLeft`/`scrollTop` in a `useLayoutEffect` keyed on `scale` so the cursor's content point stays put; a "Reset Zoom" button (top-right, so it doesn't collide with `.roomInspectorOverlay` in the bottom-right) appears whenever `scale !== 1`. `WorldScreen` instead resizes/repositions the SVG `viewBox` itself (`viewBoxOverride: {x,y,w,h} | null`, `null` = today's auto-fit-everything), converting the client-space focal point to SVG user-space via `svgRef.current.getScreenCTM()!.inverse()` (correctly accounts for `preserveAspectRatio` letterboxing) and clamping width between ~4 hexes wide and 1.5x the natural full-fit size -- a matching "Reset View" button clears the override. Since #33 made every known hex clickable (not just neighbors), zooming needed click-drag panning too (there's no native scroll on an inline SVG the way `DungeonMap`'s `.scroll` div gets) -- it reuses `DungeonMap`'s exact `didDrag`/pointer-capture pattern (a moved-past-threshold pointer-down is a pan, not a click; a capturing click handler swallows the resulting click) so panning and hex-selection coexist without either one accidentally triggering the other. One gotcha specific to `WorldScreen`: its very first render shows `TownScreen` (the `<svg>` doesn't exist yet, so `svgRef.current` is `null` when `useZoomGesture`'s mount effect first runs), and only mounts the map later in the *same* component instance once `showMap` flips true -- a `RefObject`'s identity never changes, so a `[ref]`-only effect dependency would attach its listeners exactly once, while `ref.current` is still `null`, and never notice the DOM node arriving later. `useZoomGesture` fixes this generally (both consumers benefit) by re-running its attach effect on every render instead of gating on `[ref]`.

A hex-tied dungeon is resumable **only** by physically traveling back to its hex -- `TownScreen` has no dungeon-finding beyond whatever `WorldScreen` hands it for the *current* hex (see above), so a fresh character can't jump straight into a predecessor's hex-found dungeon from some other city; they have to actually go find the hex again, the same as they'd have to the first time. This applies without exception, including to the current character's own still-active paused run: an earlier increment added a character-scoped "Continue the Dungeon" shortcut (`activeDungeonElsewhere`/`onContinueActive`) that warped `world.player` to the run's hex from anywhere, directly contradicting this rule -- removed (#44) rather than kept, since the rule itself is the intended design, not the shortcut.

**Still stubbed / not yet built** (Hexploring the World): Events on Travel (the 2d6 wilderness monster/weather table for non-location hexes); City/Fortress culture-specific extra actions per race-of-location (Human curse removal, Dwarf Lamp, Elf Boots, Gnome spell, Goblin potion, Orc Gladio) and the race Affinity table restricting entry; Portals (a 3d6 destination table) and the other inert `LocationKind`s' entry-roll effects (Reef's aground/Underwater Cave check, Oasis's mirage/heal check, Volcano's Volcanic Cave, Thin Ice's death check); boats (crossing water); Getting Money's side mechanics (Hard Work/Gamble/Thug Life/Arena).

**`Die` component** (`src/ui/components/Die/Die.tsx`) is a real 3D CSS cube (six absolutely-positioned faces inside a `perspective` container, `rotateX`/`rotateY` per face), not an image or icon-swap. It's driven by a `value` (1-6) + `rollToken` (bump this to trigger a new roll animation even if the value repeats) + optional `delayMs` (for staggering dice in a pool) + `size` prop. The rotation math (`SHOW_ROTATION` table + `angleTo` helper) picks the shortest forward rotation to the target face and adds a couple of extra full turns for a tumble effect — see the component for the derivation. `DicePool` just renders N `Die`s with staggered delays for 2d6/etc. rolls. Screens own the *timing* of when to reveal a roll's result (see `revealDelay()` in `CharacterCreationScreen.tsx`) — the Die component itself doesn't expose a completion callback.

**Theme** (`src/ui/theme/`): `tokens.css` (CSS custom properties — the ink/parchment/gold palette, font stacks), `fonts.css` (`@font-face` for the three self-hosted fonts in `fonts/`: Metamorphous for display, Spectral for body, JetBrains Mono for stats/numbers — all OFL, embedded locally, no CDN), `global.css` (resets + the torchlit-dungeon body background, imported once in `main.tsx`). Component-level styling uses CSS Modules (`*.module.css`); shared tokens/fonts are global.

**Versioning**: every push to `main` requires a version bump — `package.json`/`package-lock.json`'s `version`, a new dated section in `CHANGELOG.md`, and a matching `vX.Y.Z` git tag on the commit that lands it. Versions are `Major.Minor.Hotfix`: Hotfix for a bug fix with no user-visible feature change (e.g. 1.0.0 → 1.0.1), Minor for a new feature, Major for a significant milestone (e.g. the Core Book's full rule set landing as 1.0.0) or a breaking change. This applies even to a small fix pushed directly to `main` outside a normal feature-branch release — see the 1.0.1 tag for a hotfix-only example.

**Spellcasting** (`DungeonState.spellUses`, a `Record<spellRoll, remainingUses>` seeded at dungeon creation by `computeSpellUses()` from the character's `spells`/`fixedGrants`) covers all six Basic Spells via the `CAST_SPELL` action. Heal and Light are usable anytime (in `CombatPanel` or, out of combat, as a "Cast" button on `CharacterSheet`); Cold Ray, Lightning, and Fireball require an active fight and are rendered as extra per-monster or room-wide buttons in `CombatPanel`. Spell damage runs through `resolveSpellDamage()` in `combat.ts`, which shares Stoneskin/Intangible filtering with `resolvePlayerAttack()` but — since it's a fixed value, not a weapon "damage roll" — never triggers the roll-of-1/roll-of-6 abilities (Explosive, Firebreath, Weakness, etc.). Casting any spell mid-fight consumes the round exactly like `PLAYER_ATTACK` (the monsters counter-attack afterward), except Cold Ray also flags its target `skipNextAttack` so it sits out that round, and Teleport ends combat immediately with no counter-attack. A spell's uses, once spent, stay spent within a dungeon run — recovering them requires the Town's Rest action (see Town below).

**Teleport** ("You teleport to any empty room") is the one spell whose `CAST_SPELL` action carries extra fields (`destLevel`/`destSegId`) rather than just `spellRoll` — clicking the Combat panel's "Flee — Teleport" button (`CombatPanel`'s `onFlee` prop, kept distinct from `onCastSpell` since this spell alone needs a destination first) doesn't dispatch immediately; it swaps the panel for `TeleportPicker` in the same overlay slot, listing every already-discovered "empty" room across all levels (`isTeleportDestination()` in `dungeon.ts` -- a room-type segment, excluding the one currently in combat, with no undefeated/`sneakedPast` monsters; corridors and staircases don't count as a "room"). Only picking a listed room dispatches `CAST_SPELL` with that destination (Cancel dispatches nothing, spending no use); the reducer re-validates the destination itself before applying it, same as every other action. `draft.activeLevel`/`currentSegId`/`selectedSegId` all move to the chosen room -- unlike a death or a Town retreat mid-fight, the fled room's own `CombatState` doesn't persist anywhere, so `rerollMonstersIfNeeded()`'s "the fight you fled from is still waiting" branch (see Monster table re-roll below) is what resumes it, full HP, whenever the player later selects that segment again themselves.

**The Boss fight**: each dungeon type's `DUNGEON_TABLES[key].boss` (a `Record<1-6, MonsterTemplate>`, authored from the rulebook's per-dungeon "Table: Boss") is rolled once via `resolveBoss()` in `dungeon.ts` the moment the Final Room segment is built — both in the `descend-final` case (auto-triggered at depth 3) and `dead-end-final` (no stairs ever found). Per the rulebook, there's no Content/Monsters roll alongside it, only the Boss. The rolled `MonsterTemplate` is attached to the Final Room's existing `SegmentState.monsters` field and fed through the same `startCombatIfMonsters()` plumbing as an ordinary room, just with a new `isBoss` flag threaded onto `CombatState` — `finishIfVictorious()` checks that flag to grant a flat 2d6 Treasures instead of the normal per-monster Loot table (Treasures are flavor-only, like ordinary Loot, since there's no inventory system). `DungeonScreen.tsx` shows a gold victory panel once the Final Room's boss is marked defeated.

**Chests and Loot**: `RoomContentEntry.hasChest` flags the Room Content rows that describe an actual Chest (vs. passive flavor text like "1d6 coins on the floor," which isn't mechanically applied — see stub list below); a chest can also turn up via `secretPassageResult === "You have found a hidden Chest!"`. `RoomInspector` offers a free "Open Chest" action (no torch cost, per the rulebook) once one is available and unopened, rolling two independent dice — the higher becomes coins, the lower becomes Treasures, unless both come up 1, in which case the chest was empty and rolls on the dungeon's Trap table instead. `DungeonState.treasures`/`keys` are real counters (credited by both Chests and the Loot monster ability's `rollLoot()`) shown on `CharacterSheet` alongside coins — Keys don't do anything yet (no key-unlocks-any-door shortcut), but Treasures are openable (next).

**Secret Passage → Staircase**: a roll of 6 on the Secret Passage table ("A secret door to a Staircase") isn't just flavor text — `buildSecretPassageStaircase()` in `dungeonReducer.ts` builds a real, descendable `type: "staircase"` segment off the room (1 door, "the door in the end," exactly like an ordinary door resolving to the Segments table's own "staircase" outcome), except the door itself is brand new (appended to `seg.doors`, not one of the room's already-rolled doors) and already `opened: true`, since finding it via the search *is* the reveal. Placed in whichever cardinal direction the room doesn't already have a door facing — a non-entrance room always has at least one free direction (`assignDirections` caps a non-entrance room's own door count at 3 of 4), so this only silently no-ops (the flavor text alone still stands) for a fully 4-doored entrance, the one segment type that can actually use all four.

**The Combat panel's dice roll** renders directly under the player's HP row, before the monster list — not at the very bottom of the panel, which used to get pushed out of view once a fight had several monsters.

**Opening a Treasure**: each dungeon type's `DUNGEON_TABLES[key].treasure` (a `Record<1-6, RewardOutcome>`) models the rulebook's per-dungeon "Table: Reward" -- rows 1-4 give a Treasure-column item directly; rows that instead say "[Roll in the Wonders/Magic Item/Weapon column]" (Palace's rows 5-6; Prison additionally redirects row 4 to the Weapon table) are `RewardEffect.kind: "rerollColumn"`, resolved by an immediate second `rollDie(rng)` into that column's own table -- see Armor & Weapons for what Wonders/Magic Item rolls actually produce. Rows repeated near-verbatim across dungeon types (Health Potion, Magic Scroll, Valuable jewel) are shared constants (`HEALTH_POTION`, `MAGIC_SCROLL`, etc. in `dungeonTables.ts`), mirroring the existing `BLADE_TRAP`/`CLICK_NOTHING` pattern for traps. The `OPEN_TREASURE` action is usable anytime (not tied to a room, unlike Chests) via a die-roll bar rendered in `DungeonScreen` whenever `treasures > 0`; mid-combat it consumes the round exactly like `CAST_SPELL` (paralysis included, and now also gated on `combat.pendingDamage` being resolved first), since an unattended Health-Potion-on-demand would otherwise trivialize fights. `RewardEffect.kind: "restoreAllSpells"` (Tomb's Mana Potion) needs each spell's *max* uses, which `DungeonState` doesn't otherwise track — the action carries a `maxSpellUses` computed client-side via `computeSpellUses()` (the same helper `CharacterSheet` already uses) rather than adding a new persisted field for it. The rulebook's "worth N Coins in the town" outcomes (Ornament, Religious Object, Sinister Idol, Valuable jewel) don't convert to coins on the spot — there's no town to sell them in yet — so `RewardEffect`'s `"heldValue"`/`"heldValueRoll"` kinds instead push a `HeldItem` (`{ name, worth }`) onto `DungeonState.heldItems`, shown in the `Pack` sidebar card (`src/ui/components/Pack/`, positioned between `CharacterSheet` and the Ledger card in the dungeon, and alongside `CharacterSheet` in Town where it can actually be sold — see Town below).

**Armor & Weapons**: `docs/game-rules-reference.md`'s "Table: Armor" (1d6: Ring 0 HP / Bracelets 2 / Boots 3 / Shoulderpads 3 / Helm 4 / Breastplate 10) is identical across all six dungeon types, so it's one shared `ARMOR_TABLE` constant in `dungeonTables.ts`; "Table: Weapon" differs per type, so each `DUNGEON_TABLES[key]` carries its own `weapon: Record<1-6, WeaponEntry>`. A dungeon type's `wonders`/`magicItem` tables (`Record<1-6, WonderEntry | MagicItemEntry>`) model the Reward table's other two columns; all six dungeon types are fully authored. Every Wonder/Magic Item's unique ability is encoded as an `ItemEffect` (`extraHp`/`weaponDamageBonus`/`damageBonusVsTag`/`damageMultiplierVsTag`/`ignoresMonsterAbility`/`trapImmunity`/`doubleChestCoins`/`combatDamageBonus`/`grantsTorches`/`randomSpell`/`lifesteal`/`instantKillOnRoll`/`flavor`) -- a small, reusable vocabulary rather than one bespoke code path per item; `tags` matching (e.g. "+2 damage to Angels", or Crypt's Garlic necklace naming both Vampire *and* Ghoul) is a case-insensitive substring check against `MonsterTemplate.name`, since the rulebook itself has no formal monster-category system. `MagicItemEntry.fixedFormula` overrides the usual "roll the base Weapon table, then layer a bonus on top" shape for uniquely-named weapons that specify their own damage directly (Crypt's Boatman's Oar, 1d6+1). `ignoresMonsterAbility` covers two different shapes the rulebook conflates under one phrase: abilities a *monster* uses against the player (Undead revival, Paralyze, Deathtouch, Firebreath -- checked via `dungeonReducer.ts`'s `ignoresAbility()`) and the player's own attack being defensively blocked (Stoneskin/Intangible's damage-parity filter in `combat.ts`'s `applyDefensiveAbilities()`, e.g. Boatman's Oar bypassing Intangible) -- the latter is threaded into `resolvePlayerAttack()` as an `ignoreAbilities` array (`dungeonReducer.ts`'s `ignoredAbilities()` collects it from `equippedEffects()`), since `combat.ts` has no notion of the player's inventory on its own. `trapImmunity` is enforced by `dungeonReducer.ts`'s `resolveTrapOutcome()`, the single entry point all three trap-firing sites (`RESOLVE_DOOR_LOCK`, `ROLL_SECRET_PASSAGE`, `ROLL_CHEST`) funnel through: it blocks the *whole* trap -- torch cost included -- and consumes the item (one-shot, matching "ignores the next activated trap," not a standing immunity). `WeaponEntry.twoHanded` remains tracked but unenforced -- audited and confirmed there's currently no off-hand/shield slot (`ArmorPieceKind` is all body slots) it could conflict with, so there's nothing to block without first building the hand-economy/limb-tracking system the rulebook's "Your Hands" rule would need.
- **Acquiring items** (`resolveWonder()`/`resolveMagicItem()` in `dungeonReducer.ts`, called from `OPEN_TREASURE`'s `rerollColumn` case): a Wonder either grants its own HP-bearing item (Jester Hat, 2 HP) or, if it has no HP of its own (Amulet of the Dead), still becomes a `DungeonState.armor` entry at `hp: 0, maxHp: 0` -- never offered as a damage-absorption choice (see below), but still equipped/visible and checked by whichever system its `effect` concerns -- except `combatDamageBonus` (Potion of Fury), which instead adds directly to `combat.playerDamageBonus` if a fight is active, or is simply wasted if not. A Magic Item is always "[Armor] of X" or "[Weapon] of X" (`MagicItemEntry.grants`): resolving one rolls the *base* Armor or Weapon table for the concrete piece/formula, then layers the effect on top -- `extraHp` is baked directly into that new piece's `hp`/`maxHp` (Centurion's [Armor] +1 HP rolls, say, Boots at 3 HP and pushes a 4/4 HP piece), anything else becomes the piece's `effect` (or the weapon's `bonusEffect`); acquiring a new weapon never overwrites `DungeonState.weapon` directly (see "A spare weapons inventory" below) -- every weapon-granting site instead pushes onto `spareWeapons`, matching the precedent `COLLECT_REMAINS` sets for picking up a dead adventurer's gear (also a push, not an overwrite).
- **Wearing them**: `DungeonState.armor: ArmorPiece[]` / `weapon: EquippedWeapon | null` (mirrored on `AdventurerResources` in `town.ts` and `FallenAdventurer` for remains) flow through every place a dungeon gets (re)constructed exactly like `heldItems` already did -- `createInitialDungeonState()`, `RETURN_TO_DUNGEON` (carries them over exactly), and `App.tsx`'s retreat/return round-trip; `RESUME_DUNGEON` deliberately does **not** carry them (a new character doesn't inherit a dead one's gear -- it becomes `remains`, same as coins/heldItems). `Equipment` (`src/ui/components/Equipment/`) renders both, read-only in `DungeonScreen`'s sidebar and with a per-piece "Fix (1 coin)" button in `TownScreen` (wired to `town.ts`'s `canFixArmor()`/`fixArmor()`, the same pure-function shape as `rest()`/`buyTorch()`).
- **A spare weapons inventory** (issue #48): unlike Armor's 5 independent, additive slots, a weapon used to be a single override -- finding a new one silently discarded whatever was equipped, with no salvage and no choice. `DungeonState.spareWeapons: EquippedWeapon[]` (mirrored on `AdventurerResources` and, as plural `weapons`, on `FallenAdventurer`, threaded through every construction site exactly like `armor` already is -- `RESUME_DUNGEON` resets it to `[]`, `RETURN_TO_DUNGEON` carries it over exactly) fixes this: every weapon grant -- both `resolveMagicItem()` branches (called from `applyRoomContentReward()`, an *automatic* grant the instant a room is built with no player action at all, as well as from `OPEN_TREASURE`'s `rerollColumn` -> `magicItem` branch), `OPEN_TREASURE`'s direct `rerollColumn` -> `weapon` branch, and `COLLECT_REMAINS` -- pushes onto `spareWeapons` instead of overwriting `weapon`. The automatic grant site is exactly why the fix isn't an interrupting choice prompt (mirroring `RoomEntryPrompt`'s pattern was considered and rejected): there's no player action to interrupt when a weapon can appear just from opening a door. A weapon found this way is never auto-equipped, full stop, even the very first one -- a deliberate behavior change matching the issue's "give the player an actual choice" literally. `WIELD_WEAPON { index }` (`dungeonReducer.ts`, gated exactly like `COLLECT_REMAINS`: alive, not in combat, no pending room entry -- free, out-of-combat only, no mid-fight swap semantics invented) swaps a chosen spare into `weapon`, pushing whatever was equipped (if anything) back into `spareWeapons`; `town.ts`'s `wieldWeapon()` is the same swap as a pure resources-transform for Town, which has no `DungeonState`/reducer to dispatch against (same split `fixArmor()`/`sellItem()` already establish). `Equipment` renders a "Spare Weapons" list (name/formula/Two-handed, each with a "Wield" button via a new optional `onWield` prop mirroring `onFixArmor`) between the equipped weapon and the armor list, in **both** `DungeonScreen`'s sidebar and `TownScreen` (unlike `onFixArmor`, Town-only) since wielding is meant to be usable mid-dungeon between fights, not just back in town; the row is a two-line layout (name+button on top, formula below) rather than `armor`'s single-line grid, since a long weapon formula plus "· Two-handed" reliably overflowed a single grid row past the sidebar's fixed width. Selling a spare weapon is out of scope -- no `worth`/price data exists for weapons anywhere in this codebase, unlike `HeldItem`.
- **Taking damage with armor equipped**: "reduce this value from your HP (or armor's HP, if you're using one — your call)" is a genuine player choice, not automatic, so `applyMonsterTurn()` can't just subtract HP synchronously the way it used to. It splits a round's damage into `poisonDamage` (Poison "cannot be absorbed by armor or other means" — always hits `hp` immediately, and first, so a fatal poison tick can't leave a moot choice pending for the rest) and `absorbableDamage` (everyone else's); if the player has at least one armor piece with `hp > 0` (`hasUsableArmor()`), `absorbableDamage` is stored on `CombatState.pendingDamage` instead of applied, and the round stops there. `CombatPanel` renders an absorption chooser whenever `pendingDamage !== null` (same visual slot the paralyzed-turn message uses), listing HP and every piece with `hp > 0`; picking one dispatches `RESOLVE_DAMAGE`, which subtracts from the chosen pool (overflow past a piece's remaining HP spills onto the player's `hp`, destroying the piece at 0) and re-runs the death check. `PLAYER_ATTACK`/`CAST_SPELL`/`OPEN_TREASURE` all no-op while a choice is pending, the same way they already do during paralysis.
- **Fighting with a found weapon**: `attackBonus()`/`attackMultiplier()` in `dungeonReducer.ts` read `DungeonState.weapon?.bonusEffect` (plus `combat.playerDamageBonus`) when `PLAYER_ATTACK` computes `weaponTotal` -- the multiplier applies to just the weapon's own roll, flat bonuses are added after, since only one bonus effect is ever active on a given equipped weapon at a time (this app doesn't model combining two).

**Still stubbed / not yet built**: `WeaponEntry.twoHanded`, tracked but unenforced (see Armor & Weapons) -- there's no off-hand/shield slot in this armor system yet for a two-handed weapon to actually conflict with.

**Move Silently**: a quiet arrival (an unlocked door, a picked lock, a fresh staircase descent) into a room with monsters no longer auto-starts combat -- `finishRoomSegment()` only does that for a noisy one now. Instead it waits for a `RESOLVE_ROOM_ENTRY` action: **Attack First** (free, same outcome this app always gave before this existed) or **Move Silently** (spend 1 torch; roll one die per monster -- resolved inside the reducer via its own `rng`, since a room's count can itself be a dice roll like "1d6 Goblins" and isn't knowable client-side in advance -- any 1 means detected and the monsters attack first, otherwise the room is marked `sneakedPast` and combat never starts at all). `hasPendingRoomEntry()` blocks every other dungeon action the same way an active `CombatState` does, until that choice is made. A `sneakedPast` room's monsters wake up (combat starts, monsters first) if a later noisy action -- breaking a door, a fired trap -- happens in that same segment (`wakeSneakedPastMonsters()`); `rerollMonstersIfNeeded()` and `restoreMapFromPersisted()` both treat a sneaked-past room as still-occupied (no fresh reroll, no auto-resume from just reselecting it). Halfling's "roll two dice and discard the lowest (except in the Boss)" advantage is applied server-side in the same `RESOLVE_ROOM_ENTRY` case; Boss rooms never reach this at all, since `descend-final`/`dead-end-final` call `startCombatIfMonsters()` directly. `RoomEntryPrompt` (`src/ui/components/RoomEntryPrompt/`) is the UI for the choice, rendered in place of `RoomInspector` while pending. Both `DungeonScreen.tsx`'s and `DungeonMap.tsx`'s own client-side `pendingRoomEntry` (mirroring the reducer's `hasPendingRoomEntry()` for render/interaction gating) also check `state.alive` directly -- dying to that same room's monsters never marks `monstersDefeated`, so without this the prompt would otherwise reappear right alongside the death panel (the reducer's own `RESOLVE_ROOM_ENTRY` handler already no-ops once dead, so this was purely a decorative render bug, the same class of gap #38 fixed for the Combat panel).

**Race and class abilities**: `DungeonState.raceName`/`className` (plain strings, matched by exact `RaceDef.name`/`ClassDef.name` -- the same "no formal taxonomy" precedent Armor & Weapons' `matchesTags` already established for monster names) thread through every place a dungeon gets (re)constructed, exactly like `characterName`, so `dungeonReducer.ts` can gate ability checks without needing the full `CreatedCharacter` object. Abilities resolvable entirely client-side instead just read `character.race`/`character.cls` directly, since the owning component (`TownScreen`, `RoomInspector`) already has one or the other in scope.
- **Combat** (`dungeonReducer.ts`): Grave Digger's +2 damage checks `monster.abilities.includes("undead")` in `attackBonus()`, not a name/tag match (Undead-ness is a monster *ability*, not something every undead monster's name announces). Cook's +1 coin (except Undead) and Slimemen's `CombatState.engulfableBodies` tracking both live in `handleMonsterDefeat()`'s non-revived-kill branch -- engulfable bodies accrue for every race (harmless if never consumed), but only Slimemen ever sees the `ENGULF_BODY` action/button (`CombatPanel`'s "Engulf Body" -- consumes one body, heals to `maxHp`, then the round continues exactly like `CAST_SPELL`, monster counter-attack included). Rinoceroid's horn attack is a second per-monster button in `CombatPanel` alongside the normal Attack, dispatching `PLAYER_ATTACK` with `useHorn: true`; the reducer computes a flat `1d6` (no modifier) and skips every equipped-weapon effect (`attackBonus`/`attackMultiplier`'s new `isHorn` parameter, `instantKillOnRoll`, lifesteal) since the horn isn't the equipped weapon -- general combat buffs (Potion of Fury, Grave Digger) still apply.
- **Doors/torches** (`dungeonReducer.ts`): Locksmith's free lock-picking and Lumberjack's chance to gain a torch from breaking a door both live in `RESOLVE_DOOR_LOCK`'s existing `pickLock`/`breakDoor` branches. Miner's "if you run out of torches, you can leave the dungeon" is a `spendTorches()` special case -- instead of the Darkness killing the character, the failed action just... fails, leaving them alive with 0 torches to use the already-existing Retreat to Town button, rather than building a new "forced exit" flow for what's really just "don't die here."
- **Town** (`town.ts`): Blacksmith's `fixArmor()`/`canFixArmor()` take an `isBlacksmith` flag that swaps the cost from 1 coin to 1 torch; Cat-Person's `sellItem()` takes an `isCatPerson` flag that doubles the credited coins. Both are optional params (default `false`) so every other caller/test is unaffected.
- **Dwarf** (`RoomInspector.tsx`, entirely client-side): Find Secret Passage rolls two dice instead of one when `state.raceName === "Dwarf"` and keeps the higher (row 1 is the table's only bad outcome, so discarding the lower die is strictly an advantage) -- reuses `DicePool` (already used for Chests) instead of the single-die `Die` component.

**Monster table re-roll on return**: per the rulebook, resting in Town (or a new character taking over via `RESUME_DUNGEON`) means fresh monsters may have moved into any room that's currently empty -- "you must roll on the Monster table for each empty room you enter," and explicitly "also true if you die and venture out with another character in the same dungeon." `restoreMapFromPersisted()` (shared by both actions, see above) flags every already-built room-type segment that's currently empty or already-cleared (`SegmentState.needsMonsterReroll`) except whichever segment just had its interrupted fight respawned at full HP. The reroll itself is resolved lazily, by `SELECT_SEGMENT` (via the new `rerollMonstersIfNeeded()` helper), the next time the player actually looks at that room -- not eagerly for the whole map at once, since this app only has one `DungeonState.combat` slot at a time and eagerly rolling every empty room could produce several newly-occupied rooms with no way to fight more than one of them. `restoreMapFromPersisted()` also resolves it immediately for whichever segment the player ends up standing in right after the restore -- their own level's entry segment for both `RESUME_DUNGEON` and `RETURN_TO_DUNGEON` (see **Positional movement** below), so returning doesn't leave the player's own starting room silently unrolled. Only `monsters`/`monstersDefeated` are touched -- `roomContent`, chests, and secret-passage state are left exactly as found, since the rulebook penalty is specifically about monsters repopulating, not the room's contents resetting.

**Positional movement**: `DungeonState.currentSegId` is the segment the player physically occupies -- distinct from `selectedSegId` (what `RoomInspector` displays), though `SELECT_SEGMENT` always keeps the two in lockstep: selecting anything other than the segment you're already in only succeeds if it's *reachable*, and succeeding moves you there (also resolving that segment's pending Monster re-roll, see above). `reachableSegIds()` (`dungeon.ts`) computes the fog-of-war boundary for a level: `currentSegId` itself plus every segment directly connected to it by an already-opened door, walking each segment's doors once to build the (otherwise parent → child only, via `childId`) link in both directions. Segments outside this set render greyed out on `DungeonMap` and can't be selected, and their unopened doors can't be clicked either -- `OPEN_DOOR`/`RESOLVE_DOOR_LOCK` both guard on `action.segId === state.currentSegId` server-side (the UI already disables the buttons, but the reducer is the actual authority), so a new door can only ever be pushed open from wherever the player stands. `currentSegId` auto-advances the moment a door opens into a newly-built segment (all 6 `OPEN_DOOR` classification branches set it alongside `door.childId`) -- opening a door *is* walking through it. Stepping back through an *already-opened* staircase (`DungeonMap`'s descend button, as opposed to `LevelTabs`, which only changes which level's map is *displayed*) also moves the player: `SWITCH_LEVEL` takes an optional `segId` used only by that button, validated the same way as any other move (`reachableSegIds` happily returns a childId belonging to a different level here -- harmless, since cross-level doors are the *only* way that ever happens, and the reducer's own `OPEN_DOOR` branches are what populate them in the first place). A level other than wherever `currentSegId` actually is ends up with an empty reachable set purely as an emergent property of this -- none of its segment ids will ever equal `currentSegId` or be one of its neighbors -- so `LevelTabs` doubles as free, read-only history browsing for any level the player has already moved past, with no separate "is this level still enterable" flag to maintain. `restoreMapFromPersisted()` resets `currentSegId` (and `selectedSegId`) to `activeLevel`'s own entry segment (`segments[0]`) for *both* `RESUME_DUNGEON` and `RETURN_TO_DUNGEON` -- not all the way back to Level 1 for a multi-level run (the level transition itself is already a one-way, already-completed step) but still requiring a walk back through whichever level the player is currently on, so #18's Monster re-roll can't be silently bypassed by landing directly on a stale, freshly-repopulated room. An interrupted fight (see above) is the one exception -- it still resumes exactly where it was, full HP restored, rather than requiring a walk back into it.

**Room Content rewards**: `RoomContentEntry.reward` (a `RoomContentReward` discriminated union: `coins`/`treasures`/`magicScrolls`/`magicItems`, each carrying a `MonsterCount` -- reusing the same type and `resolveMonsterCount()` helper the Monster tables already use for variable group sizes) flags the Room Content rows across all six dungeon types that describe an automatic, mechanically-applied reward rather than passive flavor text. `applyRoomContentReward()` in `dungeonReducer.ts` resolves it the moment the room is built: `coins` credits `DungeonState.coins` (times an optional `multiplier`, for rows like Sanctuary's "2d6 paintings, 2 coins each"), `treasures` credits `DungeonState.treasures`, `magicScrolls` grants that many random Basic Spell uses (one `SPELL_TABLE` roll per scroll, each adding a use via the same bump `computeSpellUses()`'s callers already expect), and `magicItems` rolls that many Magic Items via the existing `resolveMagicItem()` (see Armor & Weapons) -- reusing it here is why a `magicItems` reward consumes two `rollDie()` calls per item, not one (the Magic Item table roll, then that item's own base Armor/Weapon table roll). `finishRoomSegment()` (also new, in `dungeonReducer.ts`) composes this with the existing `startCombatIfMonsters()` call at all 4 real room-building call sites (entrance, `reuse-normal`, `descend-normal`, ordinary `normal` doors) -- reward first, then combat -- so a room can hand out its automatic loot even if it also turns out to hold monsters. The 2 Final-Room/Boss call sites are untouched, since Final Rooms never populate `roomContent`.
