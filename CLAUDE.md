# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser-based digital adaptation of *NoteQuest* (by Tiago Junges), a solo, dice-driven, pen-and-paper dungeon-crawler board game. Target deployment is GitHub Pages (static hosting, no backend).

`docs/game-rules-reference.md` is the authoritative transcription of the rulebook (Core Book + Expanded World) — treat it as the source of truth for any table, mechanic, or dice logic. Don't invent rules not present there. The purchased rulebook PDF itself is intentionally not tracked in git (see `.gitignore`) since this repo is public.

`design/mockups/*.html` are standalone, self-contained (fonts embedded as base64) interactive layout studies used to validate screen designs before they're built as real components. They are not part of the app build, and are left as-is once ported — don't edit them expecting it to affect the app. Both current mockups (`character-creation.html`, `dungeon-building.html`) have been ported to real code; treat the mockups as historical design reference, not spec-of-record (the real code has since diverged in places, e.g. the dungeon screen's sidebar).

**Scope discipline**: v1 targets the Core Book rules only (character creation, procedural dungeon crawl, combat, torches/darkness, death/graveyard). Expanded World content (hexcrawl, cities, politics, warfare, multiplayer, advanced classes) is an explicit future phase — don't build toward it prematurely.

## Architecture

**Data-driven engine, not per-screen hardcoding.** Game rules live as typed data in `src/data/` (`races.ts`, `classes.ts`, `spells.ts`, `dungeonTypes.ts` — Segments/Secret Passage/Dungeon Name tables, shared across all 6 dungeon types — and `dungeonTables.ts` — the per-type Trap/Room Content/Monsters tables), keyed by dice-roll value, e.g. `RACE_TABLE[7]` for a 2d6 roll of 7. Pure game logic lives in `src/engine/` and takes an injectable `RNG = () => number` (defaulting to `Math.random`) so tests can pass a seeded generator (`src/test/mulberry32.ts`) for deterministic assertions instead of asserting on ranges. UI components consume `src/engine` functions and never re-implement roll logic themselves.

**The dungeon engine** (`src/engine/dungeon.ts`, `dungeonState.ts`, `dungeonReducer.ts`) is the more involved half. A dungeon is a tree of `levels[]`, each holding `segments[]` (rooms/corridors/staircases, positioned in an abstract pixel coordinate space for the map) and `connectors[]`. `dungeon.ts` holds two kinds of pure logic that are easy to conflate but are kept separate in the file: (1) *rules* — table rolls, `classifyDoorOpen()` (whether opening a given door is an ordinary roll, a staircase descent, the automatic depth-3 Final Room, the "no stairs ever found" dead-end Final Room, or a second staircase reusing an already-discovered level/Final Room); (2) *spatial layout* — `placeChild`/`placeIslandRoot`'s collision-avoiding placement algorithm, which is **not** a NoteQuest rule, just this project's own way of drawing a coherent map (the physical game doesn't need coordinates). `dungeonReducer.ts` applies `classifyDoorOpen()`'s result via Immer (`produce`) — deeply-nested state (`levels[i].segments[j].doors[k]`) makes manual immutable spreads unwieldy, hence the dependency. The reducer's extra `rng` parameter (for the silent Room Content/Monsters rolls) means it can't be passed directly to React's `useReducer` — `DungeonScreen.tsx` wraps it in a 2-arg `reduceDungeon` first.

**Torches and doors**: `DungeonState.torches`/`alive` track the resource that actually matters in this game — every door first rolls the shared `OPEN_DOOR_TABLE` (trap/locked/unlocked) via `RESOLVE_DOOR_LOCK`, *before* `OPEN_DOOR` ever runs; only "unlocked" (or a resolved lock/trap) proceeds to the existing Segments-table roll. Torch-spending (entering the dungeon, picking a lock, searching a secret passage, the ditch trap's `torchCost`) all flow through the reducer's `spendTorches()` helper, which flips `alive` to `false` (the Darkness) instead of going negative. `DungeonMap.tsx`'s door-click handler mirrors the reducer's math on the *client* side (checking `state.torches` before offering "Pick Lock", before animating a segment roll after a trap) purely so it knows whether to bother animating further dice or stop — the reducer is still the sole authority on whether the character actually dies. Locked doors show a "Pick Lock (1 torch) / Break Door (free)" popover; breaking a door or setting off a trap alerts the monsters beyond it (see Combat below).

**Combat** (`src/engine/combat.ts`) is pure, RNG-injected logic exactly like the rest of the engine; `dungeonReducer.ts` is the only thing that mutates `DungeonState.combat`/`hp`/`coins`. Whenever `buildSegment()` creates a new room segment (dungeon entrance, an ordinary door, a fresh staircase-descent level, or a second-staircase reuse) and that room's `resolveRoomExtras()` roll produced a `MonsterTemplate`, `startCombatIfMonsters()` spawns a `CombatState` via `spawnMonsters()`. Whether the player or the monsters get the first attack is decided by `wasNoisy` (true only when a door was broken open or a trap fired — see the "Combat" section of `docs/game-rules-reference.md`); a noisy start immediately runs one `applyMonsterTurn()` before the player can act. Once `state.combat` is non-null, `OPEN_DOOR`/`RESOLVE_DOOR_LOCK`/`ROLL_SECRET_PASSAGE`/`SWITCH_LEVEL` all no-op (the dungeon map itself also disables door buttons) — the only action is `PLAYER_ATTACK`, one full round per dispatch: `resolvePlayerAttack()` resolves the player's hit (Stoneskin/Intangible/Weakness, plus whichever roll-of-1 special ability fires — Explosive short-circuits into an immediate self-destruct that damages the player directly, bypassing normal damage math), the reducer applies the resulting events (Firebreath/Sorcery queue `bonusDamage`, Deathtouch/Paralyze queue onto the monster for its *own* next attack, Regeneration heals immediately, Horde/Necromancy always spawn a fixed `HORDE_ORC`/`NECROMANCY_SKELETON` regardless of whether the triggering monster died), then `checkUndeadRevival()` on any newly-defeated Undead monster before removing it (skipped entirely if the player holds an `ignoresMonsterAbility: "undead"` item — see Armor & Weapons). If any monsters remain, `applyMonsterTurn()` closes out the round: it splits the tally into `poisonDamage` (Poison "cannot be absorbed by armor or other means," so it always hits `hp` immediately, before anything else) and `absorbableDamage` (everyone else's `damage + bonusDamage`, deferred to `combat.pendingDamage` — see Armor & Weapons — if the player has usable armor, else applied to `hp` the same way poison was), then consumes queued Paralyze/Deathtouch and clears those queues — paralysis is tracked as `combat.paralyzedTurns` and simply skips the player's next N `PLAYER_ATTACK` dispatches (monsters still act). Victory (`finishIfVictorious()`) rolls `rollLoot()` once per Loot-tagged monster actually removed from the fight, credits `coins` (Treasure/Key results are logged as flavor only — no inventory yet), marks the segment `monstersDefeated`, and clears `combat`. Hitting 0 HP sets `alive = false` and `deathCause = "combat"` (as opposed to `"darkness"` from running out of torches), which `DungeonScreen.tsx` uses to pick the death message. `CombatPanel.tsx` is the primary UI surface for this — it rolls the player's weapon die (`state.weapon?.formula ?? character.cls.weaponDamage` — see Armor & Weapons for how a found weapon overrides the class one) and dispatches `PLAYER_ATTACK` with the raw roll; the reducer, not the component, applies the weapon's modifier and any bonus effects. When `combat.pendingDamage` is set, `CombatPanel` instead renders an absorption chooser (HP or a specific armor piece, each showing current/max) that dispatches `RESOLVE_DAMAGE`; `PLAYER_ATTACK`/`CAST_SPELL`/`OPEN_TREASURE` all no-op until it's resolved, same as the existing paralyzed-turn gate.

**Screens** live in `src/ui/screens/<ScreenName>/`, one file + one CSS Module each. `App.tsx` holds the only navigation state and switches between `CharacterCreationScreen`, `TownScreen`, and `DungeonScreen` — there's no router, and with only three screens that's a deliberate choice, not an oversight; revisit if a fourth needs it. Flow is `CharacterCreationScreen` → `TownScreen` → `DungeonScreen`, with `DungeonScreen` able to send the player back to `TownScreen` (a voluntary retreat, alive) or all the way back to `CharacterCreationScreen` (death, via `onNewAdventurer` — a dead character is permanently dead, there's no way to resurrect or keep playing one past `!state.alive`). See **Town** below for how resources move between screens. **Shared components** live in `src/ui/components/`: `Die`/`DicePool` (see below), `CharacterSheet` (renders a `CreatedCharacter` plus live stats, used in both the Town and dungeon sidebars), `Equipment` and `Pack` (see Armor & Weapons / Opening a Treasure; each takes an optional `onFixArmor`/`onSell` to add a per-row action button, used only in Town), `Graveyard` (see below), and the dungeon-specific `DungeonMap`/`LevelTabs`/`RoomInspector`/`RollLog`. `src/ui/rollTiming.ts` has the shared `revealDelay()` timing helper screens use to delay revealing a roll's result until its dice animation finishes.

**The Graveyard** (`src/engine/graveyard.ts`) is the one piece of state that survives a reload — everything else in this app is still fully in-memory. `loadGraveyard()`/`addGraveyardEntry()` read/write a JSON array under the `notequest:graveyard` localStorage key; both take an injectable `storage: Storage` parameter (mirroring the RNG-injection pattern elsewhere in `engine/`) so their tests run in Vitest's default Node environment, which has no `localStorage`. `DungeonScreen` records an entry in a `useEffect` keyed on `state.alive` flipping to `false` (so it fires exactly once per death, not on every re-render of the death panel), then `CharacterCreationScreen` loads the list once on mount (a fresh mount happens every time the player returns here, including right after a death) and renders it via the `Graveyard` component if it's non-empty.

**Town**: `TownScreen` sits between `CharacterCreationScreen` and `DungeonScreen`, modeling the rulebook's City Actions page, and is also where the player picks *which* dungeon to enter next (own paused run, an abandoned one, or a fresh roll — see Resuming below). `App.tsx` lifts an `AdventurerResources` object (`src/engine/town.ts` — `torches`/`hp`/`maxHp`/`coins`/`treasures`/`keys`/`heldItems`/`armor`/`weapon`/`spellUses`, i.e. everything a living character carries between dungeon and town) alongside `character`; it's seeded from the character's starting stats the moment `CharacterCreationScreen` calls `onCharacterCreated` (which always lands on Town, never straight into a dungeon), and overwritten wholesale from the dungeon's own fields whenever `DungeonScreen` calls `onReturnToTown(runId, dungeon)`. `town.ts`'s `rest()`/`buyTorch()`/`sellItem()`/`fixArmor()` are pure functions (no RNG needed -- all four are fixed-cost per the rulebook) that `TownScreen` calls and hands the result to `onUpdateResources`; `canRest()`/`canBuyTorch()`/`canFixArmor()` gate the buttons (1 coin; torches additionally capped at the rulebook's 10, a piece additionally must actually be damaged). Selling reuses `Pack`'s `onSell` prop, crediting the coins a `HeldItem` already carries as its `worth` (see Opening a Treasure) rather than re-rolling a price; fixing reuses `Equipment`'s `onFixArmor` prop the same way (see Armor & Weapons). `rest()` heals to `resources.maxHp` specifically (not a hardcoded/derived value) — this only works correctly because `maxHp` is threaded as its own field everywhere a dungeon gets (re)constructed (see the `maxHp` note below); before that fix, returning to a dungeon below full HP silently clamped the character's *true* max HP down to whatever they currently had, making Rest heal to a shrinking ceiling instead of the real max.

**Resuming an unbeaten dungeon**: per the rulebook, dying (or otherwise leaving) doesn't erase a dungeon -- a later character can pick up exactly where it was left, and returning to town to rest doesn't either. `App.tsx` lifts a `dungeonHistory: PendingDungeon[]` array (in-memory only, unlike the Graveyard) above the Town/Dungeon screens, keyed by a `runId` (`crypto.randomUUID()`) that a `DungeonScreen` instance mints once at mount and re-mints on `handleStartNewDungeon`. Progress is captured generically in an unmount cleanup (`useEffect(() => () => onLeaveDungeon(...), [])`, reading the *latest* state via a ref kept fresh by its own render-triggered effect, since writing a ref during render itself isn't allowed) — this fires no matter *why* the screen unmounts (death, a "Retreat to Town" click, "Start a New Dungeon", or beating the Final Room), so App's `handleLeaveDungeon` doesn't need to special-case the reason: it just drops the run if it never went anywhere (`levels.length === 0`), otherwise upserts it -- unlike `pendingDungeons` before it, **beaten runs are kept too**, not dropped, purely as a historical record (see Town's Dungeon History list below; `isDungeonBeaten()` is computed on demand wherever "is this one still resumable?" matters, rather than stored as a field). `App.tsx` separately tracks `activeRunId` -- the *current, still-living* character's own paused run, if any -- set whenever `onReturnToTown` reports a non-empty, unbeaten dungeon. `TownScreen` is the single place the player chooses what to enter next, via three actions (`onContinueActive`/`onResumeDungeon`/`onRollNew`) that just set App's `selectedRunId` and switch to the dungeon screen; App then derives, from whether `selectedRunId === activeRunId`, which of two very different resume paths `DungeonScreen`'s mount-time `useReducer` initializer takes (both land in the same `restoreMapFromPersisted()` reducer helper for the shared map-copying/monster-respawn logic, but differ in what happens to resources and where the player starts):
- **`RESUME_DUNGEON`** -- a *new* character taking over a *dead* one's map (Town's "Dungeon History" list, `dungeonHistory` prop -- every other touched run besides the current character's own `activeDungeon` -- see Town's Dungeon History below). Resets every character-specific field (`torches`/`hp`/`maxHp`/`weaponFormula`/`spellUses`/`coins`/`treasures`/`keys`/`heldItems`/`armor`/`weapon`) to the new character's own starting `resources`, logs "A new adventurer takes up the fallen's path.", and -- since `restoreMapFromPersisted()`'s `resetToEntrance` flag is `true` for this path -- forces `activeLevel`/`selectedSegId` back to the entrance segment regardless of where the previous character died deep in the dungeon: "he will find his backpack and clothes on the floor" implies discovering the remains by exploring, not teleporting to them. (The full multi-level map is untouched; only the initially-displayed position resets.) The dead character's own belongings stay behind as recoverable `remains` (see below) -- they are not inherited.
- **`RETURN_TO_DUNGEON`** -- the *same still-living* character coming back after a Town visit (Town's "Continue the Dungeon" button, only shown when `activeDungeon` is set). `DungeonScreen`'s `useReducer` initializer dispatches this automatically on mount whenever `activeDungeon` is set (skipping the gate entirely), carrying over `resources` *exactly* -- including `coins`/`treasures`/`keys`/`heldItems`, which `RESUME_DUNGEON` deliberately does not -- with `resetToEntrance: false`, so it picks up in the exact room/level it left off in. Logs "You return to the dungeon." instead.

Both actions `structuredClone()` the incoming persisted `DungeonState` before touching it, since Immer deep-freezes everything `produce()` returns and mutating a frozen array throws. Either way, if the previous session died or left mid-fight, that segment's monsters (there can be at most one such segment dungeon-wide, since every other room's combat had already resolved by the time further doors could open) are respawned at full HP via the same `startCombat()` helper ordinary room entry uses, matching "a room that still has monsters recovers their health." Both actions also carry an explicit `maxHp` field (as does the plain fresh-roll path, `DungeonScreen`'s `createInitialDungeonState(..., resources.maxHp)` call) -- `createInitialDungeonState()`'s `startingHp` param only ever set *current* HP historically, defaulting `maxHp` to match it, which was harmless when every entry point started at full health but broke the moment Town made "retreat below full HP, then come back" possible.

**Town's Dungeon History**: `TownScreen`'s `sortDungeonHistory()` renders `dungeonHistory` (every dungeon minus the player's own `activeDungeon`, if any) deliberately styled like the `Graveyard` component -- title, a count note, a dotted-divider row list -- since both are the same idea (a running record of what came before) applied to different subjects. Sort is most-recently-touched-first within two groups, unbeaten before beaten: `[...history].reverse().sort((a, b) => Number(isDungeonBeaten(a.dungeon)) - Number(isDungeonBeaten(b.dungeon)))` -- reversing first and relying on `Array.prototype.sort`'s stability (guaranteed since ES2019) preserves recency order inside each group without needing a separate timestamp field, mirroring `Graveyard`'s own bare `.reverse()`. Unbeaten rows render as a clickable button (`onResumeDungeon`, gold "Take up →" tag); beaten rows render as a plain, non-interactive row with a muted "Completed" tag -- there's nothing left to resume in a dungeon whose Final Room boss is already dead.

**Remains**: "all your equipment will be on the floor of that room to be recovered by your next character" (and the Darkness section says the same). `DungeonState.characterName` (seeded alongside the other character-derived fields at creation/resume) labels whose remains they are. `leaveRemains()` in `dungeonReducer.ts` fires at every death site -- the Darkness branch of `spendTorches()` (now threaded an optional `segId` from each of its six call sites, since the relevant room is whichever segment's action triggered the torch spend, not necessarily the currently-selected one) and both of `applyMonsterTurn()`'s kill branches plus `PLAYER_ATTACK`'s Explosive self-destruct branch (using `combat.segId`, since a combat death's location is unambiguous) -- and drops the dying character's current coins/Treasures/Keys/`heldItems`/`armor`/`weapon` (the Pack and Equipment contents -- see Opening a Treasure / Armor & Weapons) onto `SegmentState.remains`, merging (`names` array grows, counts add, `heldItems`/`armor` concatenate, `weapon` fills in only if nothing was left behind already) if an earlier death already left something there. `RoomInspector` shows a "Remains" row plus a free "Recover Remains" button (no roll, matching the rulebook not listing this as a torch-costing action) that dispatches `COLLECT_REMAINS`, crediting the current character and clearing the segment's `remains`.

**`Die` component** (`src/ui/components/Die/Die.tsx`) is a real 3D CSS cube (six absolutely-positioned faces inside a `perspective` container, `rotateX`/`rotateY` per face), not an image or icon-swap. It's driven by a `value` (1-6) + `rollToken` (bump this to trigger a new roll animation even if the value repeats) + optional `delayMs` (for staggering dice in a pool) + `size` prop. The rotation math (`SHOW_ROTATION` table + `angleTo` helper) picks the shortest forward rotation to the target face and adds a couple of extra full turns for a tumble effect — see the component for the derivation. `DicePool` just renders N `Die`s with staggered delays for 2d6/etc. rolls. Screens own the *timing* of when to reveal a roll's result (see `revealDelay()` in `CharacterCreationScreen.tsx`) — the Die component itself doesn't expose a completion callback.

**Theme** (`src/ui/theme/`): `tokens.css` (CSS custom properties — the ink/parchment/gold palette, font stacks), `fonts.css` (`@font-face` for the three self-hosted fonts in `fonts/`: Metamorphous for display, Spectral for body, JetBrains Mono for stats/numbers — all OFL, embedded locally, no CDN), `global.css` (resets + the torchlit-dungeon body background, imported once in `main.tsx`). Component-level styling uses CSS Modules (`*.module.css`); shared tokens/fonts are global.

**Testing**: Vitest, default environment is `"node"` (fast, no DOM). A `jsdom` dependency in this environment currently throws `ERR_REQUIRE_ESM` when jsdom is loaded as the *global* Vitest environment — if you add component tests with React Testing Library, opt that specific file into jsdom via a `// @vitest-environment jsdom` docblock at the top of the file rather than changing the global config, or re-diagnose the jsdom/csstools ESM interop issue first.

**Deployment**: `vite.config.ts` sets `base` conditionally on `process.env.GITHUB_ACTIONS` (`/notequest_browser/` in CI, `/` locally) for GitHub Pages project-page hosting. No GitHub Actions workflow exists yet — that's still a TODO.

**Spellcasting** (`DungeonState.spellUses`, a `Record<spellRoll, remainingUses>` seeded at dungeon creation by `computeSpellUses()` from the character's `spells`/`fixedGrants`) covers all six Basic Spells via the `CAST_SPELL` action. Heal and Light are usable anytime (in `CombatPanel` or, out of combat, as a "Cast" button on `CharacterSheet`); Cold Ray, Lightning, and Fireball require an active fight and are rendered as extra per-monster or room-wide buttons in `CombatPanel`. Spell damage runs through `resolveSpellDamage()` in `combat.ts`, which shares Stoneskin/Intangible filtering with `resolvePlayerAttack()` but — since it's a fixed value, not a weapon "damage roll" — never triggers the roll-of-1/roll-of-6 abilities (Explosive, Firebreath, Weakness, etc.). Casting any spell mid-fight consumes the round exactly like `PLAYER_ATTACK` (the monsters counter-attack afterward), except Cold Ray also flags its target `skipNextAttack` so it sits out that round, and Teleport instead ends combat immediately with no counter-attack — this implementation's stand-in for "fleeing a fight," since the app has no notion of player position to send a rules-accurate Teleport to an out-of-combat destination room. A spell's uses, once spent, stay spent within a dungeon run — recovering them requires the Town's Rest action (see Town below).

**The Boss fight**: each dungeon type's `DUNGEON_TABLES[key].boss` (a `Record<1-6, MonsterTemplate>`, authored from the rulebook's per-dungeon "Table: Boss") is rolled once via `resolveBoss()` in `dungeon.ts` the moment the Final Room segment is built — both in the `descend-final` case (auto-triggered at depth 3) and `dead-end-final` (no stairs ever found). Per the rulebook, there's no Content/Monsters roll alongside it, only the Boss. The rolled `MonsterTemplate` is attached to the Final Room's existing `SegmentState.monsters` field and fed through the same `startCombatIfMonsters()` plumbing as an ordinary room, just with a new `isBoss` flag threaded onto `CombatState` — `finishIfVictorious()` checks that flag to grant a flat 2d6 Treasures instead of the normal per-monster Loot table (Treasures are flavor-only, like ordinary Loot, since there's no inventory system). `DungeonScreen.tsx` shows a gold victory panel once the Final Room's boss is marked defeated.

**Chests and Loot**: `RoomContentEntry.hasChest` flags the Room Content rows that describe an actual Chest (vs. passive flavor text like "1d6 coins on the floor," which isn't mechanically applied — see stub list below); a chest can also turn up via `secretPassageResult === "You have found a hidden Chest!"`. `RoomInspector` offers a free "Open Chest" action (no torch cost, per the rulebook) once one is available and unopened, rolling two independent dice — the higher becomes coins, the lower becomes Treasures, unless both come up 1, in which case the chest was empty and rolls on the dungeon's Trap table instead. `DungeonState.treasures`/`keys` are real counters (credited by both Chests and the Loot monster ability's `rollLoot()`) shown on `CharacterSheet` alongside coins — Keys don't do anything yet (no key-unlocks-any-door shortcut), but Treasures are openable (next).

**Opening a Treasure**: each dungeon type's `DUNGEON_TABLES[key].treasure` (a `Record<1-6, RewardOutcome>`) models the rulebook's per-dungeon "Table: Reward" -- rows 1-4 give a Treasure-column item directly; rows that instead say "[Roll in the Wonders/Magic Item/Weapon column]" (Palace's rows 5-6; Prison additionally redirects row 4 to the Weapon table) are `RewardEffect.kind: "rerollColumn"`, resolved by an immediate second `rollDie(rng)` into that column's own table -- see Armor & Weapons for what Wonders/Magic Item rolls actually produce. Rows repeated near-verbatim across dungeon types (Health Potion, Magic Scroll, Valuable jewel) are shared constants (`HEALTH_POTION`, `MAGIC_SCROLL`, etc. in `dungeonTables.ts`), mirroring the existing `BLADE_TRAP`/`CLICK_NOTHING` pattern for traps. The `OPEN_TREASURE` action is usable anytime (not tied to a room, unlike Chests) via a die-roll bar rendered in `DungeonScreen` whenever `treasures > 0`; mid-combat it consumes the round exactly like `CAST_SPELL` (paralysis included, and now also gated on `combat.pendingDamage` being resolved first), since an unattended Health-Potion-on-demand would otherwise trivialize fights. `RewardEffect.kind: "restoreAllSpells"` (Tomb's Mana Potion) needs each spell's *max* uses, which `DungeonState` doesn't otherwise track — the action carries a `maxSpellUses` computed client-side via `computeSpellUses()` (the same helper `CharacterSheet` already uses) rather than adding a new persisted field for it. The rulebook's "worth N Coins in the town" outcomes (Ornament, Religious Object, Sinister Idol, Valuable jewel) don't convert to coins on the spot — there's no town to sell them in yet — so `RewardEffect`'s `"heldValue"`/`"heldValueRoll"` kinds instead push a `HeldItem` (`{ name, worth }`) onto `DungeonState.heldItems`, shown in the `Pack` sidebar card (`src/ui/components/Pack/`, positioned between `CharacterSheet` and the Ledger card in the dungeon, and alongside `CharacterSheet` in Town where it can actually be sold — see Town below).

**Armor & Weapons**: `docs/game-rules-reference.md`'s "Table: Armor" (1d6: Ring 0 HP / Bracelets 2 / Boots 3 / Shoulderpads 3 / Helm 4 / Breastplate 10) is identical across all six dungeon types, so it's one shared `ARMOR_TABLE` constant in `dungeonTables.ts`; "Table: Weapon" differs per type, so each `DUNGEON_TABLES[key]` carries its own `weapon: Record<1-6, WeaponEntry>`. A dungeon type's `wonders`/`magicItem` tables (`Record<1-6, WonderEntry | MagicItemEntry>`) model the Reward table's other two columns; all six dungeon types are fully authored. Every Wonder/Magic Item's unique ability is encoded as an `ItemEffect` (`extraHp`/`weaponDamageBonus`/`damageBonusVsTag`/`damageMultiplierVsTag`/`ignoresMonsterAbility`/`trapImmunity`/`doubleChestCoins`/`combatDamageBonus`/`grantsTorches`/`randomSpell`/`lifesteal`/`instantKillOnRoll`/`flavor`) -- a small, reusable vocabulary rather than one bespoke code path per item; `tags` matching (e.g. "+2 damage to Angels", or Crypt's Garlic necklace naming both Vampire *and* Ghoul) is a case-insensitive substring check against `MonsterTemplate.name`, since the rulebook itself has no formal monster-category system. `MagicItemEntry.fixedFormula` overrides the usual "roll the base Weapon table, then layer a bonus on top" shape for uniquely-named weapons that specify their own damage directly (Crypt's Boatman's Oar, 1d6+1). `ignoresMonsterAbility` covers two different shapes the rulebook conflates under one phrase: abilities a *monster* uses against the player (Undead revival, Paralyze, Deathtouch, Firebreath -- checked via `dungeonReducer.ts`'s `ignoresAbility()`) and the player's own attack being defensively blocked (Stoneskin/Intangible's damage-parity filter in `combat.ts`'s `applyDefensiveAbilities()`, e.g. Boatman's Oar bypassing Intangible) -- the latter is threaded into `resolvePlayerAttack()` as an `ignoreAbilities` array (`dungeonReducer.ts`'s `ignoredAbilities()` collects it from `equippedEffects()`), since `combat.ts` has no notion of the player's inventory on its own. `trapImmunity` is acquired and logged but **not yet enforced** (no trap-roll site currently checks for or consumes it) — same tier of simplification as `WeaponEntry.twoHanded`, which is tracked but doesn't block anything, since this app has no hand-economy/limb-tracking system to enforce the rulebook's "Your Hands" rule against.
- **Acquiring items** (`resolveWonder()`/`resolveMagicItem()` in `dungeonReducer.ts`, called from `OPEN_TREASURE`'s `rerollColumn` case): a Wonder either grants its own HP-bearing item (Jester Hat, 2 HP) or, if it has no HP of its own (Amulet of the Dead), still becomes a `DungeonState.armor` entry at `hp: 0, maxHp: 0` -- never offered as a damage-absorption choice (see below), but still equipped/visible and checked by whichever system its `effect` concerns -- except `combatDamageBonus` (Potion of Fury), which instead adds directly to `combat.playerDamageBonus` if a fight is active, or is simply wasted if not. A Magic Item is always "[Armor] of X" or "[Weapon] of X" (`MagicItemEntry.grants`): resolving one rolls the *base* Armor or Weapon table for the concrete piece/formula, then layers the effect on top -- `extraHp` is baked directly into that new piece's `hp`/`maxHp` (Centurion's [Armor] +1 HP rolls, say, Boots at 3 HP and pushes a 4/4 HP piece), anything else becomes the piece's `effect` (or the weapon's `bonusEffect`); acquiring a new weapon always overwrites `DungeonState.weapon` (no equip-swap UI -- you found a better one, you use it), matching the precedent `COLLECT_REMAINS` already set for picking up a dead adventurer's gear.
- **Wearing them**: `DungeonState.armor: ArmorPiece[]` / `weapon: EquippedWeapon | null` (mirrored on `AdventurerResources` in `town.ts` and `FallenAdventurer` for remains) flow through every place a dungeon gets (re)constructed exactly like `heldItems` already did -- `createInitialDungeonState()`, `RETURN_TO_DUNGEON` (carries them over exactly), and `App.tsx`'s retreat/return round-trip; `RESUME_DUNGEON` deliberately does **not** carry them (a new character doesn't inherit a dead one's gear -- it becomes `remains`, same as coins/heldItems). `Equipment` (`src/ui/components/Equipment/`) renders both, read-only in `DungeonScreen`'s sidebar and with a per-piece "Fix (1 coin)" button in `TownScreen` (wired to `town.ts`'s `canFixArmor()`/`fixArmor()`, the same pure-function shape as `rest()`/`buyTorch()`).
- **Taking damage with armor equipped**: "reduce this value from your HP (or armor's HP, if you're using one — your call)" is a genuine player choice, not automatic, so `applyMonsterTurn()` can't just subtract HP synchronously the way it used to. It splits a round's damage into `poisonDamage` (Poison "cannot be absorbed by armor or other means" — always hits `hp` immediately, and first, so a fatal poison tick can't leave a moot choice pending for the rest) and `absorbableDamage` (everyone else's); if the player has at least one armor piece with `hp > 0` (`hasUsableArmor()`), `absorbableDamage` is stored on `CombatState.pendingDamage` instead of applied, and the round stops there. `CombatPanel` renders an absorption chooser whenever `pendingDamage !== null` (same visual slot the paralyzed-turn message uses), listing HP and every piece with `hp > 0`; picking one dispatches `RESOLVE_DAMAGE`, which subtracts from the chosen pool (overflow past a piece's remaining HP spills onto the player's `hp`, destroying the piece at 0) and re-runs the death check. `PLAYER_ATTACK`/`CAST_SPELL`/`OPEN_TREASURE` all no-op while a choice is pending, the same way they already do during paralysis.
- **Fighting with a found weapon**: `attackBonus()`/`attackMultiplier()` in `dungeonReducer.ts` read `DungeonState.weapon?.bonusEffect` (plus `combat.playerDamageBonus`) when `PLAYER_ATTACK` computes `weaponTotal` -- the multiplier applies to just the weapon's own roll, flat bonuses are added after, since only one bonus effect is ever active on a given equipped weapon at a time (this app doesn't model combining two).

**Still stubbed / not yet built** (see the scope note rendered on the dungeon screen itself, which is the live source of truth for this list): race/class special combat abilities (Grave Digger's +2 vs Undead, Cook's coin-per-kill, Rinoceroid's horn attack), automatic Room Content rewards described only in flavor text (loose coins, Magic Scrolls, Magic Items — no dice are rolled or credited for these); HP damage from traps beyond the torch-costing ditch trap (blade/dart/etc. are logged as flavor only); `trapImmunity` items and `twoHanded` weapons, acquired but not enforced (see Armor & Weapons); the rulebook's "re-roll the Monster table for each empty room" penalty on returning to a dungeon after a Town rest (only an interrupted fight's monsters respawn at full HP -- already-cleared rooms stay cleared); and any persistence beyond the Graveyard (a reload loses `dungeonHistory`/`AdventurerResources` along with everything else in-progress -- resuming only survives character deaths/retreats within the same browser session, not a reload).
